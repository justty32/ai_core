# doc_03 — 函式執行形式 (Execution Forms)

> **一句話講完**：函式不是只有「執行、結束」一種跑法。這裡列出了各種不同的「執行風味」，從最簡單的一波流到複雜的 AI 代理。

---

## 1. 怎麼描述一個函式？ (九軸框架)

我們用九個維度來定義一個函式的行為：
- **I/O 型態**：輸入哪裡來？輸出哪裡去？要不要互動？
- **生命週期**：跑完就死 (one-shot) 還是長駐 (persistent)？
- **狀態**：有沒有記憶？(stateless / stateful)
- **資源**：耗多少記憶體、要不要 GPU？
- **可中斷性**：能不能暫停、能不能恢復？
- **執行保證**：能不能重複跑 (idempotent)？有沒有事務性 (transactional)？
- **組合模式**：它是代理 (proxy) 還是掛鉤 (hook)？
- **環境**：環境是預先弄好的，還是臨時生出來的？
- **隨機性 (第 9 軸)**：它是穩定的 (deterministic)，還是像 LLM 那樣會亂跳的 (nondeterministic)？

---

## 2. 常見的執行形式

### One-shot (一波流)
- 最標準的跑法：啟動 -> 處理 -> 噴結果 -> 關掉。大部分的 Linux 指令（grep, sed, jq）都是這類。

### Multi-shot (連發模式)
- 有記憶的 One-shot。雖然每次跑完都會關掉，但它會把狀態存在檔案裡，下次跑時再讀回來。

### Persistent (伺服器模式)
- 長駐在那裡等。建議設計成 **Server** (用 Socket 溝通)，而不是互動式的 REPL。

### Singleton (單例模式)
- 資源有限，這東西全系統只能跑一個（例如 LLM 門房）。

### Streaming (串流模式)
- 邊算邊噴結果，不用等全部跑完。像 `tail -f` 或 LLM 打字機效果。

### Dry-Run (演習模式)
- 告訴你「如果我真的跑了，會發生什麼事」，但不會真的動你的系統。

### Idempotent (冪等模式)
- 跑一次跟跑一百次結果都一樣。壞了就重跑，安全無痛。

### Transactional (事務模式)
- 要嘛全成功，要嘛全失敗，絕對不留「做一半」的垃圾。

### Agentic (AI 代理模式)
- AI 自己看狀況、自己想、自己決定下一步。這是一個非線性的迴圈。

---

## 3. 觸發機制

這東西是怎麼動起來的？
- **直接呼叫**：你叫它動。
- **定時啟動 (Scheduled)**：時間到了自己動 (Cron)。
- **事件驅動 (Event-driven)**：有人丟檔案過來或發 Webhook 觸發它。
