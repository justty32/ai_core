## 1. ✅ 已收斂：這些已經寫進核心 Spec 了

我們已經把一些好用的原型「扶正」進真正的 `ai_core` 函式庫：

- **宣告與攔截拆分 (Register/Intercept)**：
    - 以前 `register()` 會直接去讀 `argv` 並攔截 `--metadata`，導致你一 `import` 腳本它就跑起來了（Side effect）。
    - 現在改為：`register()` 只負責「宣告」，`intercept()` 才負責「攔截旗標」。這樣你想把腳本當 Library 引用也沒問題。
- **第九軸：隨機性 (Nondeterministic)**：
    - 正式加入第九軸，用來標記這個環節是不是 LLM。
    - **證書制度**：不只是標 `true`，還能標註「用哪個模型、測試組是誰、穩定度幾 %」。這是為了以後的治理和撤照。
- **快取 (Memoized)**：
    - **不加新軸**。如果一個東西「不隨機」且「無狀態」，我們就預設它是可以被快取的。

## 2. ⏸ 暫緩：等 v0 切片跑完再說

這些問題現在吵沒意義，等我們真的拿小模型來跑「程式碼助手」任務（v0）時，自然會知道哪個比較急：

- **語意描述 (Gap C)**：目前的九軸只講「怎麼跑」，沒講「這工具是幹嘛的」。生資產時一定會撞到，到時再定 `summary` 或 `parameters` 的格式。
- **沙箱隔離 (Gap E)**：跑 AI 生出來的 code 怕會把電腦搞掛。我們需要資源限制 (Memory/Time)，但 Python 的 `exec` 很難管。等 v0 跑驗證時再來解。
- **組合軸推導 (A4)**：當 A 工具加上 B 工具組成一條龍時，這條龍的「九軸屬性」能不能自動算出來？

## 3. 🚩 代理人決策：我先拍板了，你看看行不行

在開發原型時，我先選了這幾條路（如果不爽可以推翻）：
- **快取 Key**：用 `SHA256(版本 + 輸入 + 參數 + 檔案內容)`。
- **Hub 的本質**：它就是一個「濾鏡」，把一坨工具轉成給 LLM 看的 `Skill List`。
- **錯誤格式**：統一用 `{"ok": false, "error": {...}}`，方便程式解析。
- **交互安全閥**：規定 `max_rounds` 必填，防止兩台機器人互相吵架吵到倒閉。

1. **固化引擎 (Crystallization)**：當 LLM 處理同一個問題很多次後，我們要「手動」寫死它，還是寫個程式「自動」把它轉成 `if/else`？（**這題最優先**）。
2. **v0 選型**：我們要拿哪個程式框架來開刀？用哪個本地小模型 (<10B) 當實驗組？
3. **行數助手**：是做一個獨立的工具，還是學 Claude Code 那種字串比對的 Edit 模式？
4. **證書誰來發？**：誰有權力說「這個 LLM 節點夠穩，可以上場」？
