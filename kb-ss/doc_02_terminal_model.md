# doc_02 — Terminal 執行模型與 CLI 規則

> **一句話講完**：我們不打算發明新規則，我們直接沿用 Unix/Linux 幾十年來的優良傳統。

---

## 1. 指令解析 (Command Parsing)

當你在 Shell 裡打一行字：
```bash
command arg1 arg2 arg3
```
- **第一個字**：就是那個會跑起來的程式（Python 腳本、Shell 腳本或二進位檔）。
- **後面的字**：就是傳給它的參數。

---

## 2. CLI 旗標慣例 (大家說好的寫法)

我們會照著社群習慣來：

| 寫法 | 意思 | 範例 |
|---|---|---|
| `-v` | 短旗標 | 單一個字母。 |
| `-n 5` | 帶值的短旗標 | 給它一個數字或字串。 |
| `--verbose` | 長旗標 | 比較好讀。 |
| `--output=file.txt` | 帶值的長旗標 | 用等號或空白分開都行。 |
| `--` | 旗標結束 | 後面的東西通通當成檔名或一般文字，不再解析成旗標。 |

---

## 3. 三大串流 (Standard Streams)

- **stdin**：資料進來的地方（沒指定檔案就讀這裡）。
- **stdout**：正確的結果噴出來的地方（可以傳給下一個指令）。
- **stderr**：報錯、日誌或除錯資訊噴出來的地方（不會進到下一個指令，會直接顯示在螢幕上）。

---

## 4. 退出碼 (Exit Code)

- **0**：一切順利 (Success)。
- **非 0**：出事了 (Failure)。

---

## 5. 子指令模式 (Subcommand Style)

我們最愛 `git` 那種寫法：
```bash
git commit -m "fixed bug"
sfc intake --name hello
```
**第一個參數就是子指令**。本框架所有的工具（SFC, Router, Hub）通通都長這樣。這種寫法好懂、好擴充，而且大家都習慣了。
