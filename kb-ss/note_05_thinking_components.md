# note_05｜元件設計隨筆：路由器、小函數與資源管家

- **來源**：`thinking_routing.md` 等系列 (2026-05-21)。
- **狀態**：歷史思考留存。這裡紀錄了各個組件最初是怎麼被構思出來的。
- **一句話大白話**：把網路開發的「路由」概念搬到程式管理上，並幫「小函數」與「受限資源」找個家。

---

## 一、Routing 家族：程式的導航系統

我們借用了網路設備的名詞，但語意稍微不太一樣：

- **Indexer (索引員)**：掃描資料夾，吐出一份清單（`index.md`），讓你知道有哪些工具。它不跑程式，只做目錄。
- **Router (路由器)**：把「名字」對應到「執行檔」。你叫 `senior_dev`，它幫你跑 `./tools/smart_script.sh`。它負責「查表 -> 執行」。
- **Switch (交換器)**：有腦子的路由器。它會看條件（如：輸入是 C 還是 Python），幫你導向不同的工具。
- **Hub (集線器)**：一個「透鏡」。它把分散的工具聚在一起，並幫你加上標籤、過濾掉不需要的資訊，讓 AI 更好讀。

---

## 二、SFC (Small Function Center)：小碎片的家

有些邏輯很薄（例如：只是幫 Prompt 加個 Prefix），不值得獨立成一個檔案。這就是 **SFC (小函數中心)** 的用武之地。

我們把它分成 5 層 (Layer 0~4)：
- **L0 (純資料)**：就只是一個 JSON 檔存著函數定義。
- **L1 (工具化)**：有 `intake` 可以把函數存進 JSON，有 `router` 可以跑它。
- **L2 (快取版)**：程式啟動時就把 JSON 載入記憶體，呼叫起來超快。
- **L3 (管理 API)**：可以透過 API 動態增刪改查這些小函數。
- **L4 (完整 Server)**：有資源管理、錯誤處理、還有像 Git 一樣的 Subcommand 介面。

---

## 三、rma：腳本的資源管家

在 Shell 跑腳本最怕記憶體爆掉或跑太久沒反應。`rma` 就是為了解決這個。
雖然 Linux 有 `timeout` 或 `ulimit`，但 `rma` 更貼心：
- **結構化 Log**：跑完用多少記憶體、多少 CPU，直接吐 JSON。
- **警報回呼**：快爆掉時可以跑另一個腳本來處理。
- **溫柔殺掉**：先給 `SIGTERM` 讓它寫遺言，不聽話再 `SIGKILL`。

---

## 四、生產 vs 使用：AI 怎麼生出新工具？

- **使用管理**：我們怎麼找到、描述、呼叫工具（這是目前的重點）。
- **生產管理**：AI 怎麼生出程式碼？
    - 生出來的東西很小（例如一行 code），還要走 Shell Pipe 嗎？（太慢）。
    - 生出來後，怎麼自動幫它補上 Metadata 並登錄到 Hub？
    - **資產工廠**：這就是後來「聰明模型生資產，笨模型用資產」的雛形。

---

## 五、待補完的坑 (Pending)

1. **Singleton (單例資源)**：有些資源（如：本地 GPU、API Rate Limit）很珍貴。我們需要一個「排隊系統」來管它們。
2. **調用鏈 (Trace)**：當 A 叫 B，B 又叫 C 時，我們需要一個 ID 貫穿全場，出錯才知道是哪一節斷掉。
3. **跨邊界統一呼叫**：不管你是在跑 Python Function、Shell Script 還是遠端 API，在開發者看來，呼叫起來的樣子應該要一模一樣。
