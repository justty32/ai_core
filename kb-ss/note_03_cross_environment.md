# note_03｜跨環境思考：一套規範管全部

- **來源**：`other.md` (2026-05-21 隨想筆記)。
- **狀態**：願景與架構筆記。
- **一句話大白話**：不管是 Shell 腳本、Python 函數還是遠端 API，都應該用同一套規範來管理（增刪查改、路由、呼叫）。

---

## 一、規範的本質：幫「不完美的智能」拉護欄

我們定義的這套規範，說穿了就是在做兩件事：
1. **對不確定性建模**：LLM 每次回的都不一樣？沒關係，我們用結構框住它，讓它在規範內亂跳。
2. **引導缺損的智能**：LLM 邏輯會斷掉、會跳步？我們用流程引導它，幫它把斷掉的地方接起來。

---

## 二、大一統願景：不管在哪跑，呼叫起來都一樣

我們開發時會寫各種「可執行的東西」：有的是一段 Python、有的是一個 Shell 腳本、有的是一個 API。目前的痛點是每種東西的管理方式都不同。

我們的願景是：**跨環境統一管理**。

| 環境 | 例子 |
|---|---|
| **Terminal / Shell** | 腳本、單檔程式、Pipe。 |
| **程式內部 (In-program)** | 函數、Class Method、Lambda。 |
| **作業系統 (OS)** | Process、Daemon。 |
| **分散式 (Distributed)** | 跨機器的 API、Microservices。 |

**目標**：不管是在 Terminal 跑還是程式碼裡呼叫，**納管、CRUD、路由** 都要貫徹始終。開發者只要學一套 Spec，就能通殺所有環境。

---

## 三、執行狀態：你是「跑完就閃」還是「一直都在」？

比起「在哪跑（環境）」，我們更在意「怎麼跑（執行狀態）」。這是兩個正交的軸。

| 分類 | 語意 (大白話) |
|---|---|
| **One-shot** | 射後不理。呼叫一次，跑完就關。 |
| **Multi-shot** | 有記憶。上次跑的結果會影響下一次。 |
| **Persistent** | 常駐在那。活著等候差遣（如伺服器）。 |
| **Singleton** | 資源唯一。同一時間只能有一個在跑。 |

**這套分類是通用的**：一個 Shell 腳本可以是 One-shot，一個 Python 函數也可以是 One-shot。環境描述的是「在哪裡」，執行狀態描述的是「本質性質」。

### 3.1 幾個待補完的坑 (正交軸)

- **Streaming (串流)**：跟 One-shot 不同，它是邊跑邊吐資料（像 `tail -f` 或 LLM 的打字機效果）。
- **觸發機制 (Trigger)**：
    - **直接呼叫**：我叫它，它才動。
    - **事件驅動 (Event)**：檔案變了、收到 Message 了才動。
    - **定時 (Scheduled)**：Cron job。
- **Lazy / Warm (懶載入)**：平常死著，有人叫才活，活了之後會撐一陣子再死。這介於 One-shot 和 Persistent 之間。

---

## 總結

ai_core 不只是一個 LLM 框架，它是想成為一個**統一執行平面**。我們把環境的差異抹平，把執行狀態標準化，讓 LLM (猴子) 可以在這些標準化的軌道 (牢籠) 上跑得很穩。
