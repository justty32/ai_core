# 函式特性的 9 個維度 (Axes Metadata)

這份文件定義了我們怎麼形容一個「函式」（Function/Task）的脾氣。不管它是個簡單的腳本，還是一個背後接了 LLM 的複雜工具，我們都用這 9 個維度（軸）來標記它的行為模式。

如果你是開發者，這就是你在寫 metadata 時要考慮的「規格清單」。

---

## 快速摘要：這 9 軸都在講什麼？

| 維度 (Axis) | 核心問題 | 常見的值 |
| :--- | :--- | :--- |
| **1. I/O 型態** | 資料怎麼進出？ | Stdio, HTTP, Batch (一次性), Streaming (串流) |
| **2. 生命週期** | 跑完就關嗎？ | One-shot (跑完即刪), Persistent (常駐/Server) |
| **3. 跨呼叫狀態** | 有沒有記憶力？ | Stateless (無狀態), Stateful (有狀態，通常存外部) |
| **4. 資源特性** | 會吃多少資源？ | CPU, RAM, GPU, 執行時間 |
| **5. 可中斷性** | 執行中被殺掉會怎樣？ | Safe (安全), Unsafe (危險), Resumable (可恢復) |
| **6. 執行保證** | 重跑會出事嗎？ | Idempotent (冪等), Transactional (交易性質) |
| **7. 組合模式** | 怎麼串接？ | (目前不在單一函式規範內，由呼叫方決定) |
| **8. 環境模式** | 跑在哪裡？ | (目前假設是 Linux 環境，不另行標記) |
| **9. 確定性** | 結果可預測嗎？ | Deterministic (固定), Nondeterministic (隨機/LLM) |

---

## 細節拆解（開發者白話版）

### 1. I/O 型態 (I/O Type)
別想太複雜，這就在講「水管接哪裡」以及「水怎麼流」。
*   **1a 通訊管道**：你是吃 `stdin` 吐 `stdout`？還是走 HTTP API？或是讀寫檔案？
*   **1b 流量模式**：是一次性給整包（Batch）？還是像 `tail -f` 那樣一直噴（Streaming Push）？或者是互動式（中間要停下來問使用者問題）。

### 2. 生命週期 (Lifecycle)
*   **One-shot**：最常見，接到任務 -> 啟動 -> 算完 -> 掰掰。
*   **Persistent**：這東西啟動後不關門。可能是個 Server，也可能是個一直在背景監控的 Daemon。

### 3. 跨呼叫狀態 (State)
*   **Stateless**：每次跑都像第一次見面，給一樣的 input 就給一樣的 output。
*   **Stateful-external**：會把狀態存到外面（資料庫、檔案）。
*   **Stateful-internal**：基本上在我們的架構裡不存在。程序一停，記憶體就空了，要存一定得存到「外面」。

### 4. 資源特性 (Resources)
這不是為了好看，是為了給 `Function Hub` 做排程用的。你要老實交代：
*   這東西吃不吃 GPU？
*   記憶體大約佔多少（啟動時 vs. 運算峰值）？
*   最長會跑多久（Timeout 依據）？

### 5. 可中斷性 (Interruptibility)
當系統想把你的程序殺掉時，會有什麼後果？
*   **Unsafe**：不要隨便殺我，我寫檔寫到一半會壞掉。
*   **Resumable**：我有存 Checkpoint，殺掉重開我可以從斷點繼續。
*   **Safe**：隨便你殺，我是無狀態的，重跑就好。

### 6. 執行保證 (Execution Guarantee)
這是你對使用者的「承諾」。
*   **Idempotent (冪等)**：這最棒。呼叫一次跟呼叫一百次，結果都一樣。壞了？直接重試就對了。
*   **Transactional**：要嘛全成，要嘛全敗（Rollback），不會留下一堆爛攤子。
*   **Dry-run**：你有沒有提供「預演模式」？讓使用者看看跑下去會發生什麼事。

### 9. 確定性 (Determinism) —— **這是給 AI 用的**
這是我們為了馴服 LLM 特別加的一軸。
*   **預設值**：確定性 (Deterministic)。傳統程式碼給 A 一定噴 B。
*   **Nondeterministic**：如果你背後接的是 LLM，請主動標記。
    *   `true`：我知道這東西會隨機發瘋。
    *   `{證書}`：當這塊 LLM 經過測試、穩定度達標後，我們會給它發「認證」，註明模型版本和穩定度。

---

## 為什麼要搞這麼多維度？

我們要把 LLM (天生隨機、難搞) 跟傳統程式 (穩定、好預測) 混在一起跑。沒有這些 Metadata，自動化排程器 (Hub) 根本不知道誰能重試、誰能中斷、誰需要 GPU、或者誰的結果可能是在瞎掰。

標記清楚，系統才不會在半夜重跑一個「非冪等」的任務，結果把你家資料庫搞爆。
