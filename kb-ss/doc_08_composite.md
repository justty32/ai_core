# 複合式規範 (composite_spec.md)

前面的 9 軸是在定義「原子特性」，而這份文件是在講這些特性組合起來的「標準模式」。

如果你宣告你的工具有狀態（`state: "stateful_external"`），那你應該遵守這份檔案定義的路徑和恢復慣例，這樣別人才知道怎麼跟你配合。

---

## 1. 標準目錄慣例：你的檔案該放哪？

當你的工具需要在工作目錄下存取檔案時，請統一使用以下四個資料夾：

| 資料夾 | 語意 | 能不能刪？ |
| :--- | :--- | :--- |
| `.config/` | 你的設定檔。通常由人類寫入，工具只讀。 | 刪了工具就回歸預設值。 |
| `.cache/` | 暫存檔。 | 隨便刪，工具應該要能自己重建。 |
| `.state/` | 執行進度。紀錄你「做到哪了」。 | 刪了就會忘記進度，從頭開始。 |
| `.data/` | 產出的成果。這最重要！ | **不可隨意刪除**，這需要備份。 |

*   **格式**：建議用 `.config/<你的程式名>/` 或是 `.config/<你的程式名>.json`。
*   **宣告**：在 `register()` 時，可以用 `state_dirs=["config", "data"]` 來告訴系統你具體用了哪幾個。

---

## 2. 中斷恢復慣例：壞了怎麼救？

如果你的工具跑一半被 kill 掉，下次重開時該怎麼辦？我們用 `.state/` 底下的 `recovery.json` 來處理。

### 恢復模式
*   **Resume (續傳)**：從斷點繼續往下跑（例如下載器）。
*   **Rollback (回滾)**：把做一半的爛攤子清掉，回到執行前的狀態。
*   **Reset (重置)**：回到某個安全點。

### `recovery.json` 的長相
程式啟動時會自動去讀這個檔案：
```json
{
  "status": "in_progress",
  "mode": "resume",
  "ts": "2026-05-23T10:00:00Z",
  "payload": "recovery/offset.json"
}
```
*   `status`: 如果是 `in_progress`，代表上次沒跑完，該救命了。如果是 `done`，就當成新任務開始。
*   `payload`: 你可以自定義進度資料放在哪。

---

## 為什麼要遵守這些？

1.  **自動化友善**：系統 (Hub) 看到你有 `.state` 目錄，就知道如果你掛了，重跑時你可能會自己處理續傳。
2.  **清理容易**：使用者想重置你的工具，只需要清掉 `.state`；想清出空間，清掉 `.cache`。
3.  **備份明確**：備份腳本只需要抓 `.data`，不需要備份那堆幾百 MB 的快取。

這就是「寫好工具」的衛生習慣。標記清楚你的副作用在哪，大家才敢放心用你的工具。
