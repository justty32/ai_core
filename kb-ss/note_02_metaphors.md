# note_02｜系統隱喻：牢籠裡的猴子與滑坡軌道

- **來源**：`other.md` (2026-05-21 隨想筆記)。
- **狀態**：核心概念筆記。雖然是舊文，但這兩個隱喻對於理解系統架構非常有幫助。
- **一句話大白話**：用「牢籠與猴子」解釋怎麼框住 LLM 的隨機性，用「軌道與球」解釋 Function 的各種執行形態。

---

## 一、牢籠與猴子：怎麼不讓 LLM 搞事？

這個系統的本質，就是用**無數細線編織出來的「牢籠」**。

### 1.1 細線 = 硬編碼邏輯 (Hardcoded Logic)
別看單一一條 `if/else` 或 `regex` 很細、很弱，但只要織得夠密，就能形成一個穩固的結構。這些寫死的邏輯就是「符號主義」的守門人。

### 1.2 猴子 = LLM
LLM 的行為就像一隻猴子，牠充滿無限可能，但也充滿不可控的隨機性。我們的目標是讓猴子在牢籠裡玩，而不是在外面亂跑。

- **入出口 (Function)**：牢籠的門。門上有篩網（Filter），只有特定形狀的資料（猴子）能進來，處理完也只有合格的資料能出去。
- **邊界 (Boundary)**：牢籠就是安全沙盒。就算猴子掉了幾根毛（一點點洩漏），只要整隻猴子沒跑掉（失控），系統就是安全的。
- **空中纜線 (Prompt/Guidance)**：我們在籠子裡手工拉的繩索。這是為了引導猴子去該去的地方。
    - **手滑**：猴子偶爾會抓不住繩子（LLM 偶爾會智障），所以繩子要粗、要好抓。
    - **少即是多**：繩子拉得越少，人力的維護成本就越低。

### 1.3 終極目標：飛輪
讓這隻猴子在籠子裡練強一點，強到牠能「自己幫自己織籠子」（自我改進），這就是所謂的**飛輪效應**。

## 二、滑坡軌道與球：Function 的執行本質

如果把執行一個 Function 比喻成一顆球從滑梯滑下來：

- **軌道** = Function 本體。
- **球進場狀態** = Input。
- **球出場狀態** = Output。

### 2.1 One-shot (噴漆)
球滑過去，路邊有個噴漆口「噗」一聲幫球上色，然後球就出去了。最簡單的單次任務。

### 2.2 Multi-shot (加深顏色)
路邊立個牌子：「這條軌道專門處理上過色的球，會讓顏色變深。」
你可以重複把球丟進去，顏色會越來越深。這就是有上下文或多次迭代的形態。

### 2.3 Persist (卡在中間)
球滑到一半停住了，在那邊等球跟人互動（或是等外部事件）。不滑到底，任務就不算完。

### 2.4 報錯與終止 (跌出軌道)
- **Error**：球直接飛出軌道外，掛了。
- **Interrupt**：滑到一半被人類伸手接住，強行停止。

### 2.5 State (軌道會變)
球滑過去一次，軌道上的計數器就加一，或是軌道的形狀變了一下。這代表 Function 自己會存狀態（State），不只是靠球（Input）來帶資料。

## 結語

不管是「籠子」還是「軌道」，核心概念都一樣：**我們不信任 LLM 的直覺，我們信任「結構」。** 用寫死的邏輯去規範隨機的智能，這就是 ai_core 的底層哲學。
