# 00 · 環境與工具鏈

這台機器上的 Common Lisp 是 **SBCL（Steel Bank Common Lisp）**，用系統套件裝的，搭配
**Quicklisp** 當套件管理器。

## 裝了什麼、在哪裡

| 東西 | 版本 / 位置 | 角色 |
|------|-------------|------|
| **SBCL** | 2.6.6（`/usr/bin/sbcl`，官方 extra 庫） | 實作 + 編譯器 + REPL |
| **Quicklisp** | `~/quicklisp/` | 套件管理器（下載/載入庫） |
| **ASDF** | 隨 SBCL 內建 | build 系統（定義 system、依賴） |
| `~/.sbclrc` | 家目錄 | SBCL 啟動時載入 Quicklisp |

已裝好的庫（放在 `~/quicklisp/`，開機即可 `ql:quickload`）：

| 庫 | 用途 |
|----|------|
| `com.inuoe.jzon` | JSON（現代、快、正確） |
| `clingon` | CLI 參數解析（**原生支援子命令**） |
| `cffi` | C FFI（呼叫共享庫） |
| `fiveam` | 測試框架 |
| `swank` | SLIME 後端，給 nvim/Conjure 連 |

## 一個關鍵坑：`--script` 不讀 `~/.sbclrc`

`sbcl --script foo.lisp` 會**跳過** `~/.sbclrc`，所以 Quicklisp（`ql:` 套件）**不會被載入**，
直接用 `(ql:quickload ...)` 會噴 "Package QL does not exist"。腳本開頭要自己載：

```lisp
(load (merge-pathnames "quicklisp/setup.lisp" (user-homedir-pathname)))
(ql:quickload :com.inuoe.jzon)
```

（互動式 `sbcl`、或 `sbcl --load` 則會讀 `.sbclrc`，Quicklisp 自動就位。）

## 另一個坑：套件符號在「讀取期」就要存在

SBCL 逐個 form 讀 + 求值。若把 `(ql:quickload :cffi)` 和用到 `cffi:defcfun` 的程式碼**包在同一個
form**（例如同一個 `progn` / `handler-case`）裡，整塊會**先被讀取**——此時 `cffi` 套件還不存在，
就報讀取錯誤。解法：`quickload` 要獨立成一個 top-level form，放在用到那些符號的 form **之前**。

## 這個專案怎麼跑（`scripts/run.sh`）

```sh
scripts/run.sh run -n Alice --json   # 跑 CLI（開發主循環）
scripts/run.sh test                  # 跑測試
scripts/run.sh build                 # 編出獨立執行檔 build/cl-lab
scripts/run.sh repl                  # 開已載入 cl-lab 的 REPL
scripts/run.sh swank                 # 起 swank 給 nvim 連（見 06 篇）
```

## local-projects 符號連結

`~/quicklisp/local-projects/cl-lab` → `~/code/cl-lab` 已建好，所以在**任何地方**的 SBCL 都能
`(ql:quickload :cl-lab)` 找到這個專案。這是 Quicklisp 找本地專案的慣用手法。
