note_09 — 舊版設計世系：Agents Guide 與 44 個 Process 生態

  • 來源：git commit 684082c 之 auto/AGENTS.md + auto/FUNCTIONS.md（兩者皆由舊版 hub --gen-agents-md / --gen-functions-md 自動產生）
  • 狀態：舊版設計世系（已於 commit 7b9b8e3 被完全覆蓋；僅供溯源，非現行事實）
  • 一行摘要：舊版實際長出來的 process 生態——44 個 process（一份標 44、AGENTS 標 44、FUNCTIONS 標「44 processes total」；任務描述提及「43 processes」屬概數），以 Go/devops 程式碼輔助為主軸，加上 planner/chain 編排與 LLM/通用工具。
  • 現行事實見 note_01_vision_and_origin.md 與 doc_05_axes_metadata.md、doc_10_arch_overview.md。本檔同屬 note_07_legacy_system_design.md 的舊版世系。

⚠️ 閱讀前必讀

這份 process 清單是舊版（process 世代）的產物，由舊版 hub 自動生成，列的是舊版 processes/*.py（已隨 commit 7b9b8e3 一併消失）。其 --metadata 欄位（tags/io/version）是舊版 schema，非現行九軸。現行的範例函式見 funcs/（如 echo.sh）與 try_implement/ 原型。凡衝突以現行為準。

數量註記：任務背景提及「43 個 processes」，但 git 684092c 的兩份自動產出檔均明確寫 44 processes total（AGENTS.md：「共 44 個（degraded 0 個已隱藏）」）。本檔以實際檔案的 44 為準，43 視為概數。

一、auto/AGENTS.md（舊版 Agent Guide）

檔頭：「Auto-generated by hub --gen-agents-md. Do not hand-edit.」

系統概念（給 agent）

每個能力是一個 process（可執行檔），遵守 --metadata 約定。Hub 是一次性索引工具，不是 server。

「你能做什麼」（agent 操作指引）

  • 找已有 process：hub --search-func "<keyword>"
  • 看單一 process metadata：./processes/<name>.py --metadata
  • 跑 process：echo "input" | ./processes/<name>.py 或 ./processes/<name>.py "input"
  • 新增 process：見 README §「AI 寫 process 的標準流程」六步

Process 約定（AGENTS.md 複述）

--metadata 印一行 JSON，含必填 name/description，選填 version/tags/input/output/examples。一般執行：CLI arg 或 stdin 二擇一；成功寫 stdout exit 0；失敗寫 stderr non-zero exit。examples 內每個 {input,output} 都被 hub --validate 自動執行比對，當回歸測試。

進入專案後的建議流程（AGENTS.md 獨有）

  1. 跑 hub --search-func "<關鍵字>"；不確定時直接讀 auto/FUNCTIONS.md。
  2. 找到候選後跑 --metadata 確認 io 約定。
  3. 編排：用 processes/chain.py 串連，或用 processes/planner.py 從自然語言任務產出計劃。
  4. 沒有合適 process 時：跟使用者確認後，照六步流程新增。

AGENTS.md 的「依 tag 分群」索引（舊版 tag 生態，獨特保留）

AGENTS.md 把 44 個 process 按 tag 列出（同一 process 會出現在多個 tag 下）。完整 tag 清單（按字母）： 12factor, ai, api, architecture, bash, benchmark, bind, build, chain, changelog, channel, chat, chinese, ci, code-quality, codegen, commit, concurrency, config, context, database, debug, dependency-injection, design, dev-tool, devops, diagnose, diff, docker, docs, env, error, error-handling, errors, explain, format, general, generics, git, go, go1.18+, godoc, golang, goroutine, grpc, handler, http, interface, json, k8s, language, learning, llm, makefile, meta, middleware, migration, mock, multi-turn, openapi, orchestration, patterns, performance, planner, postgres, pr, production, proto, qa, refactor, regex, release, repository, review, server, shell, shutdown, signal, slug, sql, struct, summarize, swagger, tdd, test, testify, text, translate, unit-test, url, utility, wire, workflow, yaml。

其中 go / golang 兩個 tag 各聚集 ~25 個 process——印證舊版第一目標問題實際落在 Go 程式碼輔助。

二、auto/FUNCTIONS.md — 完整 44 process 清單

檔頭：「Auto-generated by hub --gen-functions-md. Do not hand-edit.」「44 processes total.」 每筆格式：name / description / tags / version(全 1.0) / io / path（皆 /home/lorkhan/repo/ai_core/processes/<name>.py）。下表無損保留 name + description + tags + io。

A. 編排 / meta（舊版 planner-chain 編排核心，獨特項目）

  • planner — Decompose a natural language task into an ordered sequence of process calls. tags: ai, orchestration, planner, meta. io: task description in natural language → json execution plan.
  • chain — Execute a planner JSON plan: run each process in order, pipe outputs between steps. tags: ai, orchestration, chain, meta. io: json plan (from planner) + --input <initial_input> → final step output.

B. LLM / 對話

  • llm_call — Send input text to an LLM and return the response. tags: llm, ai. io: text → text.
  • llm_chat — Multi-turn LLM conversation with persistent session history. tags: llm, chat, multi-turn. io: text → text.
  • bind_context — Prepend a system prompt to input and call an LLM. tags: llm, context, bind. io: text → text.
  • ask — General Q&A with persistent conversation history for follow-up questions. tags: qa, chat, utility, general. io: question → answer.

C. 通用工具 / dev-tool（含帶 examples 的兩個示範 process）

  • echo — Echo input text back to stdout unchanged. tags: utility, debug. io: text → text. Examples: echo 'hello' | <process> → hello；空輸入 → 空。
  • slugify — Convert text into a URL-safe slug (lowercase ASCII, hyphen-separated). tags: text, url, slug, utility. io: stdin (text) → stdout (text). Examples: 'Hello, World!'→hello-world；' Spaces Here '→spaces-here；'café_FOO'→cafe-foo.
  • summarize — Summarize text into concise bullet points. tags: text, summarize, utility. io: text → bullet-point summary.
  • translate_zh — Translate text to Traditional Chinese (繁體中文). tags: translate, chinese, language, utility. io: text in any language → Traditional Chinese text.
  • json_format — Validate and pretty-print JSON with 2-space indentation. tags: json, format, utility, dev-tool. io: json string → formatted json or error.
  • regex_explain — Explain a regular expression pattern in plain language with examples. tags: regex, utility, explain, dev-tool. io: regex pattern (optionally with sample string) → plain language explanation.
  • error_diagnose — Diagnose error messages and stack traces, suggest root cause and fix. tags: debug, error, diagnose, utility. io: error message or stack trace → diagnosis and fix suggestions.

D. 「Explain」家族（跨語言解說）

  • go_explain — Explain Go code in plain language, including patterns and gotchas. tags: go, golang, explain, learning. io: go source → plain language explanation.
  • shell_explain — Explain a shell command or bash script in plain language. tags: shell, bash, explain, utility. io: shell command/script → explanation.
  • sql_explain — Explain SQL queries and suggest index/performance improvements. tags: sql, database, explain, performance, postgres. io: sql query → explanation and improvement suggestions.
  • yaml_explain — Explain a YAML config file (k8s, docker-compose, GitHub Actions, etc.) in plain language. tags: yaml, k8s, docker, ci, explain, devops. io: yaml content → explanation.

E. Git / workflow

  • go_commit_msg — Generate a conventional commit message from a git diff of Go code. tags: go, golang, git, commit, workflow. io: git diff → conventional commit message.
  • changelog_gen — Generate a CHANGELOG.md entry from git log commits. tags: git, changelog, release, workflow. io: git log output → changelog markdown section.

三、舊版 process 生態的觀察（供溯源 / 對照現行）

  • 第一目標問題的早期形態：舊版幾乎把全部 codegen/review 火力壓在 Go（~23 個 go_* + 2 個 devops codegen），佐證 ai_core 從一開始就把「程式碼輔助助手」當主戰場。現行 roadmap 把「第一目標問題」明確定為程式碼輔助助手（上層聰明模型生資產、下層笨模型受約束填碼），與此一脈相承，但現行不再以「一堆 go_* generator process」的形式存在。
  • planner + chain 編排（獨特且已消失）：舊版用 planner（自然語言→JSON 計劃）+ chain（執行計劃、step 間 pipe、中間結果存 ~/.ai_core/chain/session.json）做任務編排。現行用 try_implement/tools/chain.py 等原型探索編排，規範未定。
  • 舊版 metadata vs 現行九軸：舊版每個 process 只有 tags/version/io（純宣告型描述）；現行九軸（lifecycle/state/guarantee/nondeterministic…）是行為契約，兩者層級不同——舊版 io 字串約等於現行 entries 概念的雛形，但缺乏狀態/保證/非確定性的形式化。
  • degraded 機制：AGENTS.md 顯示「degraded 0 個已隱藏」，對應 README 的 _status: ok|partial|absent 容錯降級設計——壞掉的 process 不炸全表，只隱藏。
