一、標準實現規範的本質

目前在定義的標準實現規範，其作用是：

對不確定性進行建模，同時對缺損智能進行引導的框架。

具體來說：

  • LLM 的行為具有不確定性（同樣的輸入可能產生不同輸出）
  • LLM 的邏輯鍊條豐富但有缺損（能推理，但推理可能中途出錯或跳步）
  • 標準實現規範提供結構，讓 LLM 在這個結構內行動，不確定性被限縮到規範許可的空間內

二、跨環境統一管理規範的願景

對於各種「可執行的東西」——程式片段、腳本、單檔程式、程式內函數——的管理，包括：

  • 納管：如何將它們收入某個系統
  • 增刪查改：如何操作它們
  • 路由：如何找到並呼叫它們

這一整套應該能夠跨越不同的環境邊界：

環境 例子
Terminal / shell 單檔程式、shell 腳本、pipe
程式內部 函數、class method、lambda
作業系統層 process、subprocess、daemon
分散式系統 跨機器的 API 呼叫、遠端服務

目標是整合成一套貫徹始終的規範，涵蓋呼叫、實作、管理三個面向，讓開發者不需要為不同環境各自發明一套做法。

目前 ai_core 的各元件（metadata 協定、router、forge、SFC、outside_progs 等）都是朝這個方向在走，但尚未被明確地以這個統一願景來框架。

環境清單補充說明：

  • Container 內部是 OS 環境，對外是 API 呼叫環境（分散式）；Serverless 同理，從 caller 角度是 API 呼叫，但從 function 作者角度有特殊約束（stateless、冷啟動、無常駐 process）
  • Browser 本質是程式內環境，頂多算 OS 層；WASM 是可攜式執行模型，能跨越多個環境邊界，值得留意
  • GPU kernel（CUDA / OpenCL）：獨立記憶體空間、不同處理器、走 driver 介面——不屬於上表任何一格，是真正額外的執行環境，暫不深入
  • 同步 vs 非同步呼叫模型是橫切所有環境的面向：Terminal / in-program 預設同步，distributed / browser 預設 async / event-driven；這會影響「呼叫」規範在不同環境下的設計

三、執行狀態分類的跨環境性

可執行的東西，可以依據其執行狀態做分類：

分類 語意
one-shot 呼叫一次、執行完畢、結束
multi-shot 跨多次呼叫保有狀態，上次結果影響下次執行
persistent 持續存在，等待呼叫
singleton 資源有限，同時只能有一個（或受限數量的）實例

這套分類應該跨越所有執行環境——無論是 terminal 中的 shell 腳本、程式內部的函數、分散式系統中的服務，都可以用這四種狀態來描述其執行性質。

這是比「環境」更本質的分類軸：環境描述「在哪裡跑」，執行狀態描述「怎麼跑、跑完後發生什麼」。兩個軸正交，合起來才是完整的描述。

補充——三個待深入設計的面向：

Streaming（增量輸出）：one-shot 是執行完才輸出，streaming 是邊執行邊輸出（tail -f、串流 LLM 回應）。跟 persistent 不同：persistent 等待多個獨立呼叫，streaming 是單次呼叫產生連續輸出。可能值得列為獨立執行狀態。

觸發機制（另一個正交軸）：上表四種狀態描述「執行後的行為」，觸發方式是另一個軸：

觸發機制 說明
直接呼叫 上表隱含的預設
Event-driven 被外部事件推送觸發（message queue、webhook、filesystem watch）
Scheduled 由時間觸發（cron）

觸發機制可疊在任何執行狀態上（如「定時跑的 one-shot」、「收到 event 才啟動的 persistent」）。

Lazy / warm（中間態）：介於 one-shot 與 persistent 之間——第一次呼叫才啟動，之後保持活著一段時間。可作為獨立分類或 persistent 的子類，待定。

以上三點待之後針對執行狀態分類做更完整的設計。
