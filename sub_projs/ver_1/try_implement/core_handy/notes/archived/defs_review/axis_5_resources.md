# 軸 5 `resources` — defs 不足審查

> 審查對象：`notes/axis_5_resources.md` Round 2 定案。
> 結構：無固定欄位、opt 預定義(memory/cpu/gpu/time/disk/network 各 struct/scalar)、extra 收自定義。

三層通則套得**最漂亮**的一軸，爭議最少。不足偏「值的型別」與「跨軸/分歧成本」。由強到弱：

---

## 1.〔值無量綱〕容量/時間全是自由字串，hub 排程得自己 parse 且無格式強制

`"4gb"`/`"30s"` 先當 `std::string`、不解析量綱（沿軸 1 慣例）。但 `resources` 的**唯一受眾是 hub 排程器**，
它做容量規劃**必須**把字串轉成數量——而：
- 無格式強制：`"4gb"` / `"4096mb"` / `"4 GB"` / `"4G"` 都能寫，每個 hub 各自 parse → 易不一致；
- 這是「描述面說給 hub 讀，但給的是 hub 得自己 parse 的非結構化字串」，與軸 1 transport 同類問題的弱化版。

依升級壓力通則：單位是「普遍且必須 hub 才能排程」→ 至少該約定正規化規則（regex / 標準單位表），
而非純自由字串。否則排程決策建在不可靠的字串解析上。

## 2.〔分歧成本〕`cpu` 是本線刻意新增（權威無），語意微妙、易填錯

- `cpu` 權威無預定義 key，本線**刻意升 opt 欄位**，語意還收窄為「平行度提示」（缺席＝1）。
- 筆記自己繞了一大圈才釐清「(A) 內部平行度 vs (B) 並發安全性」「核心數 vs 占用%」。
- 對一個「描述面給 hub 讀」的欄位，這種語意微妙性意味著**作者很可能填錯**
  （把「我想要 4 核」誤當「我會佔 4 核」或誤填占用%）。

建議：要嘛**改名**為 `parallelism`（直接點出語意，減少誤填），要嘛重估是否值得偏離權威——
偏離增加與 Python 線/權威的對齊成本，得有對應收益。

## 3.〔跨軸無意義組合未閉合〕one_shot × idle 記憶體

`memory.idle` 僅對 `persistent`（軸 2）有意義，但型別層不強制 → 可以宣告 `persistent=false` 卻填 `idle` 記憶體。
又一個「已知無意義組合、退回文件約束」的案例（同軸 3×4、軸 3×7）。見索引跨軸主題 2。

---

## 用它們自己的尺檢驗

軸 5 的三層結構本身遵守升級壓力（開放資源天生該 opt+extra）。
唯一逆反的是**值型別**：若 hub「普遍且必須」解析數量，純字串是不夠格的最小保真——
這層保真度該升（結構化單位），而非停在「先當 string」。

## 建議

- 為容量/時間值約定正規化格式（即使仍存 string，至少釘 grammar），降低 hub 解析歧義。
- `cpu` 改名 `parallelism` 或補強文件，並重新確認偏離權威的收益。
- idle×persistent 跨軸約束納入「總 metadata 驗證」一併處理（見索引）。
