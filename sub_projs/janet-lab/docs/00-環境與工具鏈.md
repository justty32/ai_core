# 00 · 環境與工具鏈

這台機器上的 Janet 是**從原始碼編譯、裝進 `~/.local`（不用 sudo、不碰系統）**。

## 裝了什麼、在哪裡

| 東西 | 版本 | 位置 |
|------|------|------|
| `janet`（直譯器 + REPL） | 1.41.2 | `~/.local/bin/janet` |
| `jpm`（套件 / build 管理器） | 隨 1.41.2 | `~/.local/bin/jpm` |
| `spork`（官方標準庫，含 json / argparse） | — | `~/.local/lib/janet/spork/` |
| `janet-lsp`（語言伺服器） | 0.0.12 | `~/.local/bin/janet-lsp` |
| 模組安裝路徑（JANET_PATH） | — | `~/.local/lib/janet/` |

`~/.local/bin` 已在 PATH 上，所以 `janet`、`jpm` 開新 shell 就能直接用。

## 三個最常用指令

```sh
janet                       # 開 REPL，(quit) 或 Ctrl-D 離開
janet some-file.janet       # 跑一支腳本
janet -e '(print (+ 1 2))'  # 一行求值
```

## 為什麼是原始碼編譯而不是套件管理器

Janet 不在 Manjaro 官方倉庫，AUR 只有一個兩年沒更新的 `-git` 包。Janet 本體是單一 C
檔、編譯只要幾秒，裝進 `~/.local` 最乾淨：不需要 root、升級只要重編、要移除直接刪
`~/.local` 下那幾個檔。原始碼放在 scratchpad，之後要升級版本再 `git fetch` 新 tag 重編即可。

## 一個小地雷（native module 與 libjanet.so）

`janet` 執行檔本身是 self-contained（`ldd` 看不到 libjanet 依賴），乾淨 shell 直接能跑。
jpm 編出來的 **native (C) 模組**（例如 `spork/json`）在載入時是靠 janet 執行檔用 `-rdynamic`
匯出的符號解析的，**不依賴** `libjanet.so`，所以「`libjanet.so` 沒進 ldconfig cache」不影響。
只有你要把 janet **嵌進別的 C 程式**時才需要處理那個 `.so`——一般開發用不到。
