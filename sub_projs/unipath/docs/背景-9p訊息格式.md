# 背景知識：9P2000 逐欄訊息格式

> unipath 先驗知識。承 [背景-inode與pipe與socket.md](背景-inode與pipe與socket.md)。
> 前篇搞懂了 qid＝inode、walk＝dentry 查找、socket 兩端怎麼接起來；本篇把「socket 兩端到底送什麼位元組」攤開。
> **這不是紙上協議**——`unipath_9p.py` 已經是一個會跑的真 9P2000 server，可被 Linux 核心 `v9fs` 直接掛載。下面逐段對照它的原始碼。

## 零、活教材是誰

`unipath_9p.py` 的 `NineP` 類（第 68～206 行）把 `unipath_env.Env`（活 Python 物件圖）服務成真 9P2000 線格式；`serve_conn`（第 209～226 行）是每條連線的收發迴圈；`selftest()`（第 242～302 行）是一個**獨立**於 server 之外、自己手刻的真 9P client——不是左手畫右手，是兩份獨立實作互通，這才驗到「線格式」而非「函式呼叫」。下面所有欄位描述都能在這支檔案裡找到對應行。

## 一、訊息框架：size[4] type[1] tag[2] body

```
size[4]   整個訊息的總位元組數（含 size 自己這 4 bytes）
type[1]   訊息型別（Tversion=100、Rversion=101…偶數 T、奇數 R = T+1）
tag[2]    這筆請求的識別碼，回應原樣帶回
body      依 type 而定的欄位
```

整數一律 **little-endian**；字串＝`size[2]` + UTF-8 位元組（無終止符，長度即定界）。編碼小工具在第 46～61 行：

```python
def p2(v): return struct.pack("<H", v)
def p4(v): return struct.pack("<I", v)
def p8(v): return struct.pack("<Q", v)
def ps(s):
    b = s.encode()
    return p2(len(b)) + b
```

`Reader` 類（第 54～61 行）是反方向：`u1/u2/u4/u8` 依序啃位元組，`s()` 先讀 `size[2]` 再切對應長度字串。收發迴圈本身在 `serve_conn`（第 213～226 行）：先讀 4 bytes 拿 `size`，再讀 `size-4` bytes 拿剩下的 body，body 第 0 byte 是 `type`、第 1～2 byte 是 `tag`，之後才是訊息專屬欄位（`r.i = 3` 這行就是把 Reader 游標跳過 type+tag，第 220 行）。回應同一套：`p4(4+1+2+len(rbody)) + bytes([rtype]) + p2(tag) + rbody`（第 225 行）。

**tag 的作用**：一個連線上可能同時有多筆請求在飛（pipeline），tag 讓 client 把「哪個回應對哪個請求」配對回去；`NOTAG`（0xFFFF）保留給第一筆 `Tversion`（協商階段還沒有其他 tag 在用）。`unipath_9p.py` 目前的實作是逐筆同步處理（`dispatch` 跑完才寫回應，見第 222～226 行），所以 tag 沒被拿來做真正的並發配對，但它仍照收即送、原樣帶回（`p2(tag)`），協議層面的欄位沒有偷懶。`selftest()` 自己也偷懶：所有請求固定 `tag=0`（第 247 行 `p4(...) + bytes([mtype]) + p2(0) + body`），因為它本來就是逐筆同步呼叫，不需要配對。

## 二、T/R 訊息成對：逐欄列出 body

下面每組先列 wire 欄位，再指到 `dispatch()`（第 108～203 行）裡對應的分支。

### Tversion / Rversion（第 109～113 行）

| | 欄位 |
|---|---|
| Tversion | `msize[4]` `version[s]` |
| Rversion | `msize[4]` `version[s]` |

`msize` 是 client 提議的最大訊息長度，server 取自己與對方的較小值：`self.msize = min(self.msize, ms)`。`version` 字串以 `"9P2000"` 開頭才承認，否則回 `"unknown"`（client 見到 `"unknown"` 該中止）。這是**唯一**沒有 fid、不需要先 attach 就能送的訊息——它是連線建立後的第一步握手。

### Tattach / Rattach（第 115～118 行）

| | 欄位 |
|---|---|
| Tattach | `fid[4]` `afid[4]` `uname[s]` `aname[s]` |
| Rattach | `qid[13]` |

`fid` 是 client 選定、之後代表「根」的把手號碼；`afid` 若填 `NOFID`（`0xFFFFFFFF`）＝不做認證；`uname`/`aname` 是使用者名／要掛載的樹名，這裡完全不驗（`r.s(); r.s()` 讀完就丟）。成功後 `fids[fid] = {"path": "/", "open": False}`——這就是「fid 誕生」的第一筆。

### Twalk / Rwalk（第 120～139 行）

| | 欄位 |
|---|---|
| Twalk | `fid[4]` `newfid[4]` `nwname[2]` `wname[s]` × nwname |
| Rwalk | `nwqid[2]` `qid[13]` × nwqid |

從既有 `fid` 的路徑出發，依序吃掉 `wname` 陣列裡每個名字（`..` 特別處理成回根，第 128 行），每吃一段就查一次 `env.stat`；**吃到哪段吃不下去就停**（第 129～133 行的 `try/except: break`），回應的 `qid` 陣列長度即實際吃了幾段。只有**全部吃完**（`len(qids) == n`）`newfid` 才會真正生效並指向新路徑；`n == 0` 是特例——不吃任何名字、單純把 `newfid` 指到跟 `fid` 一樣的路徑，等於「複製一份 fid」（`selftest()` 用這招複製出一個指向根的 fid 來讀目錄，第 279 行 `walk(2, [])`）。

### Topen / Ropen（第 141～150 行）

| | 欄位 |
|---|---|
| Topen | `fid[4]` `mode[1]` |
| Ropen | `qid[13]` `iounit[4]` |

`mode` 對應 Plan 9 的 `OREAD=0`／`OWRITE=1`／`ORDWR=2` 等（`selftest()` 用 `bytes([0])`／`bytes([1])`，第 273、294 行）。**這份實作沒真的檢查 mode**——第 142 行 `r.u1()` 讀完直接丟，任何 mode 都放行；真正的唯讀保護是在 `Env.write()` 那層對 `status`/`ctl` 之外的路徑丟 `EROFS`，不是在 open 這關擋。若目標是目錄，順便把整個目錄項預先渲染成 stat blob 列表存進 `st["dir"]`（第 147～149 行）——之後 `Tread` 直接照 offset 切這份快取，不會每次 read 都重掃。`iounit` 這裡固定回 0（表示「沒有建議的最大單筆傳輸量，client 自行決定」）。

### Tread / Rread（第 152～171 行）

| | 欄位 |
|---|---|
| Tread | `fid[4]` `offset[8]` `count[4]` |
| Rread | `count[4]` `data` |

若 `fid` 是已 open 的目錄，走「目錄項」分支：把預渲染的 stat blob 一個個接，直到超過 `offset`/`count` 邊界（第 158～164 行）——這正是 `readdir` 為何在 9P 裡看起來像「讀一個特殊格式的檔案」。若是一般檔，直接 `data[off:off+count]` 切片（第 170 行），不足的部分自然就是 EOF（回傳長度小於要求的 count）。

### Twrite / Rwrite（第 173～182 行）

| | 欄位 |
|---|---|
| Twrite | `fid[4]` `offset[8]` `count[4]` `data` |
| Rwrite | `count[4]` |

**注意**：第 174 行 `r.u8()` 把 `offset` 讀出來後直接丟掉，`self.env.write()` 永遠是整段覆寫（`unipath_env.py` 的 `write()` 對 `data` 葉子是 `parent[key] = self._parse(...)`，沒有游標概念）。也就是說這裡的 write **不是**真正的隨機存取寫入，是「一次性整段設值」——跟 pipe 的語意比較像，跟一般檔案的 seek+write 不一樣，用 unipath 掛載後如果分段寫會踩到這個簡化。

### Tclunk / Rclunk（第 192～194 行）

| | 欄位 |
|---|---|
| Tclunk | `fid[4]` |
| Rclunk | （空） |

`fids.pop(fid, None)`——單純從這條連線的 fid 表移除，對應 `Env` 這邊什麼都不用做（因為 `Env` 本來就沒有「開啟狀態」，是每次操作即時算路徑），這也側面說明 unipath 的 fid 只是 client 端的路徑書籤，不是真的持有底層資源。

### Tstat / Rstat（第 184～190 行）

| | 欄位 |
|---|---|
| Tstat | `fid[4]` |
| Rstat | `stat[n]` （見下節「雙重 size」） |

### Tflush / Rflush（第 196～198 行）、唯讀邊界（第 200～201 行）

`Tflush` 只帶 `oldtag[2]`（要取消的那筆請求），這裡直接回空 `Rflush`——因為目前是同步逐筆處理，不存在「還沒回完、想取消」的並發窗口，所以照協議回應但沒有真的取消語意。`Tcreate`/`Tremove`/`Twstat` 三個會改變樹結構的訊息一律回 `Rerror("EROFS")`（第 200～201 行）——這棵樹目前唯讀（能寫既有葉子的內容，不能新增/刪除節點）。

### Rerror（第 205～206 行）

任何分支拋 `UnipathError` 或未知 `type` 都落到這：`ps(ERRNO_STR.get(name, name))`——只有 `ename[s]` 一個欄位（base 9P2000 沒有數字 errno，那是 9P2000.u 擴充才加的 `errno[4]`）。`ERRNO_STR`（第 64～65 行）把 `Env` 丟出來的符號名（`ENOENT`/`ENOTDIR`/`EISDIR`/`EROFS`/`EIO`）轉成人類看得懂的訊息字串。

## 三、fid：client 端的檔案把手

`fid` 之於 9P，就像 `fd` 之於 Unix process——一個小整數，代表「client 現在正指著樹上某個位置」。差別在於 `fid` 是 **client 自選號碼**（不是 server 配發），且是**每條連線各自一份表**（`serve_conn` 第 210 行 `fids = {}`，這個 dict 活在該連線的 closure 裡，換一條 TCP 連線就是全新的 fid 命名空間）。生命週期三動作都在 `unipath_9p.py` 裡：

- **誕生**：`Tattach` 拿到代表根的 fid（第 117 行）。
- **衍生**：`Twalk` 從舊 fid 的路徑出發，把新路徑綁定到 `newfid`（第 138 行）——概念上像「複製並移動一份把手」，舊 fid 不受影響。
- **釋放**：`Tclunk` 從表裡移除（第 193 行），之後這個號碼可以被 client 重新拿去用在別的路徑上。

## 四、qid[13]：9P 版的 inode 身份

```
type[1]     bit 0x80 = QTDIR（目錄）
version[4]  該檔案內容變動時遞增的版本號（快取失效用）
path[8]     在此檔案系統內唯一、穩定的識別碼（＝inode 號）
```

對應 `_qid()`（第 81～86 行）：

```python
def _qid(self, path):
    if path not in self.qid_paths:
        self.qid_paths[path] = self.qid_next
        self.qid_next += 1
    qtype = QTDIR if self._isdir(path) else 0
    return bytes([qtype]) + p4(0) + p8(self.qid_paths[path])
```

`path` 這欄用一個 `路徑字串 → 遞增整數` 的表（`self.qid_paths`）延遲分配——第一次被問到某條路徑才發一個新號碼，之後同一路徑永遠拿到同一個 `qid.path`，這就是「inode 號穩定」的具體實作。**誠實的簡化**：`version` 這裡永遠回 `p4(0)`，不像真 9P 會在內容變動時遞增（讓 client 端快取知道要不要重新讀）——`unipath` 的樹本身會被背景 tick 偷偷改內容（`Env._ticker`），但 qid.version 目前沒有反映這件事，之後接「時間即狀態轉移」時這裡值得回頭補。

## 五、stat 結構與「雙重 size」陷阱

單一 stat 結構的欄位順序（`_stat_bytes()`，第 97～105 行）：

```python
def _stat_bytes(self, path):
    st = self.env.stat(path)
    isdir = st["kind"] == "dir"
    mode = (DMDIR | 0o755) if isdir else (0o444 if st["ro"] else 0o644)
    length = 0 if isdir else st["size"]
    inner = (p2(0) + p4(0) + self._qid(path) + p4(mode)
             + p4(0) + p4(0) + p8(length)
             + ps(self._name(path)) + ps("unipath") + ps("unipath") + ps("unipath"))
    return p2(len(inner)) + inner
```

也就是：`type[2]=0` `dev[4]=0` `qid[13]` `mode[4]` `atime[4]=0` `mtime[4]=0` `length[8]` `name[s]` `uid[s]` `gid[s]` `muid[s]`，最外面再包一個 `size[2]`（＝`inner` 的長度）。`mode` 借用 `env.stat()` 給的 `kind`/`ro`：目錄是 `DMDIR|0o755`，`status`（唯讀葉子）是 `0o444`，其他葉子（`data`/`ctl`）是 `0o644`；`atime`/`mtime` 老實回 0（`Env` 沒記時間戳）；`uid`/`gid`/`muid` 全部寫死 `"unipath"`（沒有使用者概念）。

**「雙重 size」陷阱**在 `Tstat` 分支現形（第 184～190 行）：

```python
if mtype == Tstat:
    ...
    sb = self._stat_bytes(st["path"])          # sb 本身已經是 size[2]+inner
    return Rstat, p2(len(sb)) + sb              # 又包了一層 size[2]
```

`Rstat` 的 body 是 `stat[n]`——這個 `[n]` 是外層再包一次「後面這包 blob 共幾 bytes」，而 blob 打開後前兩個 byte（`inner` 前面那個 `size[2]`）又自報「我（不含這兩個 byte）還有幾 bytes」。同一個量被記了兩次，只差 2（外層數字＝內層數字＋2）。這不是 `unipath_9p.py` 自己發明的怪癖——base 9P2000 規格本來就這樣定義 `Rstat`/`Twstat` 的 body——但正是最常被實作者抄錯的地方：忘記外層 wrap、或內層 size 少算/多算那 2 bytes。

**對照**：目錄的 `Tread` 回傳（第 157～165 行）沒有這層外層 wrap——每筆項目就是 `_stat_bytes()` 的原始輸出（自帶內層 `size[2]`）直接串接（`out += e`），一整包再靠 `Rread` 自己的 `count[4]` 框住。Client 端逐筆解析時，只需要照著每筆自帶的內層 `size[2]` 切一刀、切完游標跳到下一筆——`selftest()` 的 readdir 解析迴圈就是這樣做的（第 284～289 行：`sz = rr.u2(); end = rr.i + sz; ...; rr.i = end`）。所以「雙重 size」**只在單筆 `Tstat`/`Rstat` 查詢出現**，目錄枚舉時每筆 stat 只有單層 size。

## 六、一次完整對話：version → attach → walk → open → read → clunk

`selftest()`（第 242～302 行）就是這個流程的活範例，且是**獨立**於 server 之外自己手刻的真 9P client：

```
Tversion(msize=65536, "9P2000")  →  Rversion(msize, "9P2000")         第 261～262 行
Tattach(fid=0, afid=NOFID, "me", "")  →  Rattach(qid of "/")           第 263～264 行

# 讀 /0/data（活 counter）
Twalk(fid=0, newfid=1, ["0","data"])  →  Rwalk(2 個 qid)               第 267～272 行
Topen(fid=1, mode=OREAD)  →  Ropen(qid, iounit)                        第 273 行
Tread(fid=1, offset=0, count=4096)  →  Rread(counter 的文字內容)        第 274 行
Tclunk(fid=1)  →  Rclunk                                                第 276 行

# 讀根目錄項
Twalk(fid=0, newfid=2, [])  →  Rwalk(0 個 qid，複製 fid)                第 279 行
Topen(fid=2, mode=OREAD)  →  Ropen
Tread(fid=2, ...)  →  Rread(一串內層自帶 size 的 stat blob)             第 281～289 行
Tclunk(fid=2)

# 跨真 9P 寫，驗 write-through
Twalk(fid=0, newfid=3, ["2","hp","data"])  →  Rwalk
Topen(fid=3, mode=OWRITE)  →  Ropen
Twrite(fid=3, offset=0, "555")  →  Rwrite(3)                            第 293～296 行
Tclunk(fid=3)
Twalk(..., newfid=4, 同路徑) → Topen → Tread → 印出剛寫的值             第 298～300 行
```

一路下來只用了一個 `fid=0`（attach 拿到的根）當起點，`walk` 每次都衍生出新 fid 去指不同葉子，讀寫完就 `clunk` 掉——這就是 9P 的基本節奏：**先建立到根的把手，之後全靠 walk 在同一條連線上分裂出各種指向不同路徑的把手，各自獨立生滅**。

## 七、接 unipath：真碼在哪

| 訊息 | 在 `unipath_9p.py` 的位置 |
|---|---|
| 訊息框架編解碼 | `p2/p4/p8/ps`、`Reader`（第 46～61 行） |
| wire 收發迴圈 | `serve_conn`（第 209～226 行） |
| 逐訊息分派 | `dispatch()`（第 108～203 行），每個 `if mtype == T...:` 一段 |
| qid 生成 | `_qid()`（第 81～86 行） |
| stat 結構編碼 | `_stat_bytes()`（第 97～105 行） |
| TCP 建連（AF_INET，接前篇 socket 那節） | `serve()`（第 229～238 行），每連線一條 thread |
| 獨立真 9P client（活對話範例） | `selftest()`（第 242～302 行） |

已知的老實簡化（值得記著，日後要跟真 Plan 9 對齊或串 v9fs 出狀況時先查這幾條）：**唯讀**（`Tcreate`/`Tremove`/`Twstat` 全部 `EROFS`）、**fid 不跨連線共享**（每條 TCP 連線各自一份 `fids` 表）、**write 忽略 offset**（整段覆寫，非游標式）、**無認證**（`afid` 恆假設 `NOFID`）、**qid.version 恆 0**（不反映背景 tick 造成的內容變動）、**Topen 的 mode 不真的做權限檢查**（唯讀保護落在 `Env.write()` 那層，不在 open 那關）。這些都是「先跑通、能被核心真的 mount 起來」優先於「規格全合規」的取捨，跟 `AGENTS.md` 定的鐵律一致。

## 收攏

9P 的所有動詞——version 協商、attach 拿根、walk 查路徑、open/read/write 存取內容、clunk 放手、stat 問中繼資料——攤開後就是把前面幾篇講的 inode/dentry/fd 概念，逐欄位搬到位元組線上：`qid` 是 inode 身份、`walk` 是 dentry 查找、`fid` 是這條連線私有的 fd。差別只在 9P 把這套語意做成**跟傳輸無關的位元組協定**，所以同一份 `dispatch()` 邏輯，換個 socket 型別就從本機協議變跨機器協議——這正是 `unipath_9p.py` 能被 Linux 核心 `v9fs` 直接掛載、而不需要再寫一份新協議的原因。
