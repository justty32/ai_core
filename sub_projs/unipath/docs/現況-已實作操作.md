# 現況：unipath 已實作的操作清單

> 照實際程式碼盤點（非規劃）。給回頭對照用。承 [AGENTS.md](../AGENTS.md)、[OVERVIEW.md](../OVERVIEW.md)。
> 結論：**目前就一小把基本操作——而這一小把幾乎就是 Plan 9 的全部詞彙**，os-as-game 夠用。

## 已做的基本操作

| 操作 | 在 unipath 裡的意思 | 對應 shell |
|---|---|---|
| **walk / 導航** | 沿 `/0/1/2` 逐段走進路徑樹（list→數字、dict→字串鍵） | `cd`、`ls mnt/0/1` |
| **readdir** | 列一個節點的子元素 ＋ 約定檔（`data`/`ctl`/`status`） | `ls mnt/0` |
| **getattr / stat** | 問節點：目錄還是檔？多大？可寫嗎？ | `ls -l` 背後那下 |
| **open** | 存在性檢查、拿把手 | 每次 `cat`/`echo` 前核心自動做 |
| **read** | 讀 leaf 內容（`data`＝值、`status`＝狀態、`ctl`＝命令說明） | `cat mnt/0/data` |
| **write** | 寫 `data`（改值、write-through 到活物件）或寫 `ctl`（下命令） | `echo 5 > mnt/0/data` |
| **truncate** | 讓 `echo >` 能運作（echo 先截斷再寫） | 覆寫時自動觸發 |

**`ctl` 命令**（現行 `unipath_env.py`）：`set <值>`、`append <值>`、`del <鍵>`。（step 1 靜態版另有 `mkelem`/`rmelem`。）

**真 9P 訊息**（`unipath_9p.py` 處理）：`Tversion` `Tattach` `Twalk` `Topen` `Tread` `Twrite` `Tclunk` `Tstat` `Tflush`——就這 9 個。

## 刻意「沒做」的（os-as-game 多半不需要）

`create`/`mkdir`/`unlink`/`rename`/`chmod`/`chown`/`symlink`/`link`/`fsync`/`ioctl`/`mmap`/`xattr`/`poll`/檔案鎖——全未實作。控制走 `ctl`、狀態走 `status`，都用「多開一個約定檔」解決，而非「多一個動詞」（Plan 9 的省法）。

## 為什麼這樣就夠

**tick 本質只用兩個動詞**：對每個元素 `read` 出當前值 → 算 → `write` 回下一態；加 `walk`/`readdir` 讓它掃過整棵樹。
**「讀值、改值、走樹」——tick 要的就這三樣，全有了。** 其餘高階操作不做也能運轉，這正是選 9P 的省力所在。

## 各檔對照

| 檔 | 角色 | 實作的操作 |
|---|---|---|
| `unipath_env.py` | 執行態環境純邏輯 | stat / readdir / read / write / ctl(set/append/del) |
| `unipath_fs.py` | step 1 假靜態樹（FUSE） | getattr / readdir / read / write / truncate / open / ctl(set/mkelem/rmelem) |
| `unipath_live.py` | step 2 自身執行態（FUSE） | 同上，backing 換成活物件 |
| `unipath_pub.py`＋`unipath_mount.py` | step 3 跨 process（9P 形狀 RPC） | 上述操作經 socket 轉發 |
| `unipath_9p.py` | step 4 真 9P2000 | 上表 9 個 T-訊息 |
| `unipath_tick.py` | step 5 回合制 tick | 用 read/write/walk 掃樹做狀態轉移 |
