cllm — 一頁看懂

把 LLM 收成一支對外 C ABI 共享庫 libcllm.so（extern "C"）＋建在其上的 llm unix filter CLI。純 C++23，CMake+Ninja+vcpkg+glaze。

核心心法：唯一入口 llm_ask，一次吃全部、輸出走 handlers

三層交付物（由上而下依賴）

llm CLI
unix filter：prompt 走位置參數／stdin，答案吐 stdout。旗標＝反射 Client 欄位自動生成。不寫程式碼就能用。
▼ 消費
cabi.hpp
C++ 薄鏡像
llm::abi::Client / Request / Handlers。header-only，把 C 的麻煩換成 std::string／optional／function。
▼ 連結 libcllm
libcllm.so
對外 C ABI
扁平 struct ＋唯一入口 llm_ask。純 C 客戶端 #include "cabi.h"、-lcllm 即用。穩定介面。
▼ 內部地基
http + glaze
native HTTP 傳輸（Win=WinHTTP／Linux=libcurl，共用 file:// 離線 fixture）＋ glaze 編譯期反射 JSON↔struct。

llm_ask 的資料流

輸入 · 七塊正交欄位

prompt必填；沒給則讀 stdin（CLI） systemsystem role 指示（選填） schema結構化輸出（JSON Schema） tools工具定義陣列 media輸入圖／音（url 或 bytes） modalities想要的輸出模態＋參數 stream逐段 vs 整段
llm_ask()
組成一個 OpenAI 請求
阻塞；同 thread 回呼
可選 ctx：另 thread
cancel() / phase()

輸出 · 四個 handler

on_text文字：串流逐段／整段 on_tool工具呼叫（永遠拼完整才交） on_media產出媒體（如 audio） on_error傳輸／後端錯誤，一次

為什麼是「一個入口」而非四個函數：舊路四條彼此不可組合（要「串流的結構化輸出＋帶圖」就沒轍）；整合搬進唯一一處 build_body，四塊隨意組合。

phase 生命週期（ctx 可觀測）

IDLE→ WAIT→ STREAM→ DONE ｜或收在｜ ERROR CANCELLED

CONNECT／UPLOAD 本版未細分（保留給未來自持 curl）。回傳碼：LLM_OK=0、LLM_ERROR=1、LLM_CANCELLED=-1（取消<0、出錯>0、成功0）。⚠ 取消：非串流阻塞、只在派送前檢查得到；串流每塊都檢查、能乾淨中止。

最容易踩的坑

reasoning 模型別亂設 max_tokens：思考鏈與答案共用同一份預算，設太小（如 600）→ content 變空字串。打 reasoning 時乾脆不設。
schema 的 required 坑：沒列進 required 的欄位就是選填，後端常漏吐。required 是寫 schema 那方的責任。
model 欄位：本地後端可省（用已載入模型）；OpenAI 雲端必填，且 api_key 必給。
離線 fixture 從不回錯：file:// 永遠 200、欄位齊全。後端錯誤語意驗不出，得靠真後端。
