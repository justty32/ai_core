# 踩坑 · Windows 執行 / 中文

← [gotchas 索引](README.md)｜[common/README](../README.md)｜[INDEX](../../../INDEX.md)

Windows 側執行與中文相關的坑：終端碼頁、命令列參數、Smart App Control、Git Bash exec 位元。

- **Windows 中文：終端碼頁 vs 命令列參數，兩個獨立的坑**。① **輸出亂碼**＝終端碼頁非 UTF-8（繁中預設常 950）——對策：exe 啟動 `SetConsoleOutputCP(CP_UTF8)` 自救，或終端 `chcp 65001`。② **中文命令列參數壞**＝標準 `main(argv)` 拿到系統碼頁——對策：`wmain`＋`-municode`（本專案 `main.cpp` 這樣做），或中文走 stdin/檔案而非 argv。
- **Smart App Control（SAC）偶發擋掉剛編好的新 exe**：Windows 11 SAC On 時把**某顆新編、沒信譽的 exe 雜湊**判成 unknown→封鎖，報「應用程式控制原則已封鎖此檔案」。**這不是程式錯誤**。GCC 連結是**決定性的**——原始碼沒變，重連結會**重現同一顆被擋的雜湊**；`cp` 成新檔名也同雜湊照擋。真正可靠：**改變二進位內容**（`strip build/llm.exe` 去符號→新雜湊，一次放行；或真改一行原始碼再編）。`Unblock-File`（MOTW）無關無效。
- **Git Bash 跑剛連結好的 exe 報 `Permission denied`（exit 126），但 PowerShell 跑同一支正常**：這**不是 SAC**（SAC 兩個殼都擋），是 Git Bash 對「剛被 `gcc` 寫出、exec 位元狀態未穩」的 exe 偶發拒跑。**別誤診成 SAC 去重編**——改用 PowerShell/cmd 跑，或 `chmod +x`。判別法：PowerShell 跑得動＝非 SAC。
- **★ 下游 C++ 消費：Windows 全綠（`lab/cpp/run.ps1`）**（2026-07-20，mingw64 g++16.1.0）。① C ABI＋C++ 便利層（`llm.hpp`／glaze 反射 `llm_reflect.hpp`）在 mingw64 下**可 link 可跑**；`run.ps1` 就地組 `<cllm/…>` include layout（便利層＋cabi 頭進一個 `cllm/` 夾）＋`-I…/build/vcpkg_installed/x64-mingw-static/include`（glaze）＋`-L…/build -lcllm`，六大能力（ask／stream／`ask_as<T>`／tool／media out／media in）全綠。**繞過 `install-dev.sh`／pkg-config**（Windows 無 pkg-config）——`~/dev` 打包層在 Windows 尚未驗。② **shell-out 到 `llm` CLI 的單引號坑（已修）**：`popen("llm … --endpoint 'file://…'")` 走 `cmd.exe`、**單引號不是引號** → llm 收到帶字面引號的 endpoint → URL 解析失敗；另 filter 會讀 inherited stdin、不給 EOF 會卡。對策＝平台分支：Windows 用**雙引號＋`< NUL`**、POSIX 維持單引號＋`< /dev/null`（`lab/cpp/play.cpp` 已就地修）。
- **★ 下游 Python 消費（ctypes binding）在 Windows：三坑全治**（2026-07-20，`lab/python/run.ps1`，Python 3.14）。① **`.so` 寫死→已根治**：`bindings/python/llm.py` 現依 `sys.platform` 自動選 `libcllm.{dll,dylib,so}`（`LIBCLLM` 仍可覆寫整路）。② **DLL 依賴 `libwinpthread-1.dll`→已治**：libcllm（及 tools）改 **`-static`** 連結把 winpthread 包入，`objdump -p` 剩系統 DLL（KERNEL32／UCRT／WINHTTP）；ctypes 只靠 `LIBCLLM` 覆寫即載入、免 `os.add_dll_directory`。③ **〔曾記 open bug，已根治〕串流回空字串 ＋ `media`＋`modalities` 同傳崩潰（`0x9`）＝同一個 struct 錯位**：`llm.py` 的 `_Request` ctypes 鏡像**漏了 C ABI `llm_request_t.system`（prompt 後第 2 欄）**，其後所有欄位位移一個指標寬 → `stream` 旗標落錯槽（DLL 當非串流解 SSE fixture → 空字串）、`media`/`modalities` 指標讀錯位址（存取違規 0x9）。補 `("system", c_char_p)` 到第 2 欄後兩症全消、`play.py` 七步全綠。判別法：`llm.exe --stream` 打 fake_stream 正常但 Python binding 空＝binding 側的 struct 問題。**⚠ 全語言通病**（同 lisp 的 CFFI defcstruct 錯位，見 [SESSION-LOG](../../../SESSION-LOG.md)）——動 `cabi_request.h` 欄位時，9 個綁定的鏡像都要同步。
- **★ 離線 fixture 被 autocrlf 轉 CRLF → SSE 串流解成空字串**（2026-07-20）：`core.autocrlf=true` 且缺 `.gitattributes` 時，Windows 檢出把 LF fixtures 轉 CRLF。`test/fixtures/fake_stream` 是 SSE（`data:…\n\n` 事件邊界），塞進 `\r` → 抓不到邊界 → 串流回空（`fake` 是單顆 JSON、對空白寬容故照常）。判別法：`llm.exe --stream` 打 fake_stream 空、打 fake 正常＝fixture 行尾問題。**已根治**：`sub_projs/cllm/.gitattributes` 把 `core/test/fixtures/** -text`（當二進位、永不轉換）＋ `*.sh eol=lf`；既有工作區 `git add --renormalize` 或就地轉 LF。
- **下游 app 消費（handy 四工具）Windows 移植全綠**（2026-07-20）：**python-handy** 改 **list-form subprocess**（`_handy.py`、無 shell）——原 `shell=True`＋POSIX `shlex.quote` 在 cmd.exe 破；sibling 走 `sys.executable`、`.cmd`/`.bat`（如 npm 的 claude.cmd）包 `cmd /c`、`echo` dry-run 轉同層 `_echo.py` 蟬（Windows 無 echo.exe）；入口 `python <tool>` 或 `.cmd` shim。**cpp-handy** 的 `util.hpp`（`#ifdef _WIN32`）：run/capture 走 **`_wsystem`/`_wpopen`（寬字元）**——narrow `system()` 用系統碼頁（cp950）解命令 → UTF-8 中文 mojibake；`shquote` 改 cmd.exe 兩層轉義（MSVCRT argv-quote＋metacharacter caret）；`script_dir` 用 `GetModuleFileNameA`、sibling 解 `<name>.exe`；**中文 argv** 靠 `HANDY_INIT_ARGV()`（`GetCommandLineW`＋`CommandLineToArgvW` 還原 UTF-8，免 `wmain`/`-municode`，需 `-lshell32`）；mail 檔案路徑走 `fspath()`（寬字元 `filesystem::path`，中文檔名才落對，narrow 建構子走 cp950）。走 cmd.exe 的好處＝echo（內建）／claude.cmd（PATHEXT）／llm.exe／sibling .exe 全自然解析。兩者離線 dry-run 與真 `llm.exe` 端到端皆驗綠。
- **★ 離線 fixture 的 `file://` 在 Windows 要真磁碟路徑（`C:/…`），不是 MSYS 路徑（`/c/…`）**：`cli_smoke.sh` 用 `$(pwd)` 組 `file://` endpoint，Git Bash 的 `pwd` 給 `/c/code/…`（MSYS 式、磁碟無冒號），native `llm.exe` 的 `fopen` 開不了 → 所有「成功路徑」測試變 `exit=2 out=<>`、錯誤路徑照過（假象：像後端連不上）。**cllm 程式碼沒問題**——`llm.exe` 餵 `file://C:/…` 正常回 fixture。對策：smoke 在 MSYS 下用 `pwd -W` 取 Windows 路徑組 `$FX`（`uname` 判 `MINGW*/MSYS*/CYGWIN*`；已實作），Linux 不變。判別法：手動 `llm.exe 你好 --endpoint file://C:/…/test/fixtures/fake/chat/completions` 回台詞＝binary 沒事、是腳本路徑格式。
