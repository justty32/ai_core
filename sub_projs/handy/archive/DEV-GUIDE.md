# DEV-GUIDE — 結構整理參考（被動 / 按需取用）

← [INDEX](INDEX.md)

這是一份**被動參考**，**不貫穿日常每個動作**——只在你要**重構 / 整理結構**（拆大檔、分類雜亂資料夾、住戶長出子工作流）時拿出來用。內容＝**結構整理原則**（含四級成長軌跡），docs / code / 任何資料夾通用。

> **always-on 的鐵律**（試驗田不是基建、拿現成用腳本包、先執行再說、重構不改原意、未確認不 push/不開新工作、全繁中）常駐在 [AGENTS.md](AGENTS.md)，**不在本檔重複**。「要做什麼」由 [WORKFLOWS.md](WORKFLOWS.md) 派發。

## 結構整理原則（膨脹即拆 / 雜亂即分類）

兩個觸發、同一套方法：

- **觸發 A：單體過大** —— 某檔案、次級工作流、或原始碼大檔體量太大。**客觀門檻**（依內容性質分套訂）：
  1. **`workflows/` 流程文檔**單檔超過 **8192 bytes**（操作筆記性質，逼它精簡）；
  2. **使用手冊 / 產品文檔**（如 `README.md`）單檔超過 **300 行**；
  3. **程式碼**（`.fnl`/`.lua`/`.sh`…）單檔超過 **300 行**。

  門檻是**觸發檢視的訊號**、非硬性上限：本質不可分的單體可超標保留——其中已完成的一律移 `*/archive/` 凍結（不留現役夾、不拆）；已凍結的 `archive/` 不套此門檻。
- **觸發 B：資料夾雜亂** —— 某資料夾底下散落各種**不同用途**的檔案，混在一起難導航（試驗田 handy 尤其容易：住戶越丟越多）。

方法（兩種觸發都照做）：

1. **按內容／職責／用途語意分**，**不是** 1/2/3 硬切等分——拆出來的每塊要有單一清楚職責。
2. 放進**專屬子資料夾**（或既有領域檔）依用途歸類。
3. 子資料夾頂層留一個 **README / INDEX** 做導航；已有則更新。
4. 原本的頂層位置只留**精簡指標 + 指向 index 的連結**，不再塞細節。

## 一個住戶/工作流的成長軌跡（四級）

住戶不是一開始就長成資料夾，而是**按需要逐步長出來**——每升一級都是上面「膨脹即拆」的一次套用：

1. **單檔**：一個檔，**檔名即其意義**（如 `zhtw`）。把導引、結構、內容全揉在一起（檔頭註解當入口）。最輕量，多數住戶停在這。
2. **資料夾＋單 README**：膨脹到需要拆 → 變資料夾，但裡面**只有一個 `README.md`**，它一肩扛導引+怎麼用+結構全部（如 `llme/`）。
3. **資料夾內不只 README**：README 再膨脹 → 拆出獨立檔各司其職（`INDEX.md`、`session-log.md`、`gotchas.md`、內容檔…），README 退回只做「先讀我」。
4. **資料夾內有子工作流**：大到內部得再開巢狀工作流，各子工作流自己又從第 1 級開始長。

往哪一級走永遠由「是否膨脹/雜亂」決定，**不預先過度設計**（這正是試驗田的精神：需要才長）。**降級也成立**：子項消失就往回併級。
