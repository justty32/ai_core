# 一台搜尋機——gen_v0 引擎 ≅ realizer 的 director

← [galtxt INDEX](INDEX.md)

> **命題**：[gen_v0](gen_v0/README.md) 的 L1–L4 引擎（搜 act／座標）與 [realizer_l](realizer_l/README.md) 的
> [director](realizer_l/director.lisp)（搜風格）**是同一台搜尋機**，只是跑在不同粒度的搜尋空間上。
> 這條同構直接通往使用者的洞見：**若一切都是「粒度 n 的節點」，一個搜尋 kernel 就能搜任何粒度**。

## 同構對照

| 抽象步驟 | gen_v0（搜 act／座標） | director（搜風格／字面） |
|----------|----------------------|--------------------------|
| **搜尋空間** | 座標軸（topic/give/gap/ending）× beat 候選 | 每台詞槽的（模板 × 填充）組合 |
| **枚舉** | 明給軸鎖定、留空軸走 `*_order` | 每槽窮舉模板 × 填充笛卡爾積 |
| **產生式** | L2 `expand_grammar`（beat 序變體） | 模板本身即句法變體 |
| **護欄（硬剪枝）** | 門檻 gate／預算 caps／`requires` | 變項×動詞相配／收束護欄（canon） |
| **承接鏈** | `requires_pick`（前拍選了某句） | 母題同軸（會變項 承接 c2 母題） |
| **成本** | `prior ＋ history懲罰 ＋ intent ＋ ✕格定價 ＋ 變體成本` | `prior ＋ history懲罰 ＋ icost(intent)` |
| **取優** | 嚴格 `<`、先枚舉者贏平手 → 全域最小 | 嚴格 `<`、先枚舉者贏平手 → 每槽最小 |
| **確定性** | `*_order`＋嚴格`<`＋歷史是輸入 | 固定枚舉序＋嚴格`<`＋歷史是輸入 |
| **多樣生成** | 把用過句登記進 history 再跑（demo③） | `direct-n`：每輪 history 逼換措辭 |

一句話：**兩者都是「枚舉離散空間 → 護欄剪枝 → 成本排序 → 取最小 → 用 history 迭代出多樣」**。gen_v0 的 demo③（歷史入條件換次優）與 director 的 `direct-n` 是**同一個機制**。

## 唯一實質差異：耦合，不是機器

- **gen_v0 跑全域 DFS**：因為預算是**全場約束**（口頭禪≤2、問句≤1…跨拍耦合），貪心會漏最優，故必須整場一起搜、DFS＋下界剪枝。
- **director 跑逐槽獨立**：因為 `母題` 把跨槽耦合**解掉了**（母題同軸由母題參數保證、其餘護欄都在槽內），成本可分 → 每槽獨立取最小即全域最優。

這不是兩台機器，是**同一台機器、兩種成本耦合度**。若哪天 realizer 長出全場約束（例如「整場『……』上限 8」「反差拍恰一個」——正是 gen_v0 已有的 budget/leak），director 就得升回全域 DFS，程式碼會與 gen_v0 的 `fill_sequence` 逐行重合。反之 gen_v0 若某場軸間無耦合，也能塌成逐軸獨立。

## 通往「一個 kernel、任意粒度」

同構之所以成立，是因為兩層的**被搜對象在結構上同型**：一個「候選集合 ＋ 護欄 ＋ 成本 ＋ 承接」。這正是使用者定調的洞見——

> 八股／段落／句式／填充物／原子事實不是不同種類，**只是不同粒度**；都是同一種節點。

若把「act 節點」「句式節點」「填充節點」「原子事實節點」統一成**粒度 n 的節點**（各自帶：子節點/候選、護欄謂詞、成本、承接），則：

```
search(節點, 護欄, 成本, history) → 最小組合        ; 一個 kernel
gen_v0    ＝ search 跑在「act/座標」粒度（耦合成本→全域 DFS）
director  ＝ search 跑在「句式/填充」粒度（可分成本→逐節點）
未來 facts director ＝ search 跑在「事實/act 序列」粒度
```

**一台搜尋機，遞迴地跑在每一層粒度上。** 這就是把 `facts → director → act → realizer → 台詞` 整條管線塌成「同一個 search、不同粒度」的壓縮終點——與 [uniform_probe](uniform_probe/) 五路並行探索的統一節點表示是同一件事的兩面（表示 × 演算法）。
