# gen_v1 —— 事實層地基・設計定調（討論中草稿）

← [galtxt INDEX](../INDEX.md)｜前身：[gen_v0/](../gen_v0/README.md)（v1 引擎，八示範綠）

> **gen_v1 的核心手術：把「事實」與「敘事風格」切開。** 事實是死硬的、結構化可查詢的唯一真源；敘事風格是在其上呈現給玩家的投影。本檔先釘**事實層**；敘事風格層（realizer）之後再議。
>
> 狀態：設計討論進行中（2026-07-17 起），四根柱子已定，隨討論增補。

## 總架構

```
facts（事實庫：帶型別邊的 LOD 圖，唯一真源）
  ↓ query
行為層（GOAP／行為樹：走路？沉默？說話？——galgame 多半落到「說話」；薄，先佔位）
  ↓ 選中對話節點
對話 planner（← gen_v0 v1 引擎升格：搜尋空間從「句子」換成「說話行為」）
  act = f(個性, 意圖, 知識視圖, 目標, 場景, 歷史)
  ↓ act 序列（結構化說話行為，不是文字）
realizer（act → 字面：敘事風格層，之後再說）
  ↓
台詞
```

- **gen_v0 的 v1 引擎其實已是一台小 GOAP**：intent＝目標、beat＝行動、預算/定價＝行動代價、DFS＋剪枝＝planner。gen_v1 不是重來，是把這台 planner 的搜尋空間升格成 act。
- **設計原則承襲終點形態**（[corpus/固化/README](../corpus/固化/README.md)）：純程式執行、執行期零/極少 LLM、確定性、品質優先延遲不設限、LLM 移編譯期。
- **不追求完全擬真**：模擬必有抽象、必有缺失細節，無所謂——唯一要守的是**玩家不覺得矛盾**。

## 命名（暫定，未定案）

| 職責 | 暫定名 | 出處 |
|------|--------|------|
| 事實庫（死硬放在那邊） | **facts** / ledger | — |
| 決定「發生什麼」 | **director**（對話 planner 含在內） | 遊戲 AI 慣例（AI Director） |
| 決定「怎麼講」 | **realizer** | NLG 術語 surface realizer |

流程讀作：`facts → director → act → realizer → 台詞`。

## 柱一：facts / act / realizer 分層

**事實引擎產出的不是台詞**（連「不經潤色的台詞」都不是）——半成品文字會讓風格層在改寫時夾帶新事實，矛盾防線反而破功，且事實壓扁成文字一次即有損。產出的是**結構化說話行為（dialogue act）**：

```lua
-- act：一拍「發生了什麼」，不是「這句話怎麼講」
{
  speaker  = "秋穗",
  act      = "讓渡",              -- 行為類型：問/答/讓渡/洩底/岔開/沉默…
  content  = { object = "書#417", condition = "還的時候要告訴我感想" },
  stance   = { 表面 = "平淡", 底下 = "期待" },   -- 反差在這裡成立
  effects  = { relation_delta = ..., leak_budget = -1 },  -- 對事實庫的回寫
  refs     = { "事實#031", "事實#088" },  -- 引用了哪些既有事實
}
```

- realizer 只做 act → 文字的渲染，**不得斷言 refs 之外的事實**——矛盾只可能發生在事實層，而事實層是結構化資料、純程式可查。文字永遠只是投影，投影不會自己長出新事實。
- 對應 NLG 經典三段式：content determination → microplanning → surface realization。
- 對 gen_v0 的手術：**句庫從「候選句自帶事實」拆成「act 為鍵、字面為值的渲染表」**；history 記**事實/act**，不記字面（避免重複事件與避免重複措辭可分開約束）。

## 柱二：知識視圖（誰知道什麼、誰藏什麼）

每個角色看到的不是 facts 本身，而是自己的**知識視圖**：

```lua
facts = {
  靜態 = { 個性, 世界觀 },          -- 死硬，開場定死
  動態 = { 關係階, 場景狀態 },      -- 隨 act 的 effects 演進
  知識 = {
    秋穗 = { knows = {"事實#031"}, hides = {"事實#088"} },
    主角 = { knows = {"事實#031"} },
    -- 一階即夠：秋穗知道主角不知道#088（藏得成的前提）；不上完整 theory of mind
  },
  歷史 = { act 序列 },              -- 記事實不記字面
}
```

語料裡挖出的核心機制在此模型下**全部變成可計算**：

| 機制 | 知識視圖下的形態 |
|------|------------------|
| 洩底 | 一條事實從 `hides` 遷移到「已表達」的狀態轉移；「每反差恰一洩底」＝轉移計數 |
| 反差 | `stance.表面` 與 `hides` 內事實不一致——機械可判（丙-3 ③檔問題消掉一塊） |
| 矛盾檢查（最大宗） | 角色的 act 只能引用自己 `knows` 內的事實——一條查詢守住「說出不知道的事」 |
| 代行洩底（甲-4） | 第三者把**她的** `hides` 事實轉移給主角——記她的帳，預算歸屬自動可判 |

## 柱三：LOD ＋ canon 鎖（按需細化、呈現才鎖）

事實粒度**可大可小、按需細化**——敘事 LOD。粒度形式各領域不強求統一（氛圍標籤／區間／圖／具體值），統一的只有兩條紀律：

**1. 粗事實＝約束區間，細化＝在區間內取點**

```lua
事實#012   = { 層=粗, 值="90年代", 型=區間 }       -- 重點是氛圍，不是年份
事實#012.1 = { 層=細, 值="1996", 父=事實#012 }     -- 哪天真需要才長出來
```

不變式：**細化只能落在父約束內，不能推翻父**——「細化不出矛盾」的機械保證。

**2. Canon 鎖：矛盾只在玩家看過的投影上可觀測**

- 事實一經 realizer 呈現 → 鎖 canon，永不可改、只可再細化。
- 未呈現的內部事實 → 隨便改，玩家無感。世界可以為故事讓路（分佈沒鎖時，櫻花樹可以「一直都在那裡」）。
- 矛盾檢查對象從「全庫自洽」縮成「新輸出 vs canon 集」。

**細化＝約束滿足，不是自由發揮**（告白要在櫻花樹下→先查校園分佈有沒有櫻花樹）：

```
planner 拉需求（要一個告白級細度的地點）
  → 查 canon：校園分佈存在嗎？
    → 存在：在分佈裡找滿足氛圍約束的點
    → 不存在：連分佈一起細化（未鎖，可為故事讓路）
  → 取的點對既有 canon 做一致性檢查 → 通過才寫入
```

細化的內容生成是**編譯期 LLM 的完美崗位**（炸候選→約束過濾→人審入庫），執行期照舊零 LLM。

**粒度預算＝代價模型再現**：每條細事實都是未來一致性檢查的表面積，只在故事需要處付費——

| 實體 | 粒度預算 | 為什麼 |
|------|----------|--------|
| 女主心理/背景 | 最細 | 用戶意圖＝青春愛情，商品本體 |
| 關鍵場景（告白地點） | 臨頭細化到夠 | 重要時刻的沉浸 |
| 路人/非主角 | 粗，撐一拍即可 | 為故事服務，不為擬真服務；細了是純負債 |
| 年代/科技 | 氛圍級，被引用才細 | 重點是氛圍不是年份 |

## 柱四（地基）：事實庫＝一張帶型別邊的 LOD 圖

事實間的關係不只上下（LOD 父子樹），還有**橫向連結**——如 Linux symlink。三個 symlink 性質正是所需：

1. **引用不是複製**：共同回憶是一條事實、兩個 dossier 各放連結。抄本是長篇矛盾的頭號來源；連結使「一處改、處處同步」成為結構保證。
2. **讀取時解析**：連結指向**粗**事實是合法常態（「她的手機」→連「年代」），解析時拿當下最細的 canon 值；父事實細化後，引用方約束自動收緊。
3. **懸空連結合法**：連到還不存在的事實＝佔位需求，planner 解析到時觸發按需細化——與 LOD 的 lazy 同構。

圖帶來的新能力：

- **反向連結＝一致性檢查的傳播名單**：改動/細化沿「誰指著我」走鄰域即可——檢查面第二次縮小（第一次是 canon 鎖）。
- **邊帶型別**：`共享`／`導因於`／`排斥`。**排斥邊**（兩事實不可同真）把一致性約束表達成圖結構——矛盾檢查＝排斥邊兩端不得同入 canon。

**連結本身也是一種事實**——有 ID、有粒度、有形式，接受與節點事實同一套紀律：

- **可被引用**：連結能連到另一條連結（關係的關係）。
- **可入知識視圖**：「她喜歡他」這段關係本身就是可 knows／可 hides 的事實——洩底洩的常常正是一條**連結**，不是一個節點。
- **可鎖 canon**：呈現過的關係不可改，只可再細化。
- **細化方式與節點不同**：節點事實靠**分層**長樹（父約束內取點）；連結事實靠**加中繼段**長鏈——粗連結「她喜歡他」細化成多段「她欠他人情 → 那次圖書館事件 → …」。不變式同構：**多段鏈不得推翻粗連結的語義**（鏈是粗邊的展開，不是改寫）。
- **樹與多段共存**：同一條連結可以既有父（更粗的版本）又有段分解（更細的鏈）。兩種細化軸互不排斥。
- **導因鏈按需長出＝backstory on demand**：「為什麼她喜歡他」的因果鏈平時不存在，故事需要解釋時才補段——又一個編譯期 LLM 崗位（炸中繼候選→約束過濾→人審入庫）。

**連結可再與其他事實聯結——圖長成網，節點是基石，其上層層疊疊**：

- **統一表述**：每條事實（節點或連結）都有 ID；`refs` 可指向任何事實——網自然湧現。實作上始終只是一張 facts 表。
- **層數自然湧現**：節點＝第 0 層（基石）；連結的層＝所引用事實的最大層＋1。越往上越「敘事」：底層＝具體人事物，中層＝關係，上層＝戲劇結構。
- **戲劇結構本身可表述成事實**：三角關係＝兩條「喜歡」邊之間的張力邊（連結的連結）；誤會＝某角色知識視圖裡存了一條**錯誤版本**的連結——galgame 的核心戲碼全在高層網裡。
- **接地不變式**：任何事實沿 `refs` 下行最終必達節點層。懸在半空、永不接地的事實＝編造，機械可查——「基石」的機械形態。

（柱二的知識視圖本就是連結：`knows` 存 ID 不存內容——此柱是前三柱共用地基的挖明。）

## 查詢介面（草案）——planner 怎麼問這張網問題

三條介面原則：

1. **讀寫不對稱**：讀，處處可讀；寫，**只有兩道門**——執行期 `act.effects`（planner 選定 act 後回寫）＋編譯期細化管線（人審入庫）。所有不變式檢查（父約束、接地、排斥邊、canon 鎖）全架在這兩道門上，讀取端永遠拿到的是已通過檢查的網。
2. **一切讀取過視角**：`view("秋穗")` 或 `view("全知")`。替角色決定**能說什麼**→過角色視角；director 做**結構決策**（節奏、預算）→全知。視角不是子集過濾、是 **overlay**：`事實ID → (原樣 | 變體 | 不可見)`——「誤會」＝視圖裡某 ID 映到錯誤變體，查詢自動解析到該角色以為的版本。
3. **確定性**：所有查詢結果穩定排序（ID 序）；禁止 hash 序遍歷（Lua `pairs()` 之戒——gen_v0 已立過的紀律，這裡升成介面規格）。

六個動詞（Lua、條件全 lua table）：

| 動詞 | 簽名 | 用途 |
|------|------|------|
| `resolve` | `(id, view?) → 當下最細 canon 值` | LOD 讀取時解析（柱三/柱四的「連粗事實拿細值」） |
| `match` | `(pattern, view?) → {id...}` 穩定序 | 圖模式查詢：planner 的主力動詞 |
| `trace` | `(id, 邊型?) → refs 下行路徑` | 接地檢查、導因鏈/多段鏈展開 |
| `backrefs` | `(id) → 誰指著我` | 一致性傳播名單（柱四反向連結） |
| `speculate` | `begin() / apply(act) / check() / rollback()` | DFS 試探分支：effects 疊成 delta、不落盤——承接 gen_v0 預算 threading 的同一模式 |
| `refine` | `(id, 細度, 約束) → 細化 或 掛起` | 兩道寫入門之一；執行期解不出→掛起成編譯期工單 |

使用者兩例的形態：

```lua
-- 「給我她現在可洩的底」
match({ 持有者 = "秋穗", 態 = "hides" }, view("全知"))
--> 穩定序候選；洩哪條由 planner 的成本模型決定

-- 「這個地點細化到告白級」
refine("地點#校園後方", { 細度 = "場所級", 約束 = { 氛圍 = "櫻花" } })
--> canon 有校園分佈：在其中約束滿足取點
--> 沒有：連分佈一起細化；執行期解不出＝掛起工單（LLM 炸候選→人審）
```

`speculate` 是 planner↔facts 的咬合點：DFS 每個分支 `begin()`、把候選 act 的 effects `apply()` 上去、`check()` 一致性、退出分支 `rollback()`——facts 網在搜尋期間是 base＋delta 堆疊，選定整條 act 序列後才真正 commit 過寫入門。

## 未決（之後的討論）

- **realizer／敘事風格層**：使用者明言之後再談。
- **行為層（GOAP/行為樹）**：先佔位薄層，galgame 場景多半直落對話節點。
- **儲存**：現階段 lua table in-memory 即夠；真需持久化選 SQLite（嵌入式、單檔、確定性、合 C++/Lua 技術棧），MongoDB 不合（server、非嵌入）。**先定形的是查詢介面**，儲存引擎是可換零件。
- **命名定案**：director/realizer/facts 為暫定。
- 事實 ID 規則、版本化、act 類型全集、邊型別全集——動手寫碼時定。
