# 統一粒度節點 kernel — 設計註記（uniform_probe/a1）

> 一句話：**八股／段落／句式／填充物 不是四種東西，是同一種「節點」在一條粒度軸上的四個座標。**
> 本 probe 用一個 `node` macro 把河堤場景（《最後的櫻花雪》場景二）從最粗（L0 整場八股）
> 到最細（L5 canon 原子事實）**全部塌成同一種節點**，`sbcl --script uniform.lisp` 自帶 assert 可跑。

## kernel（唯一的 macro ＋ 幾個原語）

```lisp
(defvar *ns*  (make-hash-table :test 'equal))   ; id → (id level realize-λ)
(defvar *env* nil)                              ; 風格座標／母題（唯一的確定性輸入）
(defun k (id) (string-upcase (string id)))      ; 統一鍵：ASCII 折大寫、CJK 不動

(defmacro node (id level &body body)
  `(setf (gethash ,(k id) *ns*) (list ,(k id) ,level (lambda () ,@body))))

(defun lvl (id) (second (nd id)))               ; 粒度層級可查
(defun r   (id) (funcall (third (nd id))))      ; realize：遞迴組出子節點
(defun pk  (slot cands) (nth (or (cdr (assoc slot *env* :test #'equal)) 0) cands)) ; 確定性選擇
```

`node` 只做一件事：把 `(id 粒度 實現λ)` 塞進註冊表。**這個 λ 的 body 就是全部的表達力**——
- **葉節點**：body 直接求出字面。可含邏輯：`(pk ...)` 確定性選候選、可空槽（`認可` 候選含 `""`）、
  **母題導出**（`變樣句` ＝ `(format "那些熟悉的~a…" (r "母題"))`）。
- **內部節點**：body 內用 `(r "子")` 引用子節點再組合。可掛任意多條護欄／成本邏輯
  （`T-d1a` 組合前先跑 `(guard 母題 變化動詞)`；`canon地點` 葉自帶「不出 canon」護欄）。

葉與內部**沒有型別區分**，差別只在 body 有沒有 `(r ...)`。這就是「四分法塌成一條軸」的全部機制。

## 粒度軸用了哪些層級（各代表什麼）

| L | 舊四分法的名字 | 本 probe 的節點 | 種類 |
|---|----------------|-----------------|------|
| **0** | 八股（整場） | `場景` | 內部（引用五段） |
| **1** | 段落 | `段A…段E` | 內部（旁白框＋台詞槽依固定序） |
| **2** | （句式之上的槽位） | `旁1…旁4`、`a1…e1` | ★**旁白＝葉、台詞槽＝內部，同粒度不同種** |
| **3** | 句式 | `T-a1…T-e1`、`T-d1a/T-d1b` | 內部（字面骨架＋填充物引用） |
| **4** | 填充物／原子事實 | `收束動詞`、`母題`、`認可`…`不變的錨` | 多為葉；`不變的錨` 是內部（再往下拆） |
| **5** | （四分法沒有的更細層） | `canon地點` | 葉（原子事實・帶收束護欄） |

**六級連續軸**（assert 驗證 `(0 1 2 3 4 5)`）。兩個關鍵證據：
1. **L2 同時住著葉（旁白）與內部（台詞槽）** → 證明「是不是葉」與「粒度」正交，種 ≠ 粒度。
2. **L5 存在**（`不變的錨` 這個「填充物」還能再拆出 `canon地點`）→ 證明軸**不限四種、可任意深**。

## 自帶 assert（`sbcl --script` 全綠）

① 組出整場台詞（旁白框＋五段台詞）；② 確定性逐位元（兩次跑長度／逐字元／checksum＝8882283 全等）；
③ 粒度層級可查（`lvl` 逐點驗＋全軸連續）；④ 護欄會擋（風景×謝了 不相配、錨引入「咖啡廳」非 canon → 皆 error）；
⑤ 粒度軸可調（換 `*env*` 風格座標 → 同一棵樹確定性產出另一整場）。節點總數 46，全出自同一個 `node`。

## 這個統一表示 vs 四分法：好在哪、壞在哪

**好**：省掉四套資料結構與四套引擎——realizer 原本 `*scene*`／`*tmpl*`／`*fill*`／`*變項動詞*` 是四張表加專屬存取碼，
這裡只有一張註冊表加一個 `r`。**遞迴天然**：整場、段、句式、填充物用同一個 `(r "子")` 組合，沒有「哪層該用哪個函數」的接縫；
**粒度可任意加深**（L5 已示範），八股要再往上包一層「全劇」也只是多一個 L-1 節點，四分法則得硬開新類別。
**壓縮帶來智能**：邏輯（選擇／母題導出／護欄／成本）全收進節點 body 這一個位置，靠攏了使用者定調的「最少程式碼」。

**壞**：粒度扁平化**丟了型別檢查**——四分法時「填充物不可能引用段落」由結構保證，這裡任何節點都能 `(r 任何 id)`，
寫錯要靠 assert／護欄在執行期抓，不是編譯期。其次**旁白這種純葉被迫掛個 `level`**，語意上有點多餘；
且護欄邏輯散在各節點 body，全域一致性（母題同軸跨段承接）不像集中式那樣一眼可審。
結論：**探索期用統一節點（少碼、易長深）；固化進 llm_forge 時，值得在 `r` 上加一層「粒度只能引用更細粒度」的靜態護欄**，把丟掉的型別安全補回來。
