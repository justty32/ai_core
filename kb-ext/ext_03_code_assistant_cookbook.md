# ext_03 — 程式碼助手 Cookbook (第一目標問題實踐)

> **關聯檔**：`roadmap.md` (第一目標問題), `code_02` (idea 工具)

## 1. 核心工作流
本文件詳細描述如何利用 `ai_core` 現有工具鏈達成「資產工廠 -> 資產消費者」的程式碼輔助工作流。

## 2. 階段一：資產提取 (The Factory Phase)
**目標**：將框架知識固化為笨模型可消費的資產。
- **工具**：`idea.py` (綁定聰明模型)。
- **指令**：`idea scan_framework --dir ./my_framework`
- **產出**：
    - `snippets/`：常用代碼片段。
    - `schemas/`：API 調用規範。
    - `rules/`：Linter 級別的護欄。
    - `line_assistant_hints/`：標記哪些檔案的哪些位置通常需要修改。

## 3. 階段二：受約束生成 (The Consumer Phase)
**目標**：笨模型在精準位置填碼。
- **工具**：`sfc.py` + `line_assistant` (確定性函式)。
- **流程**：
    1.  **意圖翻譯**：用戶說「幫我加一個 API 路由」。
    2.  **位置鎖定**：`line_assistant` 掃描 `routes.py`，標記出 `### ROUTE_ANCHOR ###` 的行號（例如第 42 行）。
    3.  **受約束生成**：笨模型接收到 prompt：「在 `routes.py` 第 42 行插入一段路由代碼，請參考以下 snippet ... 只需回傳代碼內容」。
    4.  **驗證與寫入**：`sfc` 攔截輸出，檢查語法，若正確則寫入文件。

## 4. 飛輪在 Cookbook 中的位置
- 如果笨模型填錯了（驗證失敗），系統自動記錄失敗案例。
- 週期性地讓聰明模型分析失敗案例，並更新「資產提取」階段產出的 `rules/`。
- 最終，原本需要 LLM 的「位置鎖定」可以被固化成純字串匹配。

## 5. 實踐建議
- **錨點設計 (Anchors)**：建議在框架原始碼中保留明確的註釋錨點，大幅降低笨模型的「定位難度」。
- **行數助手 (Line Assistant)**：開發一個獨立的 CLI 工具，專門負責將「檔名 + 語義描述」轉化為「精準行號範圍」。
