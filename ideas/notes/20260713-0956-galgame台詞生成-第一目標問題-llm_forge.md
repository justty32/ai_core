## 一、為什麼 galgame 幾乎是為 ai_core 量身訂做

- **初心對死**：台詞高頻、低單價、延遲容忍高——**根本不該**每句燒聰明模型。正是「只剩便宜本地 <10B 模型」前提的理想消費者。
- **工廠/消費者分工天然**（roadmap §2）：聰明模型一次性產角色聖經（人設/口癖/禁語表/語氣/few-shot）＝耐久資產；笨模型高頻照著資產在（場景, 情緒, 好感度階）下生單句。
- **「洞」很乾淨**（§3.2）：一句台詞的產出鏈裡 LLM 只碰創意那一格，其餘全確定性——敬語/稱謂替換、姓名解析、情緒 tag、斷行、ruby、旗標/好感度增減、format tag 驗證。**這是 SFC 最初構想「一堆小命令行函式＋組合」的教科書級樣本。**
- **順手喂飽 7/8 那三件事**：每支小函式都有 `--metadata`（生成句那支 `nondeterministic`、替換/格式那些確定性）；一條台詞流水線正好是**複合呼叫路徑 ＋ `compose(hops)` caller 折疊**的真實場景；生成那支要 `input_schema`（吃啥上下文）＋ output `schema`（吐啥結構）——[缺口①](20260708-1040-一切皆檔案-list-統一描述子-複合路徑.md) 立刻有消費者。

**選題正確性的證明**〔使用者洞察〕：**LLM 本質就是對話機器人，galgame 台詞也是對話機器人。** 程式碼其實是**比較難鍛的鐵**（LLM 在天性的鄰域幹活，規則零碎）；對話是 LLM 的**本命域**，要固化的「詞彙＋規則」在對話領域**乾淨到會自己結晶**。

## 二、固化階梯（llm_forge 的核心機制）

〔提案，使用者共識〕固化 ＝把任務沿階梯往下踹（5→1），洞一格格縮小：

```
5 自由創作句        ← 洞最大，純 LLM
4 受約束生成        ← 護欄咬住
3 模板 + 槽填充      「{greeting}、{name}。{observation}」槽來自庫
2 規則分支          好感度/情緒 → 句式，確定性
1 純詞彙查表        口癖/敬語表，零 LLM
```

字面上就是 **Futamura partial evaluation**：LLM＝解釋器，角色聖經＋累積規則＝殘餘程式，固化＝拿 LLM 對「固定角色/世界」做部分求值，鍛出一把專用快刀。飛輪＝每次 LLM 老產同一類東西，就結晶成下一級規則/詞庫。

**開發策略〔提案〕＝把 §3.3「洞的大小 = f(模型能力)」反著跑**：第一刀挑**洞最小、護欄能完整驗證**的窄任務先轉飛輪（例：「生一句必須提到道具 X、字數 < N、敬語階 = Y」——正確性被護欄咬死、不需 judge 就能證明「笨模型＋鷹架 > raw 笨模型」），飛輪轉起來再把洞往純創作端擴大。

## 三、兩種固化，別混（關鍵裂縫）

| | 批次固化（工廠期） | trace 驅動的自動固化（飛輪） |
|---|---|---|
| 做法 | 聰明模型跑一次，刻意挖詞彙/規則當耐久資產 | 盯笨模型 trace，發現重複樣式，自動結晶成新規則 |
| 難度 | 容易，**完美 v0** | `roadmap.md §3.6`「手動 vs 自動」，標「這題優先」、**未決** |

〔決定方向〕**批次固化就足以轉第一圈**，最硬的自動固化引擎往後放。先當人肉鐵匠鍛幾把刀，證明「笨模型＋刀 > 生笨模型」，之後才問要不要讓爐子自己打鐵。

## 四、驗證：人錨 + 語料錨 + 評分級聯

### 4.1 人工錨定——但驗的是「爐子」不是「每一句出廠品」
〔決定〕台詞品質**只能人工驗證**（沒有 `ast.parse` 那種免費確定性驗收）；使用者「一目十行、讀中文超快」＝真資產。但位置要擺對：

- **執行期（消費期）**：笨模型產句 → 確定性護欄擋壞的 → 直接出。**沒有人。** 品質下限＝護欄。
- **固化迴圈（工廠期，離線）**：使用者掃一**批**生成句判好/壞 → 餵固化的礦，**不攔截 runtime**。

**眼睛用來煉刀，不是站生產線末端當 QA。** 前置一次、攤提到之後每一句。

**礦不是 ✓/✗，是「✗ 的理由」**〔決定〕：只留 ✓✗ ＝評分、固化沒東西可鍛。要抓**為什麼打叉**（「用詞 OOC」→ 新禁語；「太親暱」→ 新的階→語域規則），implicit 品味才變 explicit 規則。

### 4.2 快速標籤 rating 工具〔使用者需求 + 我方設計〕
使用者要求：**不想每句手寫評價**，要能滑鼠打勾/按鈕，但**留手寫（打字）餘地**。設計：

- 理由用**快速標籤按鈕**（`用詞OOC`/`語域不符`/`情緒錯`/`太長`/`崩人設`…），不打字。**通過＝一鍵；打叉＝一鍵叉＋一鍵標籤（最多兩下）。** 手寫框永遠留一個逃生口。
- **硬原則**：每個叉至少帶一顆標籤（一次點擊），可不打字——掃得快，礦不空。
- **自相似飛輪**：標籤面板自己也固化——常打的手寫理由畢業成新按鈕。**你點的標籤＝正在長出來的規則分類法**（`用詞OOC` 疊 → 禁語表；`語域不符` 疊 → 好感度階→語域規則）。固化階梯同時作用在台詞、與「評價台詞的詞彙」兩層。
- **技術落法**（守零相依）：`http.server` 起本機網頁（HTML/JS 內嵌、零相依、跨平台），點擊 POST 回寫 NDJSON。是 **trace log 之上的消費端 app**，不碰核心契約。建議做成「可點也可全鍵盤」（空白＝過、數字鍵＝標籤、`/`＝評論框）——快速分診鍵盤比滑鼠快。
- **標註形狀**（append 進 trace NDJSON）：`{"line_id", "verdict":"pass|fail", "tags":[...], "comment":"(選填)", "rated_by":"human", "ts"}`。
- 這支工具**可本週開工**。

### 4.3 語料錨——直接掏 galgame / 戀愛小說文本〔使用者提案〕
**這就是 roadmap §5「上層：讀框架原始碼 → 生資產（結構化抽取）」的 galgame 版**——語料＝那個「框架原始碼」。聰明模型（工廠）不憑空寫，是**讀語料、蒸餾成資產**，比從零編角色聖經可靠得多（抽取有 schema、可驗證、guard 咬得住）。三個作用：

1. **參考/few-shot 來源**：真台詞＝「這型別角色實際怎麼說話」的黃金標準。
2. **從另一側攻下驗證**——現在有**三個 reliability 訊號**：

   | 訊號 | 誰算 | 性質 |
   |---|---|---|
   | 護欄通過率 | 程式 | 下限，免費 |
   | **語料匹配度** | 程式（對語料算） | 資料錨、免費、新增 |
   | 人工接受率 | 人，一目十行 | 真品質，貴 |

   紅利：**禁語/允許詞邊界語料自動劃**——「語料裡從沒出現過的詞」＝資料生出的確定性 OOC 護欄，省下大半手工點擊。
3. **帶答案卷的 benchmark**：挖掉真場景一句、生成、跟原句比＝`§6`「對照組 vs 實驗組」的形狀但**有 ground truth**；配對主鍵、pivot 表，與 ATP v0 評測法同構。

**兩個錨的分工〔提案〕**：語料錨＝外部/穩定/統計＝**粗胚**（廉價廣泛 bootstrap 詞彙/規則/分布）；人錨＝內部/語意/會演化＝**精修**（本專案特定品味）。使用者不從零標註，是在語料生出的預設上做修正——**標註負擔大幅下降**。

**一條版權設計線〔提案〕**（固化階梯的一格）：蒸餾出的**規則/分布**＝乾淨鍛造品，隨便用/出；**逐字 snippet**＝生礦，當 prompt 參考/few-shot 可以，**逐字搬進出廠遊戲**踩版權。→ 優先把語料蒸餾成規則（往下踹），別把整句複製進 snippet 庫。

### 4.4 評分級聯——LLM judge 仍需要，且受治理領證
〔使用者糾正〕上一輪 Claude 說「人工＋語料可省掉 LLM judge」是 over-rotate；**judge 仍要**，且**自動評分系統很重要**。收回並和解：**人工不取代 judge，人工是讓 judge 可信的那個錨。**

**為何非要 judge——中間有一段誰都摸不到**：護欄抓硬壞（二元）、語料匹配抓風格離群（淺，統計像 ≠ 語意對）、人抓語意但貴且不可規模化。中間「**要理解才判得動的語意/情緒/連貫品質**」＝ judge 唯一守備範圍，填「統計上像」與「人點頭」之間那條縫。有它才能機器速度轉飛輪、人只抽查。

**真結構＝成本階梯的級聯**（鏡像生成側 §3.3 的「笨→聰明」升級；forge 兩隻手「打鐵/驗貨」同形狀）：

```
每句評分 = 便宜→貴，短路早退
  Tier 0  確定性護欄        免費   fail 立斃
  Tier 1  語料統計匹配       便宜   離群低分斃
  Tier 2  LLM judge(校準過)  中價   抓語意/情緒/連貫
  Tier 3  人工抽查          貴     校準 judge + 抓漂移（抽樣，非每句）
```

**judge 受同一套治理管（§3.4），這才自洽**：judge 是 `nondeterministic` 節點 →〔要素〕測試組＝人工黃金集、reliability＝與人一致率、漂移偵測＝人工抽查、可撤照＝某類判斷被便宜確定性 proxy 取代就踢下 Tier 0/1。**沒有人錨 → judge＝「LLM 評 LLM」無限迴圈；有人錨 → judge＝校準過、有證書、被抽查的儀器。** 人錨（4.1）＋ judge（4.4）＝一台可信自動評分機，缺一不可。

兩附帶結論：**(a) 時機＝現在（開機期）**——趁聰明模型便宜把 judge 校準出來＝「資產工廠」套用到評分本身，judge 是工廠期鍛的耐久資產；**(b) judge 也是固化來源**（第三次出現此自相似）——judge 老為同一理由斃句 → 結晶成 Tier 0/1 便宜護欄 → 工作往級聯下層遷移 → judge 只留守「還沒被固化的語意前沿」。**評分級聯自己也有飛輪。**

## 八、世界環境資訊：世界模型（真值）≠ context（LLM 投影）〔7/13 續談〕

### 8.1 核心原則〔決定〕
對話生成的上下文要有充分的「世界環境資訊」（說話者身分背景、關係、地點、當前狀態…）。但
**LLM 的 context 很敏感——即使是正確的世界資訊，也可能讓 LLM 產出錯的東西**。所以要做的不是
「把真值塞進 prompt」，而是**把世界模型翻譯成「LLM 用得順的 context」**：

> context 可**省略**部分世界資訊、甚至**故意用錯的資訊**——唯一目標是**讓 LLM 產出我們要的**。
> **context 為「輸出品質」優化，不為「對真值忠實」優化。**

### 8.2 為何「兩層分離」是收益而非 hack〔提案〕
- **世界模型（真值）**：權威、一致、可查的正規化表示；職責＝**讓整部作品的虛構跨全遊戲前後一致**。
- **context（投影）**：世界模型的**有損、可失真**投影，per-call、為當下那句輸出品質服務＝**一次性的謊**。

payoff＝**同時要到全域一致 ＋ 局部 LLM 操控**：改世界模型本身騙 LLM → 失一致性；堅持 context 不失真 → 失輸出品質；**分開 → 兩個都拿**（真值層乾淨、失真只活翻譯層、用完即丟）。

### 8.3 落在架構哪裡〔提案〕
- **翻譯層＝世界感知版的 context binding（元件2）＋ §3.2 prompt assembly**：一個 `world_model → context` 函式。
- **它是固化目標**：「哪些世界事實該進/省/扭」初期靠聰明模型或人判，久了固化成確定性 context 組裝規則、LLM 洞縮小（固化階梯同一機制）。
- **它的輸出型別＝生成函式的 [`input_schema`](20260708-1040-一切皆檔案-list-統一描述子-複合路徑.md)**（缺口①的真消費者）。
- **失真必須一等公民、被記錄**：原則**寧省略勿造假**（omission 先於 falsification）；真要造假就做成**具名、可重用、可稽核的轉換規則**，「這裡對 LLM 說了 X（假）因為比真值 Y 產出更好」要落 trace。呼應治理主線＝知道「在哪騙了 LLM、為什麼、多穩」，也讓除錯分得清「世界模型錯 / 翻譯過頭 / LLM 崩」。

### 8.4 世界模型的組織形式〔懸，使用者明說「還得再考慮」〕
需求：說話者身分背景，背景裡各類資訊物件互連（國家、學校 → 連到另一個檔）。使用者過往在別 repo 用
**markdown 連結拆檔** → **低效**（要解析 md、欄位無型別）。未來傾向 **Linux 檔案系統 ＋ symlink ＋ 腳本**
（偏好此），但「一些東西可能還是得放 sqlite」。

**我方判斷**：此偏好**與 repo 哲學一致**——[20260708 一切皆檔案 × list](20260708-1040-一切皆檔案-list-統一描述子-複合路徑.md)
正是「物件＝目錄、連結＝symlink、宣告＝`_meta`」；且該則結論「**靜態＝檔案 tier 專屬、動態走 DB/query**」
給出乾淨切分（也回答「一些東西放 sqlite」）：

| 資料 | 性質 | 放哪 | 理由 |
|---|---|---|---|
| **手寫世界設定**（角色/背景/lore/物件間連結） | 靜態、人讀、要版本史 | **檔案系統 ＋ symlink ＋ git** | git 免費給演化史；`cat`/`grep` 可檢視；symlink＝物件間連結；＝真值的家 |
| **runtime/衍生狀態**（好感度/旗標/已訪場景）＋**跨圖查詢索引** | 動態、高頻寫、要查詢/完整性 | **SQLite** | JOIN/filter（「A 國精英校好感>50」）、外鍵擋斷鏈、可變狀態不進 git |

**橋接**：SQLite 是**從檔案系統衍生、可重建的索引/狀態層，不是真值來源**——真值留檔案系統（git 版本化），
SQLite 給快查＋可變狀態。git 版本化的人寫真值 ＋ 快查 ＋ runtime 狀態三者兼得，正好落在 7/8 靜態/動態分界。
順帶治好 markdown-link 低效：目錄＝物件、單一結構檔＝型別化欄位、symlink＝圖，不解析 md。

**兩個護欄**：
1. **粒度**（7/8 硬邊界）：別把每 scalar 炸成一檔（inode 稅）。**物件＝一目錄＋一 data 檔（自身欄位）；FS 結構只承載「圖」（symlink）與分組。圖在 FS、資料在檔。**
2. **方法論剎車**：世界模型組織是使用者自己在 roadmap §1/§4 點名的「想得愈來愈大」陷阱。**別先蓋大 ontology**——由第一個生成切片拉動（一角色、一場景、最小世界資訊起手），讓生成任務告訴你哪些欄位/連結真需要。

## 九、方法論轉向、技術棧、LLM 接口地基、新 repo 規劃〔7/13 續談〕

### 9.1 方法論轉向〔決定〕
「一開始就定死規範是好理想，但執行很麻煩。**正確做法＝先執行再說，邊執行邊生規範、調規範。**」
＝llm_forge 套在開發流程本身：**規範之於程式碼，如固化規則之於 LLM 輸出——從執行裡挖出，不開工前立法。**
這也是對 core_handy 自身 [00_index.md](../../sub_projs/core_handy/notes/00_index.md)「工作節奏」的自我修正（那套是 spec-first：
一軸一軸攤定義→N 輪討論→全拍板才寫碼）。往後**別再複製九軸 N 輪會議才寫碼的儀式**。

### 9.2 技術棧〔決定/方向〕
少用 Python（太慢）、少用 bash；主力 **C++ ＋ Lua**；使用者手寫時用 **Lisp（Fennel / s7 / SBCL）**。
重心自然收斂：**C++ 效能核心 ＋ 內嵌 Lua VM 當通用腳本層**（Fennel 編譯成 Lua ⇒ Lisp 前端同 runtime，不必多養）。
語言中立的縫＝**LLM 的 socket 協定 ＋ 函式的 `--metadata` 文字契約**。

### 9.3 LLM 接口地基**已存在**〔事實〕
使用者想「先弄 C++、Lua 的 LLM 接口，底層 server-form」——C++ 那半 core_handy 已有能跑原型：
[examples/llm_entry.cpp](../../sub_projs/core_handy/examples/llm_entry.cpp)（＝roadmap 元件1），拼 `impl/{http,json,llm,rate,serve}.hpp`：
- **one-shot**：prompt(stdin)→completion(stdout)；**`--serve <sock>`：長駐 daemon，LLM 單資源循序佇列、共用 `RateMeter` 跨呼叫累計 consume-rate**；env config（OpenAI 相容 `/v1`，預設 ollama）；自述 `--metadata`（entries/network/uncertainty=50）。

**server-form 對 polyglot 棧是唯一合理解**〔提案〕：(1) LLM＝單例資源 → 一個 daemon 獨佔；(2) 一 daemon ＋ N 瘦客戶端，每語言只要「開 socket、寫 prompt、讀回覆」約 20 行 → **永不用每語言寫 LLM SDK**；(3) 所有呼叫穿過此 daemon → 是寫 NDJSON **trace log（近期焦點③）＝galgame forge 礦堆**的天生落點。
**協定岔路**〔懸，先不選〕：Unix socket 字串協定（現況、KISS） vs 本機 HTTP/OpenAI 相容 gateway（通用、curl/MCP 可打，較重）。→ 先用能跑的字串版，等消費者（生成句要帶 model/params/context 結構）逼著長。

### 9.4 唯一從零的一塊＝Lua
棧裡目前**無任何 Lua/Fennel**。兩角色：(1) 打 daemon 的瘦客戶端；(2) 取代 bash 寫 SFC 確定性小函式（敬語替換/格式驗證…）的膠水。
