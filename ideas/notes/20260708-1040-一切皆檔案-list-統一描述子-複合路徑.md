## 一、核心提議：list 的載體換成檔案系統

兩個哲學：
- **Unix**：file ＝被命名/定址/帶屬性的資料塊；namespace 就是檔案系統。streaming/device/socket 全 reify 成這種塊。
- **Lisp**：list ＝唯一結構，同像（code=data）。

**簡單結合**〔決定〕：把 list 的載體從 cons cell 換成檔案系統的樹——即「把 AST 攤平成目錄樹」。
- atom/leaf → 檔案；list/branch → 目錄；read（不 eval）→ `cat`；eval → 執行；讀宣告 → `--metadata`/檔案屬性；組合 → 路徑安排，pipeline → 有序目錄。

三個舊懸案在此交會被解：
1. **同像性**——上一則哀悼 Lisp 丟了同像性、只能用 JSON metadata 補 data 半邊；檔案系統用另一形式還你：目錄樹既能 `cat`（read 不 eval）又能執行。
2. **image 漂移**（上一則 §六 最痛的紀律題）——list-as-files 直接消解：list 本身就是檔案，**repo 就是 image**，沒有 image 要冷重建。
3. **SFC 檔案爆炸**——重新定性：不是 bug，是這套哲學的必然代價；要治的是「樹怎麼組織」，而 git-style dispatcher 本來就是目錄。

**粒度是硬邊界**：cons cell 幾 byte，檔案是 inode＋syscall＋最小 KB，`(+ 1 2)` 不可能攤三個檔。所以檔案化只在**粗粒度（函數間組合層）**成立。恰好落回上一則的邊界：
- **list-as-files ＝外圈的 list 表示；list-as-cons（image）＝內圈的。** 兩個哲學不是誰取代誰，是各據一圈，結合點正好是那條邊界線。

**深理論（〔懸〕，「太遠」）**：使用者開場命題——「事物變化的路徑需用數據塊表示，而數據塊本身也是變化規則的顯化，兩者可互轉」。它在系統裡已半成形：code→data ＝跑規則產出資料／固化 memoize／tracing JIT 的 trace；data→code ＝讀資料歸納成規則（LLM as macroexpander）。**兩者交會的具體物件＝內容定址 store（git object / Nix derivation）**——用內容 hash 定址的資料塊＝一次被凍住的計算。先做簡單結合＝替這個極限方向鋪地基。

## 二、目錄的求值模型：其實是 dict，不是 list

Lisp list 是**位置定址**；目錄天生用 string/路徑標示 ＝ **dict/alist（名字→值）**。所以求值需要補回 list 的兩個位置語意：
- `_exec`〔提案，名字暫定〕＝補回「誰是 operator」（car）：對資料夾求值時要執行的核心標的。
- `_order`〔提案，名字暫定〕＝補回「參數順序」（dir 無序，list 有序）。

**關鍵框架**〔提案〕：一個 **folder-as-callable ≈ 一次 Python 呼叫** `_exec(*ordered, **named)`——具名檔＝kwargs、`_order`＝挑哪些按什麼序當 positional `*args`。這又落回「Python ＝ dict-Lisp」那條線（上一則 §4.3）。而目錄同時有 name 定址（檔名）＋position 定址（`_order`），比 Lisp（只位置）和 dict（只名字）都richer——正是 Python 的 positional-or-keyword 參數。

**塌縮求值 ＝ shell `$(...)`**〔提案〕：「子資料夾當參數要塌縮」＝執行它的 `_exec`、抓 stdout、替換進來，字面上就是 command substitution 遞迴套在目錄樹上。值預設是 stdout 文字，可經 entry `format`/`schema`（7/3）升級成 JSON。→ 不用發明 eval，是形式化 `$(...)`。

**岔路二〔懸，我方傾向已給〕**：`_order` 只管順序，還是也管求值策略（strict/quote/lazy ＝ special form）？
- 我方建議：**`_order` 只做排序**（保持是一行 plain data）；「要不要塌縮/quote」不進 `_order`，而是**進被引元素自己的 `--metadata`**（元素自宣告「值/可求值」）。理由：讓求值策略是被呼叫方的宣告，避免 `_order` 膨脹成 DSL（KISS、不自訂 DSL）。使用者未明確追認。

**岔路一〔提案〕**：`_exec` 是 operator 本身（傾向此，像 `__main__.py`/`init.lua`），非「指向 operator 的指標」。代價：把 code/data 區別烤進結構（有 `_exec` ＝callable）；但 read 層次的同像性仍活（可 `ls`/`cat` 不跑）。

**附帶紅利**：資料夾 ＝可求值 list ⇒ SFC 的 store 就是這種樹、hub 掃描 ＝讀樹上 `_exec` 的 `--metadata`。SFC 檔案爆炸 ＋ hub 掃描兩題被同一目錄語意順手收掉。

## 三、三層架構與統一量尺

〔決定〕三層：
| 層 | 是什麼 | 開了哪些軸 |
|---|---|---|
| 檔案內部（碼） | 純核心 Lisp，表達式級（Python/Lua 符合 Lisp 哲學） | 幾乎不開軸（local/in-process/零延遲/全有全無） |
| 檔案系統 | Lisp ＋資料塊組織 ＋ process/資源管理 | state/state_dirs/resources/lifecycle/interruptible |
| 分散式 | 正式納入延遲、可靠性、安全 | ＋guarantee、reliability、＋新失敗語意 |

**統一視角**〔提案〕：三層不是三種介面，是同一個「具名函數 ＋ `--metadata`」介面在三個設定下的樣子；**往上爬一層＝多宣告幾個軸**。這保住 KISS。

**分散式為何是獨立一層**：不是因為協議（HTTP/MCP 只是 transport，最簡單的部分），而是**部分失敗（partial failure）**——本地是全有全無，遠端可以「只發生一半」，「它到底發生了沒」變無法回答。retry/idempotency/timeout 全是這句的下游。
- 背書：Waldo et al.〈A Note on Distributed Computing〉(Sun, 1994)：本地/遠端四本質差異＝延遲、無共享位址空間、部分失敗、並發。Deutsch 分散式八謬誤。
- 「用 stdout/streaming 把分散式當檔案系統延伸」＝ Plan 9 9P / NFS 幹過的，必然漏在延遲/可靠性 → 使用者「不太好」直覺正確。

**系統已蓋一半**：`guarantee` 軸（none<idempotent<transactional）＝部分失敗的詞彙；`reliability` 非軸欄位（7/3）＝節點可靠性；retry/guard ＋ let-it-crash（上一則 §五 OTP）＝補救機器；LLM Entry Manager ＋ RateMeter ＋ `nondeterministic` 軸＝已是活的層 3 客戶端。
- **Plan 9 學哪半**：命名/定址整合進檔案系統（遠端函數照樣有路徑/名字/metadata）；語意透明不繼承（層 3 呼叫顯式帶 timeout/retry/失敗模式）。**表面統一、契約不同（軸值變了）。**
- MCP 不是外掛：`--metadata` ＋ entry `schema`（7/3）產出的正是 MCP tool 需要的；**層 3 對 MCP 的相容 ＝把本來就在吐的 metadata 用 JSON-RPC serve 出去**。

## 四、統一描述子：一個形式，兩種載體

〔決定〕載體按粒度分：**`_meta` 用於資料夾**（at-rest 檔、`cat` 得到、進 repo，因組成是結構、不跑就能讀）；**`--metadata` 用於檔案**（即時生成，因行為只有問本人才知）。這從 read/eval、atom/list 二元自然掉出來。

〔決定〕使用者要把所有靜態＋動態**整合成統一形式**。〔提案〕草案骨架（四段固定、葉子開放）：
```json
{
  "identity": { "name":…, "path":…, "hash":"sha256:…" },   // WHO；path 統一容納 檔/資料夾/碼符號/遠端URI
  "static":   { "kind":"file|dir|function|endpoint", "type":{…}, "format":…, "output_schema":{…}, "size":… },
  "dynamic":  { "input_schema":{…}, "entries":[…], "lifecycle":…, "state":…, "guarantee":…, … 軸 },
  "reliability": { "value":0.94, "provenance":… }           // 活值，非軸
}
```
**同一骨架四種填法**：純資料檔填 static 為主；碼函數填 dynamic 為主（＝函數簽名）；遠端 endpoint 兩半＋reliability；資料夾 `_meta` 兩半由 `_exec`＋children 組出。**「統一」＝統一骨架（固定四段），不是把所有欄位塞給所有東西。**

**護欄**：統一「骨架」、葉子保持逃生口 dict（沿用 `type:{base,mime}`、schema-dict）。**骨架固定、葉子開放**，層專屬東西（HTTP status、inode…）沉進葉子。相對現在的 `--metadata` 只多兩件：**(1) 顯式切 static/dynamic 兩段、(2) 補 `input_schema`**。

**深理論回扣**：這個描述子＝「數據塊也是變化規則顯化」的最小實現——一個物件，static 面是資料塊、dynamic 面是被編碼的變化規則，`identity.hash`（內容定址）是兩面黏起來的釘子。

### ★ 可直接動工的 spec 缺口 #1：input_schema
現在的 entry `schema` 只描述**輸出**（7/3）。動態要含「參數形式」、MCP tool 也**必須**有 `inputSchema`。**「補全」＝給 entry 加一個對稱的 input schema**：一改三雕（填檔案系統動態格／解鎖 MCP／對稱 output schema）。

## 五、歸化（naturalization）：三層都投影到檔案表面

〔決定〕貫徹「一切皆檔案」＝存取入口全留在檔案系統；分散式節點、程式碼內部（**只公開想公開的 API，不用全部**）都變成檔案。這是「另外兩層是 adapter」從方向變動作。兩個例子：
1. 腳本包 curl 打某 MCP ＋`--metadata`（分散式→檔）＝元件 #3 shell-as-function ＋「brownfield 經 wrapper」＝層 3 的**消費**方向（上上輪 serve 的反方向；同一座 MCP↔描述子橋，兩頭走）。
2. C 城市裡把函數變 CLI 子命令、腳本包 ＋`--metadata`，或 C 內部直接配（像 `git submodule --help`）＝ SFC ＋ `_core.py` 的 `register_subcommand`/scoped `<sub> --metadata`。

**外衣兩等級**〔提案〕：
- **wrapper（外掛腳本）**：零侵入、brownfield 友善，但 metadata 手寫 → **漂移**。
- **native（源碼自答 `--metadata`）**：同址、不漂移，但要動得了源碼。
- 判準：brownfield→wrapper；自己擁有→native。

**漂移的解，順帶替 A4 投票**〔提案〕：能自我描述的源頭就 **derive 不 hand-author**——MCP 有 `tools/list`（自述 inputSchema）→ 外衣的 metadata 去問遠端生成；C native 靠同址。這也是資料夾 `_meta` 該從 `_exec`＋children **推導**（＝暫緩的 A4 組合軸推導）的同向證據。

**防爆炸紀律扶正成原則**〔決定，源自使用者原話〕：**檔案化是 opt-in、只在邊界**——函數**預設私有**（層 1、in-image、表達式級），只有需要跨邊界時才掙來檔案外衣。這是治理原則（拒絕為預設/LLM 零領地）套到檔案表面，才是 SFC 檔案爆炸的**真解**（不是「組織好爆炸」，是「只讓跨界的冒頭」）。

**紅線**〔提案，使用者未明確追認〕：**歸化介面，絕不歸化契約**。外衣讓遠端 MCP 看起來像本地檔（介面統一，好），但**必須誠實宣告** `resources:[network]`/`guarantee`/`reliability`/延遲，否則重建 NFS 漏掉的透明。表面可騙、契約不能騙。

## 六、複合呼叫路徑的屬性合成

問題：`sfc some_small_func --args`，若 sfc 與 small_func 屬性衝突（sfc 不 guarantee、small_func guarantee），屬性標示該說啥？＝ A4 被 SFC 逼上台面。

**直接答案**〔提案〕：不存在「一個融合值」。`sfc X --metadata` 該回**那條路徑的 scoped 值**（＝ `register_subcommand`/resolver 的存在理由），頂層 metadata 只描述 sfc-standalone。

**通則：合成規則每根軸不同，分四族**〔提案〕：
| 軸 | 合成法則 |
|---|---|
| guarantee | **meet**（取路徑上最弱的有主張 hop） |
| interruptible / dry_run | **AND**（全 hop 都要有） |
| state / nondeterministic | **OR**（任一 hop 有 → 路徑有；污染型） |
| resources | **聯集** |
| reliability | **乘積**（序列系統可靠度，0.99×0.95=0.94） |
| lifecycle / entries / schema | **外層呈現**（caller 面對最外殼） |

兩方向：污染型（join，存在即擴散）／最弱環（meet，缺失即拉低），**都往悲觀/誠實那邊倒**＝「契約不能騙」的遞迴。

### ★ 可直接動工的 spec 缺口 #2：`guarantee: none` 一詞兩義
必須拆：**主動 none**（我會在呼叫外加東西、能打破下游保證 ＝真 bottom，參與 meet）vs **透傳/無主張**（我只 `exec` 轉發、不碰這軸 ＝合成的 identity，靠**排除**實現而非標成 top——標 top＝謊稱最強）。不拆就無法對「sfc=none, func=idempotent」判斷該取哪個。此原則對每根軸皆成立：每 hop 對每軸，要嘛主張一個值、要嘛宣告透傳。

**完整模型（〔懸〕，對 v0 過重）**：每 hop 對每軸其實是**轉移函數** lift/pass/degrade（sfc 用 transaction 包非冪等下游 ＝ lift；爛網路 ＝ degrade），路徑屬性 ＝沿路徑 fold 轉移函數。v0 先只要「四族預設 ＋ passthrough/none 切分」，轉移函數完整版留給真有 lifter（guard/sandbox）出現時再開。

### 誰折疊 → 〔決定〕caller 折疊
使用者拍板：**caller 自己管理**——它知道自己透過 sfc 呼叫，去看 sfc 是否影響它對 small_func 的呼叫。優點：
- **統一了 dispatcher 與 pipeline**：兩者都 caller 折疊，不用分兩套。
- **每個檔案/hop 只描述自己**，路徑是 caller 的 traversal 不是一個檔 → 最忠於「一切皆檔案」的 locality。

**必要的澄清/護欄**〔提案〕：
1. **別把最難推理丟給最笨 agent**：caller 就是笨模型、多跳折疊正是它弱項。「caller 管理」要精確成 **caller-side 確定性程式碼折疊，不是 caller-model 用腦折疊**——折疊機械（四族代數）＝確定性程式碼領地。⇒ 具體可 ship 物：一支確定性可重用的 `compose(hops)→effective`，擺 `_core.py` 旁；笨模型只消費折好的表、不需知道 sfc 存在。
2. **仍要求 hop 自宣告 transfer**：使用者「看 sfc 是否影響我」這句本身要求 sfc 暴露 per-軸效果（pass/degrade/lift）。「caller 折疊」與「hop 宣告 transfer」正交——選了前者不豁免後者（否則 caller 只能悲觀假設 degrade、冤枉乾淨 sfc）。
3. **與 LLM「邊界管」不衝突**：有跨 caller 共享狀態（rate/單例）→邊界管；只是折路徑屬性→caller 管。SFC 屬後者。

### 〔懸〕折到多深（使用者明說「之後再說」）
純 caller 折疊 ＋ hop 只自述 ⇒ caller 要遞迴走完整 transitive 路徑（`sfc→sfc→curl外衣→遠端`，N 次抓取、要完整 topology）。兩條路：
- (a) **誠實走到底**：最準，caller 背整棵深度。
- (b) **可驗證 horizon 快取**：hop 用同一支 `compose()` 把「它以下」預折成快取（像 `_meta`、像 hub「context budget 逐級收斂」），caller 折到 horizon 可停。**快取折疊 ≠ 擁有折疊**（演算法仍是 caller 的，caller 可不信、自己遞迴驗證），故不違反「caller 折疊」的決定。

## 七、本次逼出的待辦（彙整，非決定）

**兩個可直接動工的 spec 缺口**：
1. entry 加對稱的 **`input_schema`**（填檔案系統動態／解鎖 MCP／對稱 7/3 output schema）。
2. **拆分 `guarantee: none`**（主動 none vs 透傳/無主張），並推廣成「每 hop 每軸：主張值 or 宣告透傳」。

**一個確定性核心原語**：`compose(hops)→effective_metadata`（四族合成，caller-side，擺 `_core.py` 旁）。

**待追認/懸而未決**：岔路二（`_order` 只排序、求值策略進被引元素 `--metadata`）；歸化紅線（誠實貼標 vs 完全透明）；A4 組合軸推導（傾向 derive）；轉移函數完整模型（留待有 lifter）；折疊深度 (a)/(b)；深理論「互轉」（太遠）。

**統一形式相對現況只多兩件**：顯式 static/dynamic 分段 ＋ input_schema——其餘是既有欄位重新歸位。
