# 回到初心：把 LLM 當函數，更靈活更精確地使用

> 記於 2026-07-02。這是一次「回到初心」的定調筆記——在定義了九軸等一大堆東西之後，決定回頭把**最初的需求**做出來。

## 一、最初的目的（很簡單）

只是想要一個能**更精確操控 LLM 的使用方式**。

核心構想：
- 把 **LLM 當作一個函數**去使用。
- 用 **shell script** 去處理它的 input / output。
- 順便串接一些其他東西。

就在這個最初想法的基礎上，誕生了初版的 `helper`、`hub`… 等元件。

## 二、後來怎麼一路長大的（演化脈絡）

1. **從「函數」聯想到軟體架構**：既然都把 LLM 當函數了，那一些軟體架構的思想應該也能套用在它上面。
2. **想到 LISP 思想**：函數式組合的範式，拿來架構 LLM 的呼叫與組合。
3. **三個層次 (os / code / api)**：因為初版是 shell script 做的、依託於 OS，於是進一步發想出「連接三個層次來貫徹軟體範式」的方法。
4. **開始重構**：自此有了重構的想法。
5. **反覆討論、頭腦風暴發散**：一遍又一遍地發散想法。
6. **定義出九軸**，以及後續的各種東西。

## 三、現在的決定

在定義了這麼多東西之後，打算**回來，把最初的需求實現**：

> 將 LLM 當作函數，去**更靈活、更精確**地使用。

也就是說：先把發散出去的規範暫時放一放，回到「LLM as function + shell 處理 IO」這個最小、最原始、也最實用的內核，把它真正做出來、用起來。

## 四、更深一層的定調（2026-07-02 追加）

### 4.1 擺脫對 Python 的依賴
現在的 LLM 都可以用 **API** 去呼叫，那麼**所有語言都可以**做這件事。不必綁在 Python 上。

### 4.2 理想語言是 Lisp
理想的程式語言是 **Lisp**。

> 網友的話（很認同）：「現在會使用 LLM 的十倍工程師，在二十年前，就是那些 Lisp 巨集很厲害、很會用的傢伙。」

把 LLM 當函數、當可組合的元件來玩，本質上就是 Lisp/巨集那套「用程式生成與組合程式」的思維在新時代的對應物。

### 4.3 LLM ＝ 萬能函數（但有等級）
把 LLM 本身看成一個**萬能函數**：
- **缺點**：可能出錯、運行很慢、貴。
- **能力有高低差距**：
  - **厲害的 LLM** → 才真的是（用於文字處理的）**萬能函數**。
  - **SLM（小模型）** → 就當作**一般的自然語言處理**來用。

### 4.4 核心態度：把 AI 當「工具」而非「人」
想去**發掘 LLM 的工程性質**，而不是像現在普遍那種「把 AI 當人用」的使用方式。

> **我要把 AI 當工具用，我要去馴化他。**

這是整個專案的底層姿態：不是對話、不是擬人助理，而是把這個「會出錯、慢、貴的萬能函數」用工程手段馴化成可靠、可組合、可控的元件。

## 五、這正是「先規範、後實作」的理由（2026-07-02 再追加）

先前之所以堅持「**先定標準規範，再去做標準實作**」，這就是原因之一：**為了不受限於語言**。

- LLM 的呼叫**一定可以**弄成**跨語言 / 跨 API 的標準規範**。
- 所以正確的分層是：
  - **規範層（source of truth）＝語言中立**。九軸、`register/intercept`、LLM 呼叫契約，本質都是「規範」，不綁任何語言。
  - **實作層＝可抽換的載體**。Python 只是**參考實作**的宿主，不是規範本身；哪天換 Lisp，換掉的是實作，不是規範。

這也回頭解釋了 §4 的張力其實**不是真衝突**：`CLAUDE.md`「只用 Python 標準庫」約束的是**參考實作**，不是**規範**。規範天生就該語言中立；Lisp 是理想的實作載體，Python 是先跑通的參考載體，兩者都消費同一份語言中立規範。

## 六、一個很有趣的 tradeoff：性能考量退位（2026-07-02 再追加）

因為 **LLM 需要長時間計算**，很多**以前的性能考量會變得不那麼重要**了。

- 很多時候可以**「大力出奇蹟」**——用暴力、冗餘、多次呼叫換正確性，而不必省算力。
- **Python 慢也沒關係**、**記憶體消耗大也沒關係**：宿主語言的執行開銷，放在 LLM 的延遲/成本旁邊幾乎可以忽略。

**推論（很關鍵）**：離開 Python 的理由**從來不是「它慢」**。在 LLM 面前語言的快慢不值一提，所以實作語言的選擇可以**純粹由「表達力 / 範式契合度」來決定**——這正是偏好 Lisp 的理由（可組合、巨集、把程式當資料），而不是效能。

反過來看，這個 tradeoff 也讓一整類工程手段變得划算：retry、多數決 vote、best-of-N、蒙地卡羅取樣、冗餘校驗——這些「拿算力換可靠性」的招數，在馴化 LLM（把會出錯的萬能函數變可靠）時本來就是主力，而它們的成本在此語境下幾乎免費。

## 七、「貫穿三層級」的真正本意：統一接口 + 自我描述（2026-07-02 再追加）

先前提過「連接三個層次 (os / code / api) 來貫徹軟體範式」，這裡把它的**本意**講清楚——它不是「三層各做各的」，而是一個**極廣的抽象**：

### 7.1 統一接口，貫穿所有層級
想把不同**計算環境 / 運算平台**下的三種呼叫：
- **函數呼叫**（code 層，語言內的 function call）
- **shell 呼叫**（os 層，process / pipeline）
- **API 呼叫**（api 層，遠端服務 / LLM）

全部收斂成**同一個統一接口**，貫穿所有層級。對使用者（人或 LLM）而言，呼叫哪一層都是「同一種東西」。

### 7.2 差異不進接口，而是交給自我描述（`--meta`）承載
各平台、各計算環境之間**實際存在的差別**（在哪跑、要什麼資源、有沒有狀態、可否中斷、成本…）——不靠統一接口去表達，而是靠**函數的自我描述接口**去承載：

> 接口統一「怎麼呼叫」，`--meta` 描述「它到底是什麼、跟別人差在哪」。

這正是九軸 / `--metadata` 契約的**真正定位**：它不是附加功能，而是「把差異外掛到 metadata、讓接口本身保持統一」的那塊拼圖。等於在做一層涵蓋面極廣的抽象——一個**「函數界的 `.h` / lib」**：對外只暴露統一調用面，各實作的差異全寫在自我描述裡。

### 7.3 最終理想樣式：Lisp 式的 list element + 自我描述
理想中最終呈現的樣子類似 **Lisp**：
- **所有呼叫都是一個 list element**（不管底層是 function / shell / API），用 **list 的方式**去組合、巢狀、傳遞。
- 每個 element 都帶一個**自我描述接口**：`(meta XXX)`，或 `(XXX --meta)`——呼叫它就回傳這個 element 的資訊。

這把 §4.2（Lisp 是理想載體）、§二的「三層貫穿」、以及九軸 `--metadata` 契約**三者接了起來**：Lisp 的 `(f a b)` 天生就是「統一調用面」，而 `(meta f)` / `(f --meta)` 天生就是「自我描述面」——同一個 element 兩個面，正好對應「統一接口 + 差異外掛到 meta」。

## 八、意外的格局擴大：受益的不只 LLM（2026-07-02 再追加）

做完 §七那層抽象後，會發現一件事：**在這個統一接口 + 自我描述的基礎上，能被「很精細、很靈活地使用」的，不只是 LLM。**

- **最初的目的**：只是想更好地使用 LLM（§一）。
- **意外的結果**：一旦把「函數 / shell / API 呼叫」統一成帶自我描述的 list element，**任何工具**都套進同一個調用框架——LLM 只是其中一種 element，shell 工具、本地函數、遠端 API 全都是平權的 element。

換句話說：
> 我們是為了馴化 LLM 才建這套抽象，但抽象一旦成立，**它馴化的是「一切可呼叫的東西」**。LLM 從「目的」退位成「這個統一世界裡的一等公民之一」。

這跟 roadmap 的終點形狀其實是同一件事的兩種說法：roadmap 說「成熟系統裡 LLM 退縮到只剩翻譯層 + 模糊前沿」，這裡說「LLM 只是統一接口下眾多 element 之一」——都是**把 LLM 從舞台中央挪開、讓它變成可組合工具箱裡的一件工具**。這也再次印證 §4.4 的姿態：把 AI 當工具，而工具本來就該跟其他工具用同一種方式被拿取、被組合。

## 九、後設反思：這是不是一種新的軟體工程範式？（2026-07-02 再追加）

講到這裡會忍不住覺得：**我好像正在做一種全新的軟體工程範式。**——同時直覺也告訴自己：**應該有人過去想到過。**

誠實的判斷：它**大概不是全新的原子概念**，而是把幾條**既有的老脈絡**重新接在一起、再放進 **LLM 這個新語境**。這不減損它的價值——範式的創新往往就是「舊零件的新組合 + 新語境」。它明顯站在這些前人肩膀上：

- **Unix 哲學 / 「everything is a file」**：小工具 + 文字介面 + pipeline 組合。§7.1 的「統一調用面」是它的延伸。
- **Lisp / 同像性 (homoiconicity) / 巨集**：程式即資料、`(f a b)` 統一調用、用程式生成組合程式。§4.2、§7.3 直接繼承。
- **IDL / RPC / 服務自描述**（WSDL、gRPC reflection、OpenAPI）：用一份「描述」讓異質系統統一被呼叫。§7.2 的 `--meta` 是這條線的對應物。
- **Language Server Protocol / capability negotiation**：接口統一、能力靠描述協商。跟「差異外掛到 meta」同構。
- **當代 LLM tool-use / function-calling / MCP**：把工具描述成可被 LLM 呼叫的 schema。我們的差別是**反過來**——不是「讓 LLM 呼叫工具」，而是「**把 LLM 也降格成眾工具之一**」（§八）。

那**真正可能是新的**、值得主張的差異點，或許是這幾個的**合流**：
1. **把 LLM 本身也塞進同一個統一接口**當一等 element，而非讓它凌駕於工具之上（§八）。
2. **語言中立的規範層**當 source of truth、實作可抽換（§五）。
3. **「性能考量退位」使得拿算力換可靠成為預設手段**（§六）——這是 LLM 語境才成立的新前提。
4. 明確的**治理姿態**：把 AI 當工具去馴化，用 `--meta` + 九軸給每個 element 發「身分證」（§4.4、§七）。

> 判斷：與其糾結「是不是全新」，不如把它定位成**「Unix 哲學 × Lisp 同像性 × 服務自描述」在 LLM 時代的合流**。這個定位既謙虛又精準，也剛好指出下一步該去讀誰的東西來對照、避免重造輪子（符合 No wheel-remake）。

### 9.1 Prior-art 掃描結果（2026-07-02，deep-research 查證）

跑了一次跨源、對抗式查證的 prior-art 掃描，**結論：這個「四合一」尚無任何單一、命名過的來源完整提出，屬於既有零件的新組合。** 四條線各自都有成熟且命名過的 prior art：

- **① 統一異質呼叫成單一接口**：gRPC + Protocol Buffers（IDL/RPC，遠端偽裝成本地）、Nushell（結構化管線、plugin 共用資料模型）、MCP（單一 JSON-RPC `tools/call` 統一異質後端）。**最接近者＝ToolRegistry**（arXiv 2508.02979，明文 *protocol-agnostic*，統一 Python 函數/MCP/OpenAPI/LangChain）——但它的「協定」只指 LLM tool-calling 慣例，**不含 shell/process 或 Plan 9 式跨環境**，只坐實第一支柱。
- **② 自我描述 + capability negotiation**：MCP base protocol 內建 client/server capability negotiation 握手，工具靠 `tools/list` 回傳 JSON-Schema 自我描述（`inputSchema`/`outputSchema`/`annotations`）＝本構想 `--meta`／`(meta X)` 的直接類比；ToolRegistry 用型別 introspection 自動生 schema。
- **③ Lisp 同像性當組合基底**：**「From Tool Calling to Symbolic Thinking: LLMs in a Persistent Lisp Metaprogramming Loop」**（arXiv 2506.10021, 2025）——持久 Common Lisp REPL 讓 LLM 定義/呼叫/演化自己的工具，明文把 homoiconicity 當與 LLM 天然互補的組合基底。**是最接近本構想的單一來源**，但只覆蓋第 3 支柱 + 第 4 支柱的一半。
- **④ 把 LLM 降格為眾工具之一**：**這裡與主流分道揚鑣。** MCP 明文 *model-controlled*（LLM 居工具之上驅動 discovery/invocation）、ToolRegistry ＝ *tool-augmented LLM*、DSPy（arXiv 2310.03714）雖把 LM pipeline 抽象成可組合宣告式模組但 **LLM 仍是圖中核心節點、未被降格**。本構想的「LLM 降格成與工具平權的 element」與它們全部相反。

**一句話結論**：沒有單一命名範式同時涵蓋這四者；最接近的單一來源（Lisp metaprogramming 論文）也不碰跨環境統一呼叫接口。**四合一＝既有零件的新組合，這正是本專案的論述立足點。**

> **caveat（誠實標註）**：這是「缺席證明」，證據集偏重 2023–2025 LLM 生態。提問點名的**較舊 prior art**（CORBA/WSDL/SOAP、Plan 9、HATEOAS、Smalltalk introspection、gRPC reflection、PowerShell/Oil-YSH、LMQL/Guidance/LCEL）本輪**未逐一直接查證**。故「新組合」對近年 LLM 範式紮實，對「有沒有更老的靈魂」留餘地——不宣稱絕對原創。

## 十、範式的缺點，與「為何偏偏適合 LLM 領域」（2026-07-02 再追加）

補一個定位：**Unix 哲學是這個範式裡很重要的一環**（統一文字介面、小工具組合、pipeline），不是眾多靈感之一，而是骨幹。

但要誠實面對——這套範式**有兩個天生的缺點**：

1. **抽象的代價（效能 / 操作性流失）**：不同層級、不同環境、不同平台之間，終究有**不可弭平的差別**。強行把它們抽象成統一接口，一定會**失去一些效能，也失去一些貼近底層的操作性**（能做的細膩操作被統一介面磨平）。
2. **開發變麻煩（metadata 稅）**：用這套範式做專案其實**很囉唆**——每個 element 都要額外去寫 `--meta` / 自我描述那一套東西，是實打實的額外成本。

**關鍵洞察**：這兩個缺點，一旦把範式**限定用在 LLM 相關應用的開發**，就雙雙迎刃而解——

- **缺點 1 被 LLM 的慢抵銷**：LLM 本身計算就慢、就貴，應用裡其他部分的效能損耗**相對之下不值一提**（呼應 §六「性能考量退位」）。抽象磨掉的那點效能，在 LLM 的延遲旁邊約等於零。
- **缺點 2 被 LLM 的輔助抵銷**：寫 metadata 的囉唆，正好**交給 LLM 去做**。在 LLM 輔助下開發速度上去，「額外做自我描述」這件麻煩事的邊際成本被壓低到可接受——甚至這正是 roadmap「上層聰明模型讀框架生資產」要生的其中一類資產。

> 收斂：**這不是一套通用的、放諸四海皆準的軟體範式**——拿去做效能敏感或底層操作密集的一般專案，它的兩個缺點會咬人。它是一套**為 LLM 應用量身、且只在 LLM 語境下缺點被中和**的範式。這也回頭鎖死了本專案的**適用邊界**（scope），避免它被誤用、被無限擴張——正好呼應 roadmap「目標問題＝停止鍵」的方法論剎車。

## 十一、討論收斂：真正獨有的是「LLM-as-peer → 固化飛輪」（2026-07-02，與 Claude 討論後）

看完 §9.1 的掃描結果，使用者的反應是「這套還不夠獨創，但基底不是重點，蓋在上面的東西才是」。討論後收斂出幾個判斷：

### 11.1 別用「獨創」衡量基底
真正成為基底的東西（Unix「everything is a file」、Lisp S-expr、TCP/IP、關聯式模型）**沒有一個是原子級原創**，都是既有零件的一致組合。基底的價值＝「**讓什麼變得可能 + 多少人站上來**」，不是新穎度。糾結基底獨創性本身就是抓錯衡量軸——使用者的直覺（基底不是重點）是對的。

### 11.2 四支柱裡三支已被做掉，唯一結構性分歧＝第四支柱
掃描顯示：統一接口、自我描述、Lisp 組合這三支已分別被 **MCP / ToolRegistry / Lisp metaprogramming 論文**做掉。若只有這三支，本專案幾乎是在重造 MCP。**唯一真正逆主流的是第四支柱「LLM 降格為眾工具之一」**，而它不是風格選擇，是**結構性前提**：

- **MCP ＝ LLM-as-orchestrator**：LLM 是 host / runtime / 驅動者，**架構上拔不掉**。
- **本構想 ＝ LLM-as-peer**：LLM 只是圖裡一個平權 element，**正因可替換，才可能對它「固化 / 撤照」**——把穩定的 LLM 節點換成確定性程式碼。

→ 第九軸憑證、拒絕為預設、固化引擎**這整套治理，只有在 LLM 是可替換 peer 時才成立**。「降格」＝固化飛輪的結構性前提，不是美學。

### 11.3 要蓋的殺手級應用＝飛輪本身（熵減系統）
順此，該蓋的是**「只有 LLM-as-peer 才可能存在」的東西**——最強候選就是 roadmap 已有的飛輪：
> 一個會**自己把自己的 LLM 用量降到零**的系統。聰明模型生 element（初始帶 `nondeterministic:true`）→ 測試組下穩定 → 固化成確定性程式碼（撤照）→ 系統的 LLM 依賴隨時間**單調遞減**。

這個「熵減 / LLM 依賴遞減」性質，MCP / DSPy / ToolRegistry **結構上都給不出來**（它們的 LLM 在舞台中央、拔不掉）。一句話立足點：
> **「別人做的是 LLM 驅動工具；我做的是讓工具反過來逐步吃掉 LLM。」**

### 11.4 反方向提醒：小心重造 MCP
三支柱已被做掉的現實含意：**若從頭自寫統一接口 + 自我描述層，很可能是在重造 MCP/ToolRegistry**，違反 No wheel-remake。值得認真評估的選項——**直接站在 MCP 上，只把第四支柱（降格 + 固化飛輪）加上去**：把 MCP 當「統一接口 + 自我描述」的既成基底，原創火力全投在它給不出的那層。（待評估，非定案。）

## 十二、逐點回應與決斷（2026-07-02，使用者對 §十一討論的一次性回覆）

跟 Claude 做了三輪「盲點 / 市場共識 / 白地」的壓力測試後，使用者逐點給了判斷。以下按落地優先序整理——**這是使用者的定位與決斷，不是提案。**

### 12.1 立即要補強的（near-term，具體）
- **meta 要補「輸出的 JSON schema 格式」**：目前 meta 確實沒描述輸出格式，這是真缺口，要補。**順帶把「非文字內容格式」也補進去**（不只 text 的 I/O）。使用者判斷：補上這塊能一次解掉很多問題（呼應 §十一「接縫 typing」盲點）。
- **可靠度先做成一個「數值」欄位**：憑證/腐壞是後面的事；**目前的最低要求＝meta 輸出一個「可靠度」數值，可變動**。完整憑證機制之後再疊上來。
- **「誰讀 meta」＝已定案**：最初設計就是 **JSON 格式、除必要 key 外不多做限制**，目的正是**同時讓 LLM 讀、也讓確定性程式讀**。§十一那個「雞生蛋」fork，使用者的答案是「兩者都讀，用低限制的 JSON 兜住」。

### 12.2 已納入 / 確認為方向（多半進標準庫或排程器）
- **三種固化機制全包**：memoize / distill / codegen **三種都會有、都能用**——框架目標之一就是**靈活，想用啥用啥**（不收斂成單一機制）。
- **成本 × 穩定度的排程/拍賣＝核心思想之一**：讓排程器依此挑元件。原本預期「等基底打完，這用法會自動冒出來」。
- **雙向棘輪（de-固化）＝必須納入**：漂移時自動退回機率態，好想法。
- **圖拓撲即資產**：認同，smart model 不只填節點、也發現並凍結拓撲。
- **憑證 SBOM（AI 衍生碼物料清單）**：認同（帶法規順風）。
- **可靠度預算跨圖分配**：使用者評「神來之筆」，納入。
- **負空間認證（認證元件「絕不做什麼」）**：好想法，**可能放進標準庫**。
- **可觀測性**：有在考慮，**可能把「呼叫鏈歷史機制」放進標準庫**。
- **LLM 上下文累積**：不衝突——使用者的函數集裡**已有「對 LLM 上下文的處理與累積」的種類**（不是全都純函數，是提供多種上下文處理型別）。
- **延遲相加**：確實是問題，**會在後面排程器的規劃與實作中納入考量**。

### 12.3 立場澄清 / 對既有工作的定位
- **DSPy**：確實跟本構想很像，但沒關係——**本框架完成後會「包攬」其 prompt 優化功能**；DSPy 的做法可參考。
- **HATEOAS 的下場沒差**：`--meta` **本來就沒要求每次讀**，想怎麼用就怎麼用，不受 HATEOAS「動態發現輸給 hardcode」的教訓約束。
- **`LLM → shell`（injection/RCE 邊界）只是一種使用方式**：可用可不用；**標準庫會提供其他使用範式**，不強制走這條危險邊。
- **「everything is text」不成立**：使用者**沒有**做 text-only 的限制（配合 12.1 補非文字格式）。
- **inference-time compute（模型內部 best-of-N）＝只有部份有用**：模型自己做的那種花算力只解一部分，外層跨元件/跨工具/帶固化回饋的花算力仍有價值。
- **Bitter Lesson 不會失效——而是被飛輪滿足**：使用者的重要重構——**等飛輪轉起來，本身就「能隨算力擴展」，正好符合苦澀教訓所說的**。所以不是對撞，是「飛輪＝一種可隨算力擴展的通用方法」。
- **小模型會變聰明？——早就不信 LLM 與 benchmark**：使用者的賭注不靠 benchmark 敘事支撐。
- **部分求值 / Futamura ＝早已在構想裡（`ai_forge` 的鑄刀隱喻）**：使用者曾想把 `ai_core` 改名 **`ai_forge`**，意含就是**從 LLM 提取有用的短邏輯鏈——從一塊生鐵鑄造出一把鋒利的刀**。這個「萃取殘餘邏輯」的概念**已內含在 ai_core 中**（＝Futamura 部分求值的直覺版）。

### 12.4 明確押後（承認方向，但現在不做）
- **憑證的腐壞/過期機制**：是必須方向，但**後面再弄**；現在只要可靠度數值（見 12.1）。
- **測試組哪來 / 憑證細節**：都還沒定，**之後再說**。但心裡已有底稿——**大概會走 Sutton 那套（強化學習 / 探索 / 認知）的方式**去生成/篩選。
- **固化整體**：是**核心宗旨**，但會是**很後面**；前期算隱含目標。**前期暫時以「更好地使用 LLM」為主。**
- **燃料窗口（smart model 消失後無法再固化的穩態）**：太遠，先不管。

> **一句話收束本節**：近期焦點＝**把 meta 的 I/O 格式（JSON schema + 非文字）與可靠度數值補起來、把 LLM 用得更好**；固化/憑證/腐壞/測試組是已認領但押後的縱深；排程器（成本×可靠度×延遲）、雙向棘輪、拓撲即資產、SBOM、負空間認證則是「基底打完會自然長出來 / 進標準庫」的既定方向。**核心宗旨仍是 `ai_forge` 式的『從生鐵鑄刀』——只是先讓爐子能穩定燒起來。**

## 十三、換腦重想：整套系統＝「語意的 tracing JIT」（2026-07-02，Claude 換模型後的再綜合）

把 §一～§十二 全部攤開重看一遍，發現先前三輪討論給出的東西是**零散的**——雙向棘輪、負空間認證、測試組來源、三種固化機制、可靠度數值……各自成立，但彼此沒有結構。這次找到了把它們**全部統一**的既有架構原型：

### 13.1 這套系統就是一個 tracing JIT——只是編譯的對象是「語意」

Tracing JIT（PyPy / LuaJIT / V8 那套）的運作：解釋器慢慢跑 → 觀察熱路徑、記錄 trace → 把熱 trace 編譯成快碼 → 快碼帶 **guard**（守衛檢查）→ guard 失敗就 **deoptimize** 退回解釋器。逐項對映：

| JIT 概念 | 本系統 |
|---|---|
| 解釋器（慢、通用、定義語意） | **LLM**（慢、貴、萬能函數） |
| trace 紀錄（觀察實際執行） | **呼叫鏈歷史**（每次 LLM 呼叫的 input/output/model 紀錄） |
| inline cache | **memoize**（輸入→輸出快取） |
| baseline compile | **distill**（蒸餾成 SLM） |
| optimizing compile | **codegen**（生成確定性程式＝鑄刀） |
| guard（編譯碼的前提檢查） | **負空間契約 + 輸出 schema 驗證**（「輸入在我被認證的域內嗎」） |
| deoptimization（guard 失敗退回解釋器） | **de-固化 / 雙向棘輪**（退回 LLM 元件） |
| tier-up 條件（夠熱才編譯） | **憑證准入**（夠穩才固化） |

三種固化機制不是三個平行選項，而是**同一條 tier 階梯的三階**；先前散落的白地全部各就各位。而且**每一塊都有三十年的成熟工程解可抄**——不是抄程式碼，是抄架構決策（No wheel-remake 的正確用法）。

### 13.2 由此導出的四個具體修正 / 收穫

1. **de-固化不需要複雜的漂移偵測**——JIT 早就解了：給每個固化物掛 **guard**（便宜的執行期檢查：輸入是否在認證域內、輸出是否過 schema）。guard 失敗 → deopt 退回 LLM → 視情況重新 trace。**輸出 schema 驗證就是最簡單的 guard**——所以 §12.1 的「meta 補輸出 schema」不只是接縫問題，它是在蓋日後 deopt 機制的地基。
2. **「測試組哪來」有了具體機制**：tracing JIT 的答案是——**測試組＝錄下來的真實流量**。trace log（聰明模型實際跑過的 input/output 對）就是認證 codegen 的測試組原料；再配 **shadow execution**（SRE 的影子部署：同一輸入同時走固化碼和 LLM、比對、更新可靠度）就能**線上認證，不需人手寫測試組**。這與使用者 §12.4 的 Sutton RL 底稿吻合：**排程器的 explore 就是認證資料的收集迴路**——認證不是離線流程，是排程的副產品。
3. **可靠度數值的精確身分＝bandit posterior**：它不是靜態標籤，是隨每次觀測更新的後驗估計。推論一：它**必須帶 provenance**（至少一個「在什麼上量的」指標），否則排程器吃到的是無條件機率、垃圾進垃圾出。推論二——**組合深度定理**：鏈可靠度≈各節點乘積，0.95⁵≈0.77——**馴化不是品味，是組合深度的數學前提**；不固化（或不驗證）就無法深組合。這給 §八「統一抽象的價值」補上了硬論證。
4. **近期優先序升級**：§12.2 裡「呼叫鏈歷史＝考慮中」應升格為**承重的近期件**——**飛輪的第一圈就只是「把每次 LLM 呼叫記下來」**（entry manager 順手做，幾乎零成本）。trace log 同時是：memoize 的表、測試組的原料、可靠度數值的計算來源。使用者挑的兩個近期焦點（輸出 schema、可靠度數值）恰好是 tracing JIT 開工的頭兩塊地基——直覺選對了，這是體系一致性的佐證。第三塊就是 trace log。

### 13.3 兩個舊結論的修正（誠實自我更正）

- **「站在 MCP 上 vs 重造」是假二選一**（修正 §11.4）：MCP 是**傳輸/發現層**，本系統是**認證/編譯層**——正交、不競爭。正解：把 MCP 工具包成**一種 element adapter**（眾多可呼叫物之一），既不重造它、也不被它的「LLM is host」世界觀綁架。
- **「LLM 降格為 peer」可以再精確**（修正 §八/§十一）：在 JIT 視角下，解釋器不是老闆也不只是 peer——它是**參考實作 / 語意基準（semantic oracle）**：定義每個函數「正確」是什麼，所有更快的 tier 都從它導出、對照它驗證。「降格」的精確語意＝**從駕駛座降到基準面**。這比「peer」更能說清楚為什麼 LLM 拔不掉也不該當家。
- **附帶承認**：先前宣稱的兩塊「白地」縮水了——雙向棘輪≈JIT deopt 的既有機制、「spend-to-stabilize」≈JIT 攤銷編譯成本的老邏輯。**但換來的是整條工程路徑變具體、每一步都有前例可循**——用「原創性」換「可執行性」，划算。真正剩下的原創主張收斂成一句：**把 tracing JIT 的架構施加在「語意函數」上，以 LLM 為語意基準、以憑證為 tier-up 閘門**——這個施加本身，目前沒人做。
