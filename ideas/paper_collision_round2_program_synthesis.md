# 論文碰撞・第二輪：程式合成 × ai_core

> **碰撞主軸**：ai_core 的第一目標問題（程式碼輔助助手）正好＝程式合成社群的兩個經典子問題——
> **上層「讀框架→生資產」＝結構化抽取／庫學習**，**下層「精準標記位置填碼」＝受約束生成（programming-by-example）**。
> 程式合成圈這十年的硬核成果，幾乎都能直接對接 ai_core 的具體待決規範（`roadmap.md §6/§7`、`DECISIONS.md` A4/B 系列、ATP v0）。
>
> 一個貫穿全篇的關鍵對位：ai_core `roadmap.md §3.2` 把系統描述成「**確定性前濾網（matcher）→ 命中走純程式碼；落穿才掉進 LLM（cache-miss / `switch default`）**」。
> 用程式合成的語言講，**matcher＝歸納（induction，可執行可驗證的程式），LLM 留白＝轉導（transduction，直接生成）**。
> 也就是說——**ai_core 的核心架構，本來就是 BARC [[2411.02272]] 那組「歸納⊕轉導」二分的工程化身**。這讓下面 7 個火花有共同地基。

## 火花 1：歸納⊕轉導雙軌填洞——把 ai_core 的「前濾網/LLM」二分做實

- **來源技術**：BARC [[2411.02272]] 的核心經驗結論——即使**同資料、同架構、同先驗**，歸納（先合成可執行程式 f 再套用）與轉導（神經網路直接預測輸出）所能解的題**幾乎不重疊、跨種子穩定（p<0.004）**；分工是「歸納擅精確計算與多概念組合、轉導擅模糊知覺」。實用配方＝**歸納為主（可驗證、知道何時成功），無解時轉導兜底（總有答案）**。

- **ai_core 問題**：`roadmap.md §5` 下層「在精準標記位置填碼」目前只設想「笨模型直接生」這一條轉導路；但 §3.2 同時又講「前濾網命中走純程式碼」。兩者其實是 BARC 的兩條軌，卻沒被當成**可集成的互補對**來設計。

- **具體做法**：把「填洞」明確拆成雙軌並按 BARC 配方集成——
  1. **歸納軌（可驗證、優先）**：先讓確定性 matcher / snippet 庫嘗試（命中即＝「合成出的 f 在範例上相符」），對應 ATP 的 `pruning.strategy` 與既有 snippet 注入。命中就走純程式碼、**零 LLM 成本**。
  2. **轉導軌（兜底）**：matcher 落穿才叫笨模型直接生 patch。
  3. **集成裁決**：沿用 BARC「歸納為主＋轉導兜底」。關鍵差異——歸納軌的命中**可被 `ast.parse`＋簽名確定性驗證**（ATP V0 已有），所以「知道何時成功」這個 BARC 賣點在 ai_core 是現成的。

- **效益/風險**：效益＝給 §3.2 的「洞」一個有理論背書的填法，且歸納軌天然省錢（呼應 §2 資產工廠/消費者）。風險＝BARC 的互補性是在 ARC 知覺任務上測得，「程式碼編輯」場景的歸納/轉導互補比例未知，需用 ATP 的 `tier×method` pivot 表（`roadmap.md §6.1`）實測互補度，別假設照搬。

- **roadmap 落點**：`§3.2`（把前濾網/LLM 正名為歸納/轉導雙軌）、`§5 下層`、ATP `skeleton.py + line_assistant.py` 之間加一層「歸納軌優先嘗試」。

## 火花 2：多候選 + 貴智能偽標籤篩假設——受約束生成的裁決層與成本梯度

- **來源技術**：SYNTRA [[2509.17393]] 把程式合成重述為「**在『程式輸出定義的有限假設類』上做主動學習**」：枚舉一批與訓練對一致的候選程式（假設），用 **LLM 當 transduction 模型**對精選測試輸入產高準確偽標籤，**逐一淘汰不一致的假設**；再用 **greedy maximin**（選「最壞情況下能淘汰最多假設」的輸入）把超出下界的 LLM 呼叫**腰斬**。任務準確率最高 +196%。

- **ai_core 問題**：`roadmap.md §3.3 / §7` 的「先笨模型想、沒過邏輯測試再升級聰明模型」目前是個口號，缺**具體裁決機制**；而 §2 的「per-request 成本梯度」也缺一個「何時、用多貴的智能、問幾次」的省錢準則。第九軸證書（§3.4）的 `stability`（穩定度 %）也還沒有可操作的量測來源。

- **具體做法**：
  1. **笨模型生 N 個候選 patch**，全部過確定性閘（`ast.parse`＋簽名＋插入後目標節點存在，ATP V0 已有）→ 得到一個「都語法合法但語意可能不同」的**假設類**。
  2. **貴智能當偽標籤器**：只在「候選之間輸出分歧」的精選輸入上呼叫貴智能（OpenAIBackend/AnthropicBackend，`lib/llm_call.py` 已有），產偽標籤，淘汰不一致候選——**貴智能只做裁決，不做生成**，這正是 §3「LLM 退縮」的精神。
  3. **greedy maximin 省錢**：用 SYNTRA 準則挑「最能拉開候選的測試輸入」，把貴智能呼叫降到下界附近——直接落實 §2 成本梯度。
  4. **副產品＝證書**：候選在篩選中的**存活率**天然就是第九軸 `certificate.stability` 的量測來源。

- **效益/風險**：效益＝把「笨想→貴裁決」從口號變成可實作的主動學習迴圈，且順手解掉證書 `stability` 沒來源的問題。風險＝偽標籤受貴智能能力上限影響（SYNTRA 自承會誤淘汰正確假設）；且「枚舉假設類」要先有夠好的候選生成器（笨模型生 N 個），N 太小則假設類貧乏。

- **roadmap 落點**：`§3.3 / §7`（裁決機制）、`§2 成本梯度`、第九軸 `certificate.stability` 量測；實作掛在 ATP 消費端 `sfc.py` 與 `lib/llm_call.py` 之間，新增一個「多候選 maximin 裁決」filter。

## 火花 3（旗艦）：wake-sleep 庫學習＝ai_core 固化引擎的現成藍圖

- **來源技術**：
  - **DreamProver [[2604.26311]]**：DreamCoder 式 wake-sleep 搬到形式證明。**wake** 用 LLM 解題、收集「可學習的中間定理」；**sleep** 把中間定理**語意分群（句嵌入＋K-Means＋elbow）→ 抽象出候選引理 → 結構相似度驗證 → 樹編輯距離去重 → 形式驗證 → LRU 遺忘維持 <100 條**。消融鐵證：移除 wake-sleep 精煉 104→76 解、再移除語意分群 76→53（**甚至低於無庫**）——證明「**緊湊抽象庫 > 龐大低階庫**」。
  - **SOAR [[2507.14172]]**：**hindsight relabeling**——任一採樣程式雖對原任務錯，但對「**它自己實際產生的輸出**」必然正確 → 化失敗為訓練對；「搜尋↔學習」閉環抬高整條 scaling 曲線。

- **ai_core 問題**：`roadmap.md §3.6 / §8` 標「**這題優先**」的**固化引擎**——飛輪的免費能量來自「把 LLM 老在處理的同類模糊案例凍成確定性 matcher/snippet」，但**誰來做、自動還是手動，至今未決**。同時「資產庫該多大、怎麼維護不爆炸」也沒答案。

- **具體做法**：把 wake-sleep 直接映成 ai_core 的自動固化引擎——
  ```mermaid
  graph LR
    subgraph WAKE["wake（既有飛輪）"]
      A["笨模型填洞請求"] --> B["ATP trace[] 落 NDJSON<br/>（成功/失敗都記）"]
    end
    subgraph SLEEP["sleep（新增：貴智能定期離線跑）"]
      C["掃 trace NDJSON"] --> D["語意分群<br/>找『反覆出現的同類洞』"]
      D --> E["貴智能抽象成<br/>候選 matcher / snippet asset"]
      E --> F["確定性驗證<br/>（重放歷史 trace 是否仍相符）"]
      F --> G["AST 正規化去重<br/>（呼應火花5）"]
      G --> H["併入資產庫<br/>LRU 淘汰最少用 asset"]
    end
    B --> C
    H -.下一圈更多洞被 matcher 接住.-> A
  ```
  - **資料來源是現成的**：ATP V0 已把 `trace[]` 落 NDJSON（`DECISIONS.md` 2026-06-21、`roadmap.md §6.1`），正好是 wake 階段的軌跡庫。
  - **hindsight relabeling 在 ai_core 的版本**：笨模型一個沒解原任務的 patch，只要過了 `ast.parse`＋簽名閘，它對「自己產生的 (錨點上下文 → 這段碼)」就是一條**合法 few-shot/snippet asset**——把失敗 trace relabel 成資產餵回庫（SOAR 化失敗為訓練資料的 ai_core 翻版）。
  - **「<100 條 + LRU 遺忘」直接回答資產庫維護**：LRU 淘汰＝§3.4 的**撤照**的資料層對應（少用的 asset 退役）。「緊湊抽象庫 > 龐大低階庫」這個被消融強力支持的結論，給「資產庫別無限長」一個硬依據。

- **效益/風險**：效益＝把 §3.6「自動 vs 手動」這道專案最硬的題，從「憑空猜」變成「照 DreamProver 配方落地」；且 wake 資料源（trace NDJSON）已存在，sleep 是純離線批次、用貴智能、低頻——完全契合 §2「資產工廠一次性、貴、稀有」。風險＝(a) DreamProver 引理庫是**領域特定**的（每個框架要各自演化一個庫，不可跨框架共享），(b) 語意分群需要嵌入模型——與 ai_core「純標準庫、零相依」硬約束衝突，**sleep 階段須允許借用貴智能 API 算嵌入**（屬「資產工廠」開機期，可接受），但不能滲進笨模型消費端。

- **roadmap 落點**：`§3.6 + §8`（固化引擎自動版的具體演算法）、資產庫維護策略（<100＋LRU）、`hub.py`（庫的聚集與檢索）。**這是把飛輪從比喻變成可開工管線的關鍵一塊。**

## 火花 4：兩層抽象 schema——上層生資產該抽成什麼形狀

- **來源技術**：TheoryCoder-2 [[2602.00929]] 用 LLM **in-context** 合成兩層抽象——**高層 PDDL 運算子**（抽象的前提/效果）＋**低層 Python 謂詞分類器與轉移函數**（把抽象 ground 到具體環境）；唯一人工輸入是**初始 prompt ＋幾個極簡、與實際環境無關的示例**，純粹用來**控制抽象粒度**（讓 LLM 不抽太細也不抽太粗）。跨 episode 重用、成長一個抽象庫。

- **ai_core 問題**：`roadmap.md §7` 的 **Gap B1（語意/用途欄位）**——八/九軸只描述「執行特性」（lifecycle/state/guarantee…），沒有「這個資產**做什麼、吃什麼、什麼時候該用它**」。生資產一定撞到這個缺口；ATP 雖給了 `origin/payload/target` 最小形狀（`DECISIONS.md` 2026-06-21），但「上層該抽出什麼結構」沒有設計依據。

- **具體做法**：照 TheoryCoder-2 兩層結構設計 asset schema——
  - **高層（語意/觸發層）**：類 PDDL 的「**前提/效果**」描述＝「這個資產適用於什麼錨點上下文（前提）、會產生什麼編輯（效果）」。這正是 Gap B1 缺的「用途描述」，也是 §3.4 證書要綁的「適應症」。
  - **低層（grounding 層）**：可執行的 matcher 謂詞（`reads X` / `is_pkg_mgr` 風格，布林函數）＋ snippet 模板。對應 ATP 的 `payload/target`。
  - **借「極簡示例控粒度」**：給貴智能（資產工廠）幾個與目標框架無關的極簡 asset 範本，當「抽象該多粗」的尺——避免它把整個檔當一個 asset（太粗）或每行一個 asset（太細）。

- **效益/風險**：效益＝給 Gap B1 一個有來歷的 schema 骨架（高層語意＋低層 grounding 兩段），且與第九軸證書、固化引擎（火花3 抽出的就是這種兩層 asset）天然吻合。風險＝PDDL 式前提/效果對「程式碼編輯」未必是最自然的抽象語言，可能需要換成更貼近 AST 的描述；「極簡示例控粒度」是工程訣竅、非保證，粒度仍可能漂。

- **roadmap 落點**：`§7 Gap B1`、ATP `asset.py` 的 `origin/payload/target` 擴充、`core_nature/` 未來的語意欄位規範。

## 火花 5：等式圖語意去重——讓資產庫與 memoize 不為「同義不同形」重複付費

- **來源技術**：eggp [[2501.17848]] 把編譯器的**等式飽和／e-graph** 接進 GP：用 e-graph 記錄整個搜尋史的所有已訪表達式（含等價形），靠 pattern matching 高效查「**某式或其等價形是否已存在**」，強制交叉/變異只產生「未曾訪問過」的式子。極簡修改即追平 PySR/Operon、模型更小更可靠。DreamProver 也用**樹編輯距離去重**引理。

- **ai_core 問題**：(a) `DECISIONS.md` 把 **memoize 定為純 runtime（`lib/memoize.py`）**，但 cache key 若是字面字串，「語意相同、字面不同」的請求會 miss、重複燒笨模型（§3.6「便宜≠免費」）。(b) 火花3 的固化引擎抽 asset 時，會大量產出**語意等價但寫法不同**的 matcher/snippet，庫會被同義條目灌爆，違背「緊湊庫」。

- **具體做法**：
  1. **memoize cache key 用 AST 正規化**：對輸入請求做語意正規化（變數改名、等價語法歸一）後再當 key——把 eggp「查等價形是否存在」搬成「查等價請求是否已快取」，提升命中率、省笨模型錢。純標準庫 `ast` 即可，不破壞零相依。
  2. **資產庫去重用等價類**：火花3 sleep 階段的去重（圖中 G 節點）用 AST 正規化＋樹編輯距離（DreamProver 法），把語意等價的 matcher 收進同一等價類，只留最小代表——直接服務「<100 條緊湊庫」。

- **效益/風險**：效益＝低成本（純 `ast`）顯著提升 memoize 命中率與資產庫純度，兩處都直接省 §2 的高頻消費成本。風險＝完整等式飽和會指數爆炸（eggp 自己只敢做「單步」折衷），ai_core 應只做**保守的 AST 正規化**（改名、常數摺疊、交換律歸一），別追求完備等價判定。

- **roadmap 落點**：`lib/memoize.py`（key 正規化）、火花3 固化引擎的去重步、`hub.py` 資產庫維護。

## 火花 6：執行引導填洞 + 條件獨立攤平——行數助手的理論背書與升級

- **來源技術**：GridCoder [[2411.17708]] 兩個洞見——
  1. **條件獨立攤平**：把自回歸 token 機率近似成「只依位置、不依先前選擇」，於是只需幾次推論就算完整棵樹——**把「決定填什麼」拆成低自由度的獨立小決策**。
  2. **核心警告**：搜尋只能補「網格層高熵 OOD」，**補不了「程式結構低熵 OOD（機率硬卡 0/1）」**；唯一解方是 **LTS（執行引導）**——把程式**中間執行狀態回饋進每步解碼**，proxy 實驗把結構泛化從 10%→80%。

- **ai_core 問題**：`roadmap.md §5` 的「行數助手精準標記＋模型只負責粗粒度決定填什麼、填哪」目前是直覺主張，缺理論依據；且笨模型填洞是**一次盲生**整段，沒有用上 ATP 已有的 `subprocess` 隔離執行（`isolation.py`）做中間回饋。

- **具體做法**：
  1. **條件獨立攤平＝行數助手的正名**：GridCoder 證明「把高自由度生成拆成多個低自由度獨立決策能大幅提速並提準」。ai_core 的 `line_assistant.py`（定**位置**）＋笨模型（定**內容**）正是這種拆分——這給 §5「自由度愈小笨模型愈不會搞砸」一個量化背書（GridCoder 79.3% > 不拆的 75.9%）。
  2. **執行引導填洞**：別讓笨模型一次盲生。改成**逐步**：填一段 → ATP `isolation.py` 跑 `ast.parse`／單元測試 → 把中間結果（報錯、缺哪個符號）回餵笨模型再續填。把 GridCoder 未實作的 LTS，用 ai_core 現成的軟隔離執行落地。

- **效益/風險**：效益＝(a) 給行數助手設計一個來自頂會的理論依據，(b) 用 ATP 已凍結的 `isolation.py` 把「結構泛化＝靠執行引導」這條 GridCoder 主結論變現，可能是「raw 笨模型做不到、包鷹架做得到」（§6 驗收標準）最關鍵的一塊。風險＝逐步執行回饋會增加笨模型呼叫次數（與省錢張力），須與火花5 的 memoize、火花2 的 maximin 一起做成本控制；ATP V0 是「插入單一函式」的窄任務，執行引導的價值要等任務複雜化才完全顯現。

- **roadmap 落點**：`§5 行數助手`、ATP `line_assistant.py + isolation.py`、`§6 驗收標準`（這是讓對照組翻盤的候選關鍵技術）。

## 火花 7（旗艦）：ILP 學判別謂詞——自動固化「具體長出什麼 matcher」的演算法核心

- **來源技術**：
  - **AutoSpec [[2606.24245]]**：把「維護 LLM 智能體安全護欄」框成 **CEGIS（反例引導歸納合成）**：把**偽陰性當正例、偽陽性當負例**餵 ILP（ILASP），求「**常出現在 FN、罕見於 FP**」的**最具判別力謂詞**，把規則編輯的指數空間剪成一張短清單。關鍵分工——**ILP 只提名謂詞、不寫規則**；4–5 次迭代收斂、F1 達 0.98、規則**可讀可審計**，比啟發式 CEGIS 高 4.8×。
  - **Poker [[2507.16405]]**：自監督 ILP——**偵測矛盾**自動造負例，理論證明**正確率隨例子數單調遞增**。

- **ai_core 問題**：火花3 給了固化引擎的「庫維護骨架」，但**沒回答「自動固化具體怎麼從 trace 長出一個新 matcher」**——這正是 `roadmap.md §3.6` 自動版最難的一步。另外，§3.4 證書要的「**必要性**（確定性程式碼真的辦不到才放 LLM）」也缺一個可操作的判定。

- **具體做法**：把「固化＝合成新確定性 matcher」直接做成一個 CEGIS×ILP 迴圈——
  - **反例來自 trace**：掃 NDJSON trace，把兩類事件標成正/負例——
    - **偽陽性（FP）**＝「本該由純程式碼/既有 matcher 接住、卻誤掉進 LLM」的請求；
    - **偽陰性（FN）**＝「笨模型搞砸、本該升級貴智能或本可被某 matcher 攔下」的請求。
  - **ILP 提名 matcher 謂詞**：把上述當正/負例餵 ILP，求「最能乾淨分離兩類」的判別謂詞（如「錨點上文含 `class` 定義」「目標檔 import 了 X」）——**ILP 只提名候選 matcher，不直接定案**，提名後交確定性驗證（重放歷史 trace）。這就是火花3 圖中 `E（抽象成候選 matcher）` 的演算法實體。
  - **「必要性」閘＝可被 matcher 分離與否**：若 ILP 能找到乾淨分離某類洞的謂詞 → 該洞**不該再用 LLM**（撤照）；找不到 → 證明「確定性辦不到」，LLM 留白**通過 §3.4 必要性閘**。Poker 的「準確率隨例子單調遞增」對應證書 `stability` 隨 trace 累積而收斂。
  - **可審計性直接服務治理**：AutoSpec 學出的規則人類可讀——正是 §3.4「整個系統可稽核：永遠知道哪幾個點放了 LLM、為什麼」要的性質。

- **效益/風險**：效益＝(a) 補上火花3 缺的「固化的演算法引擎」，兩顆旗艦合體即覆蓋 §3.6 全題；(b) 把 §3.4「必要性」「撤照」「可稽核」三個治理口號全部變成 ILP 迴圈的可操作步驟；(c) 4–5 次迭代收斂、規則可讀，契合 KISS。風險＝ILP 求解器（ILASP/Popper）是**外部相依**，與 ai_core「純標準庫」硬衝——固化是**開機期、貴智能、離線**的資產工廠工序，可接受在工廠側引入求解器，但**產物（matcher 函數）必須是純標準庫**才能進笨模型消費端；謂詞庫仍需人預先設計（AutoSpec 每域 16–24 個），ai_core 要先有一組「程式碼編輯領域」的基礎謂詞種子（ATP 已有 `SENSITIVE_NAMES` 黑名單種子可作起點）。

- **roadmap 落點**：`§3.6 自動版`、`§3.4 治理`（必要性閘／撤照／可稽核）、第九軸 `certificate`、ATP `skeleton.py` 的 `SENSITIVE_NAMES` 擴成謂詞庫種子。

## 誠實排除：連續潛在搜尋（LPN / NLI）為何此刻不採用

- **LPN [[2411.08706]]**（編碼器→連續潛在 z→梯度上升搜尋→逐像素解碼器）與 **NLI [[2604.18907]]**（離散 token 程式＋可微解釋器＋潛在空間梯度搜尋）是程式合成的另一大支，技術上漂亮（LPN 把測試時搜尋內建進可微架構、OOD 翻倍；NLI 組合泛化 OOD 達 91–100%）。

- **但與 ai_core 前提硬衝**，故列為**負面火花**而非採用：
  1. **需要訓練神經網路 + 梯度**——違反「純標準庫、零相依、純 CPU、省算」四條硬約束（`CLAUDE.md` 設計哲學）。
  2. **潛在 z 不可解釋、不可驗證**——與 §3.4「可稽核、憑證准入、可撤照」治理原則正面矛盾；ai_core 要的恰是**離散、可讀、可執行驗證**的資產（這也是 deep 綜述 §3 的結論：程式合成珍貴正因「程式可執行＝完美 verifier」）。
  3. 兩篇自己都承認**組合泛化不穩健 / 潛在搜尋算力貴**。

- **唯一可借用的一點**：deep 綜述 §3.3 與 LPN 未來工作都指向「**連續粗搜 → 投影到離散程式做精修與驗證**」的混合。若 ai_core 哪天要做「資產的模糊檢索」（給定一個洞，找最相近的既有 asset），可借**句嵌入做粗排（連續）→ AST/執行做精篩（離散，火花5/6）** 的兩段式——但這屬遠期，且嵌入只能放在貴智能資產工廠側，不入消費端。

## 收束：兩條可立即開工的線

1. **固化引擎（§3.6「這題優先」）= 火花3 ⊕ 火花7**：wake-sleep 給「何時固化＋庫怎麼維護（<100＋LRU）」，ILP-CEGIS 給「固化具體長出什麼 matcher＋必要性閘」。資料源（ATP trace NDJSON）已存在，兩者皆屬離線貴智能工廠、不污染笨模型消費端——是把「飛輪」從比喻變成管線的最短路徑。
2. **v0 驗收翻盤技術（§6）= 火花2 ⊕ 火花6**：多候選 maximin 裁決（省貴智能）＋執行引導填洞（用現成 `isolation.py`），最有機會做出「raw 笨模型失敗、包鷹架成功」的對照組翻盤，且兩者都直接落在 ATP 已凍結的檔案上（`sfc.py / line_assistant.py / isolation.py`）。

> 全部火花刻意守住一條線：**貴智能只在「開機期、離線、低頻」的資產工廠側出現（生 schema、抽象、ILP 提名、偽標籤裁決），產物一律是純標準庫、離散、可驗證的資產**；笨智能消費端維持零相依。這正是 `roadmap.md §2`「貴智能產抽象 → 便宜智能高頻復用」的具體化。
