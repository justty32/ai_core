- **一句話大白話**：回顧我們是怎麼從「分層架構」進化到現在的「本質軸」設計。

---

## 一、我們的定位：幫 AI 蓋「自我進化」的工廠

早期我們把 ai_core 定位為 **Heuristic Learning (啟發式學習)** 的基礎設施。
聽起來很玄，說穿了就是：**讓 AI 自己寫程式，跑完測試後自己改進，不斷疊代。**

- **Entry Manager**：AI 監督者的通訊介面。
- **Author**：AI 用來產生或修改程式碼的工具。
- **Hub**：存了一堆策略（Functions）的圖書館。
- **Metadata**：給 AI 讀的說明書，讓它知道怎麼用這些工具。

---

## 二、Metadata 的真正任務：別讓 AI 燒腦

我們寫 Metadata 不是為了好看，是為了**把「認知負擔」從 AI 身上搬走。**
AI 的腦袋（Context/Memory）有限，如果 Metadata 寫得爛，AI 就得在那邊猜：
- 「這個 Function 執行失敗，我該重試嗎？(Retry safe?)」
- 「這東西跑完產生的資料，哪些是重點、需要記住？(Memory hints?)」
- 「這東西用起來會不會很複雜？(Complexity?)」

**Metadata 的職責就是：把這些問題直接回答掉，讓 AI 不用猜。**

---

## 三、四大執行模型：你是哪種程式？

我們不用「技術層級」分，我們用「怎麼執行」分：

1. **Pipe-and-Script (快餐)**：簡單的 Shell 腳本，用完即丟。
2. **One-shot (正餐)**：比較複雜的單次任務，需要管記憶體跟執行時間。
3. **Persistent (長跑)**：長期活著，有狀態，可以互動（像 LSP 或 Git）。
4. **Server (管家)**：因為資源（如 LLM Token）有限，需要有人排隊管理併發請求。

---

## 四、CLI 與 OOP 的大一統

不管是 Python 的 Class 還是 Shell 腳本，我們認為它們本質上是一樣的。

- **Metadata**：Python 裡是物件屬性，CLI 裡就是 `--metadata` 吐出來的 JSON。
- **Index**：Python 裡是 Module 路徑，CLI 裡就是檔案路徑。
- **Hub**：就是把這兩類東西都「聚」在一起的目錄。

**寫一份 Python Class，就能自動生出 CLI 介面。** 這是開發者最大的福音。

---

## 五、狀態管理：這四個資料夾別放錯

如果一個程式需要「多次執行且記得上次做了什麼」(Multi-shot)，請按這四個語意存檔案：

- **`.config/`**：人類寫的設定，程式只讀不改。
- **`.cache/`**：隨便你刪，程式沒它也能跑。
- **`.state/`**：進度條。清掉就代表「重新開始」。
- **`.data/`**：這才是真正的成果。清掉會出事，要記得備份。
