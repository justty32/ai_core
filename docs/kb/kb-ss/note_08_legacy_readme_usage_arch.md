# note_08｜舊版操作指南：以前我們怎麼玩 ai_core？

- **來源**：舊版 `README.md`、`usage.md`、`architecture.md`。
- **狀態**：**考古操作手冊**。這裡的指令和格式都已經過時了，僅供參考。
- **一句話大白話**：以前我們靠「Shell 腳本 + `--metadata`」通殺一切，讓 AI 幫我們瘋狂寫工具。

---

## 一、核心理念：OS 就是最強框架

那時候我們只有三招：
1. **Process (程式)**：只要是一個執行檔，且支援 `--metadata` 參數，它就是一個工具。
2. **Hub (索引工具)**：跑一下 `hub --build-list`，它就會掃描目錄，把所有工具的說明書寫進 `list.json`。
3. **Session (會話)**：用一個簡單的 JSON 檔存對話紀錄，像回合制遊戲一樣，跑完這動存個檔，下一動再讀出來。

---

## 二、以前的工具長什麼樣子？ (Skeleton)

那時候每個 `.py` 腳本都長這樣：
- **`--metadata` 模式**：印出自己的名稱、功能描述、輸入輸出格式。
- **執行模式**：看 `argv` 有沒有參數，沒有就讀 `stdin`，處理完直接 `print` 到 `stdout`。

**AI 寫工具的 6 個步驟**：
1. 搜一下 Hub，看有沒有重複的功能。
2. 叫 AI 寫個 `.py` 腳本。
3. `chmod +x` 給它執行權限。
4. `hub --validate` 跑一下測試案例 (Examples)。
5. 自己手工測一下 (Smoke test)。
6. `hub --build-list` 更新清單。

---

## 三、那時候的 Hub 能幹嘛？

Hub 是一個 One-shot 工具（跑完就結束，不是 Server）：
- **建索引**：把幾十個腳本彙整成一份 `list.json`。
- **搜工具**：關鍵字搜 `name` 或 `description`。
- **生文件**：自動生出給 AI 讀的 `AGENTS.md`。
- **轉格式**：把工具清單轉成 OpenAI 或 Anthropic 的 Tool Schema 格式。

---

## 四、架構邊界：誰管什麼？

- **工具 (Process)**：只負責把自己的事做好。不准知道別人的存在，也不准管別人的 Session。
- **Hub**：只負責管清單。不准執行工具，也不准管 Session。
- **Session Lib**：只負責讀寫 JSON。不准管 UI，也不准管排程。

---

## 五、為什麼現在不用這套了？

雖然這套「極簡主義」很爽，但有幾個硬傷：
1. **依賴太重**：以前核心居然依賴 `litellm` 等一大坨 Package，現在我們追求**零相依**。
2. **管理太鬆**：以前覺得 AI 生工具越快越好，現在我們覺得「沒經過認證的 AI 工具」是風險。
3. **缺乏全局觀**：以前每個工具都是黑盒，互不相干；現在我們需要「LLM Entry Manager」來統一管理 Token 消耗和頻率限制。

---

## 結語

看這份手冊，你會發現 ai_core 的核心一直都是 **「工具的自動化與標準化」**。雖然指令變了、規範嚴格了，但「讓 AI 容易使用與擴展系統」的靈魂還是在的。
