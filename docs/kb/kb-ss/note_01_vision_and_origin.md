# note_01｜願景與初心：為什麼要做 ai_core？

- **來源**：`roadmap.md` (2026-05-24 初心檢討討論)。
- **狀態**：**戰略北極星（最新事實）**。決定「該做什麼、先做什麼」的最高準則。
- **一句話大白話**：預測未來聰明模型會變貴，所以我們要趁現在聰明模型還便宜時，把邏輯「固化」成程式碼；讓以後的「小笨模型」只需填空就能跑。最終目標是讓 LLM 只負責翻譯意圖，其他能寫死的都寫死。

---

## 0. 懶人包 (TL;DR)

- **核心假設**：AI 泡沫會破，聰明又穩的模型會變貴。以後我們手上可能只剩本地跑的小模型 (<10B)。
- **解決方案**：做一個框架來「輔助」這些小笨模型。CLI 是最適合 LLM 的介面，所以形態就是「一堆小工具 + 邏輯組合」。
- **現在該幹嘛**：趁大模型現在還便宜，趕快用它來產出高品質的「資產」（程式碼、few-shots、骨架）。現在投資，以後讓笨模型消費。
- **最終形態**：LLM 應該被擠到只剩兩個地方：
    1. **翻譯層**：把人話轉成程式懂的格式。
    2. **模糊地帶**：還沒被寫成固定程式碼的暫時處理器。
- **治理原則**：**預設拒絕**。別隨便把 LLM 塞進流程。除非證明「非它不可」且「測試超穩」，才發給它一張「准入證書」（標明型號、穩定度）。

---

## 1. 它是怎麼長出來的？

這專案不是拍腦袋想出來的，它有一條工程邏輯：
1. **省錢前提**：假設以後只有小笨模型可用。
2. **最小工具**：做個框架幫小模型（Prompt 片段管理 + 輸出後處理）。
3. **遇到資源瓶頸**：模型推論慢、一次只能跑一個 -> 引入 **Singleton 資源管理** (LLM Entry Manager)。
4. **發現本質**：這玩意兒其實像「符號神經網路」——大部份是寫死的 Function，少部份節點才是隨機的 LLM。
5. **自我改進**：既然 LLM 能寫 Prompt，那就能自我進化 -> **數據飛輪**。

> **設計哲學**：KISS、輕量、不造輪子、最小依賴。這不是文青情懷，是因為我們「窮」，養不起龐大的基礎設施。

---

## 2. 策略：資產工廠 vs 資產消費者

我們現在正處於一個「智能紅利期」。

| | 現在：聰明模型 (貴/稀有) | 未來：笨模型 (便宜/高頻) |
|---|---|---|
| **角色** | **資產工廠** | **資產消費者** |
| **工作** | 讀原始碼、生 API 定義、生 Few-shots、搭骨架。 | 照著生好的資產，在窄小範圍內填程式碼。 |

**玩法**：現在花一點錢請大模型把「耐操」的東西做出來，以後換成笨模型也能照著跑。

---

## 3. 核心設計：LLM 應該退縮到什麼程度？

### 3.1 兩大介面
在成熟的系統裡，LLM 只該負責兩件事：
1. **意圖翻譯**：模糊人話 -> 結構化 JSON。
2. **處理例外**：還沒寫成 `if/else` 的模糊 case。

這兩個洞永遠補不滿，但其他的洞都要想辦法填成程式碼。

### 3.2 別再叫它 "Skill" 了，它叫「帶洞的程序」
別把 LLM 想成全能的 Skill。它其實就是：
> **一段寫死的程式碼，中間留幾個洞給 LLM 填。**

最好的類比：**LLM 是 `cache-miss handler`**。
- 請求進來，先過一堆 `regex` 或 `matcher`。
- 有命中的走固定邏輯（快、省、穩）。
- 沒命中的（落穿），才丟給 LLM 處理。

### 3.3 洞的大小取決於模型智商
- **笨模型**：過濾器寫多一點，洞留小一點（不敢信任它）。
- **聰明模型**：過濾器可以少一點，洞留大一點。
- **動態擴縮**：先叫笨模型試試看，沒過測試再升級給聰明模型。

### 3.4 治理：預設拒絕 + 憑證准入
**誰想用 LLM，誰就要舉證。** 每個 LLM 節點都要有「證書」：
- 「這東西測試過，用 Z 模型，穩定度 98%，目前真的沒辦法寫死，核准上場。」
- 哪天有人寫出了固定算法，這張證書就撤銷，LLM 滾蛋。

---

## 4. 解藥：目標問題 = 停止鍵

框架最怕無限擴張。解決辦法就是**釘死一個目標問題**。
只有目標問題需要的功能，才做進框架。

- **A 層 (最小工具)**：SFC、簡單組合、資源管理。**這是真的要用的產品。**
- **B 層 (北極星)**：規範、治理、飛輪。**這是指北針**，告訴 A 層該往哪走。

**開發順序翻轉**：
不要先定標準再找問題。要**先找問題，讓問題逼你定標準**。

---

## 5. 第一個目標：程式碼輔助助手 (Code Assistant)

傳統 Agent (如 Claude Code) 很強但很貴，每次都要讀整坨 code。
我們的拆法：

1. **上層 (聰明模型)**：跑一次，把框架 API、用法、few-shots 抽成結構化資產。
2. **下層 (笨模型)**：拿著資產，只負責「在第 N 行填入這個片段」。
    - 笨模型最怕「重寫整個檔案」（自由度太高必出事）。
    - 把它限制在「受約束生成」：配合**行數助手**，只准它改那幾行。

---

## 6. v0 最小切片 (做給你看)

拿一個真實框架，跑一次流程：
> **大模型抽資產** -> **小模型拿資產 + 行數助手** -> **產出正確編輯**。

**驗收標準**：裸奔的笨模型會寫錯，但套了 ai_core 鷹架的笨模型能寫對。打通這條線，專案就活了。

---

## 7. 待解的硬骨頭：固化引擎
飛輪的核心是「自動化」：LLM 處理多了，能不能自動幫我寫出 `if/else` 來取代它？
- **手動**：人看 log 改 code。
- **自動**：聰明模型掃 log 生 code 提案。

目前這題還沒定論，是 v0 要撞牆的地方。
