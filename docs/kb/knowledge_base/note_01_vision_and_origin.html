note_01｜願景與初心（Vision & Origin）

  • 來源：roadmap.md（建立於 2026-05-24，源自一次「檢討初心」的討論；同日第二輪討論把重心從「隔開貴/便宜智能」搬到「LLM 退縮到兩介面 + 拒絕為預設的憑證准入」，已整合）。
  • 狀態：戰略主文件 / 北極星（最新事實）。這是判斷「那些東西該不該做、該先做哪個」的尺；不是規範（規範在 core_nature/，見 doc_01~doc_08），也不是實作（探索在 try_implement/）。活文件，逐項可改，標「待議」者皆未定案。
  • 一行摘要：未來只剩便宜小笨模型可用 → 做框架輔助它；趁現在聰明模型還便宜當「資產工廠」轉飛輪；終點是把 LLM 擠到「意圖翻譯」與「模糊前沿」兩介面，以「拒絕為預設、憑證准入」治理，飛輪持續把模糊地帶固化成確定性程式碼；第一目標問題＝程式碼輔助助手。

交叉引用：待決規範表（§7）與開放問題（§8）的詳載見 note_06_decisions_and_open_questions.md。九軸/register/intercept 等核心契約見 code_01_core_library.md 與 doc_05_axes_metadata.md。

0. 一頁摘要

  • 初心：未來聰明可靠的 LLM 會因 AI 泡沫化而變貴；假設手上只剩便宜的本地 <10B 小模型 → 必須做一套框架去輔助小笨模型，把它包裝成可靠的東西。CLI 天生適合 LLM，所以基本形態是「一堆小命令行函式 + 它們的組合」。
  • 時間維度：趁現在聰明大模型還便宜，先用它生產出耐久的資產（程序 / snippet / few-shot / 骨架），讓未來的笨模型去消費。 飛輪的第一圈要趁早轉。
  • 終點形狀：在成熟系統裡，LLM 被擠到只剩兩個介面——(1) 人類意圖 → 程式可理解意圖的翻譯層；(2) 尚未被固化的模糊地帶的暫時處理者。其餘全是確定性程式碼。
  • 治理原則（拒絕為預設）：LLM 預設什麼都不碰；只有**「千錘百鍊證明非它不可」＋「測得夠穩」，才核准它進入某個環節，並發給一張證書**（用哪個模型、測試組、穩定度 %）。舉證責任在「讓 LLM 進來」那邊，不在「把它拿掉」那邊。
  • 由此得到的專案定義（升級版）：

    ai_core = 一個讓 LLM 退縮到「意圖翻譯」與「模糊前沿」兩介面的基底；它以「拒絕為預設、憑證准入」治理 LLM 的領地，並用飛輪持續把模糊地帶固化成確定性程式碼。

  • 第一個目標問題：程式碼輔助助手——上層「讀框架原始碼 → 生資產」（＝結構化抽取），下層「在精準標記的位置填碼」（＝受約束生成）。
  • 方法論剎車：先選目標問題，再讓它告訴你哪些規範該先定。目標問題 = 停止鍵，是「東西愈做愈大」的解藥。

1. 初心與演化鏈

這個專案不是憑空長大的，它有一條有內在邏輯的演化鏈（這段以前只在腦子裡，今天落檔）：

  1. 前提（約束）：AI 泡沫化 → 聰明可靠的 LLM 太貴 → 假設只有便宜的本地 <10B 小笨模型可用。
  2. 初心（最小形態）：既然只能用小笨模型，就得做框架去輔助它；CLI 天生適合 LLM → 最初只是幾支小命令行程式，做 prompt 片段管理 + 組合 + 輸出後處理（＝最原始的 SFC）。
  3. 第一次擴張：推論要時間、一次只能回一串 → singleton 資源 + 時間/記憶體管理（＝ LLM Entry Manager 的由來）。
  4. 質變的領悟：擴著擴著發現這形狀像符號神經網路——一張計算圖，多數節點是確定性的 shell/py 函式，少數節點是那個隨機的 LLM。
  5. 野心的引爆點：LLM 能自己生 prompt → 自我改進迴圈 → 數據飛輪。為此東西從「幾支小程式」膨脹成「整套規範 + 框架 + 概念」。

設計哲學（KISS / Lightweight / No wheel-remake / Least dependency）不是美學偏好，是前提的邏輯必然：一個前提是「窮、只能用便宜算力」的專案，更付不起一座龐大基礎設施的維護成本。

2. 時間維度：資產工廠 vs 資產消費者

初心只講了「未來」的約束。今天補上的洞察是現在這個窗口的價值：

聰明模型（現在便宜） 笨模型（未來便宜）
角色 資產工廠 資產消費者
跑法 貴、稀有、一次性 便宜、高頻
做什麼 讀框架原始碼、生程序/skill、生 snippet 庫、生 few-shot、搭骨架、做固化（見 §3） 照著資產，在窄任務上填碼 / 套用

策略：把現在的「便宜智能窗口」當成一筆一次性的資本投資，產出折舊很慢的資產。聰明模型今天做好的東西，明天換成笨模型還能用。

此分工有兩個尺度：

  • per-phase（粗）：工廠（一次性生資產）vs 消費者（高頻用資產）。
  • per-request（細）：同一個請求內，先笨模型填，沒過驗證再升級給聰明模型——一條成本梯度（見 §3 的「洞」與升級）。

3. 終點形狀與治理原則（本專案的核心）

3.1 終點：LLM 退縮到兩介面

在成熟系統裡，系統大部分是確定性程式碼，LLM 只剩兩個介面：

  1. 意圖翻譯層：人類的模糊意圖 → 程式可理解的結構化意圖。
  2. 模糊前沿的暫時處理者：還沒被固化成程式碼的模糊地帶，暫由 LLM 扛著。

這兩個洞是永遠不會完全閉合的（總有新意圖、總有新模糊案例）；其餘的洞都該隨時間被填成程式碼。

3.2 單位不是「skill」，是「帶 LLM 留白的程序」

我們一度叫它「skill」，但它太重了，這個詞不準。它其實是：

一段大部分為確定性程式碼的「程序 / 套」，中間留著少數 LLM 才填的洞。

最精準的類比——LLM 是 cache-miss handler，或一個巨大 switch 的 default: 分支：

  • 請求先過一個確定性前濾網（format matcher，如「改 XX 檔 XX 行：…」）。命中 → 走純程式碼。
  • 只有落穿前濾網的、沒被預先定義的那塊（範例裡的 …），才掉進 LLM。
  • 它也不限於一個函數，而可以是一整條調用鏈；且會隨所用模型型號而變化（見下）。

3.3 洞的大小 = f(模型能力)

關鍵性質：同一段程序，笨模型 → 塞更多 matcher、洞更小（不敢信它）；聰明模型 → matcher 更少、洞更大（可放手）。同一套程序、不同的「填充比例」。

  • 「先笨模型想、沒通過邏輯測試再叫聰明模型」＝ per-request 動態縮放這個洞。
  • 橫切的 使用者習慣層（如「我是繁中使用者 → 產出都用繁中」）疊在整條鏈上，是 llm_call context binding 的 user-profile 級升級。

3.4 治理原則：拒絕為預設 + 憑證准入

舉證責任在「讓 LLM 進來」那邊。 LLM 預設零領地；要佔據某個環節，得通過兩部分閘門：

  1. 必要性：千錘百鍊證明確定性程式碼辦不到、非 LLM 不可。
  2. 穩定性：在測試組下做得夠穩，過可靠度門檻。

兩個類比：

  • 最小權限（least privilege）：LLM 預設無權，每個環節單獨申請、單獨核准。
  • 新藥審批：一個 LLM 環節 = 一款藥，臨床試驗證明療效（非它不可）＋安全性（夠穩），才核准用於該適應症。

後果——每個 LLM 留白帶一張證書（這把飄很久的 nondeterministic 軸做實了）：

不是含糊的「這裡是隨機的」，而是「此環節經測試組 B、用模型 Z、認證穩定度 X%」。LLM 不是被信任，是被認證；可被撤照（哪天有人想出確定性解法就踢出去）。整個系統因此可稽核：永遠知道哪幾個點是「沒辦法才放 LLM 進來」、各自多穩、用什麼擋著。

3.5 兩個 regime，飛輪是它們之間的遷移

「拒絕為預設」的嚴格和「趁便宜先轉幾圈」的寬鬆，表面衝突，其實是兩個 regime：

regime 時機 姿態
開機期 現在，聰明模型便宜 寬鬆——讓 LLM 多做、先把骨架搭起來、先轉幾圈
成熟期 未來，只剩笨模型 嚴格——拒絕為預設、憑證准入、footprint 極小

飛輪 = 從寬鬆遷往嚴格的力：持續地 (a) 把 LLM 環節換成確定性程式碼（撤照）、(b) 給活下來的 LLM 環節發證書（認證）。開機期容許鬆，正因為飛輪保證它會收緊。

3.6 最硬的未決問題：固化引擎

飛輪的「免費能量」來自固化（crystallization）——把「LLM 老在處理的同一類模糊案例」凍成確定性 matcher / 分支。但固化不是免費的：誰做？

  • 手動：人觀察、人寫新分支 → 這專案是「一個好用的工具」。
  • 自動：聰明模型定期掃 log、自動提案新的確定性分支 → 這專案是「一個會自我改進的系統」。後者難非常多。

此問題決定 v0 切片要不要把「固化」也做進去（見 §8，詳載於 note_06_decisions_and_open_questions.md）。目前未決。

4. 設計張力與它的解藥

張力：設計哲學要 KISS（前提的必然），但北極星（飛輪 / 自我改進）天然往「大」拉。兩股力反向。

解藥 = 目標問題：框架沒有目標問題時會無限長——每條軸、每個複合規範看起來都「可能有用」。一旦釘死一個具體目標，它變成剎車：只有這個目標用得到的部分才做。目標問題 = 停止鍵。

A / B 兩層分法（避免把兩個東西當一個做，那正是「感覺在爆炸」的根因）：

  • A：能用的最小工具（SFC + 幾個組合原語 + singleton 管理）。今天就能交付、忠於初心、天然小。這是產品。
  • B：北極星願景（規範家族 + 治理原則 + 飛輪）。這是羅盤，用來指引 A 怎麼長，不是 v1。

方法論翻轉（重要）：

不要「先把八軸 / 複合 / 多函數合作的標準全定完，再選問題」。 要「先選目標問題，讓它告訴你哪些標準該先定、哪些可以晚點」。 理由：try_implement/DECISIONS.md 裡那堆懸案的優先序取決於目標是什麼。沒有目標只能憑空猜；有了目標，它們自動排好序。

5. 第一目標問題：程式碼輔助助手

場景：手上有一個程式框架。理想做法是用聰明大模型 + AI agent（如 Claude Code）讀框架原始碼 → 生 skill → 輔助編寫相關程式碼。效果好，但很花錢（每次都燒聰明模型）。

ai_core 的版本：把它拆成兩層，聰明模型只跑一次性的上層，笨模型跑高頻的下層。

上層：讀框架 → 生資產（＝結構化抽取）

  • 聰明模型讀原始碼，抽出有 schema 的資產：API 簽名 / 用法 snippet / 觸發條件 / few-shot / 護欄 / verifier hook。
  • 本質是結構化抽取問題：schema 可驗證，retry_until_valid / guard 咬得住。
  • 「結構化抽取也不錯」這個念頭就安放在這——它不是另一個目標，是同一個目標的上層。

下層：在精準標記的位置填碼（＝受約束生成，§3.2 的「洞」）

  • 笨模型最不該做的是「重寫整個檔」（自由度爆炸 → 必錯）。
  • 把輸出收縮到窄面：固定 snippet 寫進 prompt + 行數助手精準標記「在第 N 行插入 / 填這個洞」。自由度愈小，笨模型愈不會搞砸。
  • 行數助手 = 那個確定性函式；模型只負責粗粒度決定「填什麼、填哪」。
  • 佐證：Claude Code 自己的 Edit 工具就要求 exact string match、不準重寫整檔——聰明模型都需要這層約束，笨模型只會更需要。

正名：我們生的不是「skill」（一個文字檔），而是 §3.2 的**「帶 LLM 留白的程序」**——可能是一整條調用鏈、隨模型型號變化、且每個 LLM 留白都該帶 §3.4 的證書。

6. 最小垂直切片（v0）

先別鋪整套規範，先切一條最小的線證明飛輪能轉：

拿一個真實框架 → 用一次聰明模型呼叫，生出一小份資產（程序 + snippet 庫）→ 再讓一個便宜 / 笨模型 + 行數助手 + retry/guard，產出**「raw 笨模型做不到、包了鷹架就做得到」**的正確程式碼編輯。

驗收標準：同一個編碼任務，對照組（raw 笨模型直接寫）失敗 / 出錯，實驗組（ai_core 鷹架）成功。一條線打通，勝過十條規範。

（具體選哪個框架、哪個任務、用什麼笨模型、要不要含「固化」步，待議——見 §8 / note_06_decisions_and_open_questions.md。）

v0 切片會自然撞到 try_implement/DECISIONS.md 裡的幾個懸案（語意/用途欄位 B1、in-process 沙箱 B3、memoize 快取、nondeterministic 軸含證書、組合軸推導 A4），到時它們從「憑空猜」變成「擋路、得解」。v0 切片本身就是規範的篩子。

是這樣的，我想要重新檢討自己做這些事情的初心。我記得上一次討論這個專案的目標的時候，有說是想要做一個標準規範以及標準實現，用於駕馭LLM的不確定性。我想這依然是沒有改變的，只是說我突然回想起好幾個月前的，與這個專案有關的，更原始的構想。那時候，我是想說，未來，聰明且可靠的LLM會愈來愈貴，那是因爲AI泡沫化的原因，所以，我假設未來我只有一些可以在本地跑的，便宜的10B以下的模型可以用，這樣的話，那我就必須去做一個框架，去很好的輔助這種小小笨模型，並且那時候我有想到說，命令行天生就適合LLM的相關使用，所以那時候的想法其實就只是簡單的SFC，一種管理prompt片段與其組合方式，還有管理LLM輸出並進行後處理的，幾個小命令行程式。後來才想到說，考慮到模型推論需要時間，且一次只能回應一串，也就是singleton資源的概念，然後還有時間和記憶體的管理這些。之後慢慢擴張，直到最後我發現到，這其實有點類似符號神經網路的概念，並且某天我突然想到說，因爲LLM可以自行生成promp，所以可以形成自我改進的循環，變成數據飛輪，于是爲了達到這個目標，我想要做的東西愈來愈龐大，從最初的幾個小程式，變成了一整套標準規範乃至於框架,概念等

③ 緊迫性 + 第一目標＝程式碼助手

並且我突然想到一點，那就是，其實我會想要儘快把這些東西做出一點成果，至少在當下，比較聰明的大模型還算便宜的時候，先讓飛輪轉幾圈，打個骨架，這樣可以省下今後不少功夫。然後第一個目標問題，我想先做一個程式碼輔助助手，好比說我有個一個程式框架，以前用其他很聰明的大模型+ai agent，如claude code，請他去讀框架原始碼，然後去生skill，之後再去輔助編寫相關程式碼，那當然很好，但很花錢。目前的想法是這樣，還是先讓聰明的大模型去生skill，但是在編寫相關程式碼部分，我們可以做細化，比如把一些固定snippet寫入prompt，配合行數助手精準標記要寫入的目標...你覺得呢？等等，其實結構化抽取也不錯

④ 搜尋報告例子 + skill 超重（是函數/調用鏈）+ LLM 退縮到兩介面

然後我還有個想法，好比說我想要讓模型去搜尋某個topic，然後做報告，然後我對那個topic稍有了解，知道它應該要先讀取某些文件，然後再去某些網站用APi取得資料，所以我會給一個prompt，模型就會依據這套prompt去採用一套。然後這一套還可以繼續改進，比如說模型可能不知道API有哪些，那麼模型本身會跑另一個已經固定的流程，去分析網站的形式，若是A形式，那可以透過某些方式取得API列表之類，除此之外，還有使用者習慣這個功能，在做這一整套流程時，模型會知道我是繁體中文使用者，所以他產出的文件都會是繁體中文。然後你說的ai_core的skill，那確實會超級重，那不會只是一個文字檔，而是一個函數，並且會受到所使用的模型型號所影響，有所變化，乃至於不限於一個函數，而是一整個調用鍊條。然後我預期中的使用方式會是這樣，好比說我給了一個promp，這個prompt會先經過一個固定函數，判斷說是否符合某些特定format比如說"改XX文件的XX行：..."，然後去進行相關處理，而只有在...部分這塊沒有被預先定義的格式，才會拿模型去進行分析，並且有可能會先讓笨模型思考，然後若笨模型思考沒通過邏輯測試，那會讓聰明模型重新思考一次。在我的構想中，在成熟的系統中，LLM只會成爲人類意圖與程式可理解的意圖的轉換接口，還有一些尚未被定義的糢糊地帶的暫時處理者。所以，嚴格來說，其實這不太能說算是skill。

⑤ 治理原則的修正：拒絕為預設、憑證准入

或者說，在經過千錘百鍊的測試後，確定了某個環節只有LLM才能處理，並且處理的足夠穩定，才會讓LLM成爲該環節的一部分
