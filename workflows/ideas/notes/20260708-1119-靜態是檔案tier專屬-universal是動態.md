# 靜態是「檔案 tier」專屬，universal 是動態——「一切皆檔案」的降級修正

> 記於 2026-07-08，接續同日上午 [統一描述子/複合路徑那則](20260708-1040-一切皆檔案-list-統一描述子-複合路徑.md) 的下半場。
> 一場踩了「停止鍵」的修正：先砍掉「靜態屬性三層化」這個假問題，再把「一切皆檔案」降級成「一切皆動態存取點」。
>
> **狀態標記**：〔決定〕＝使用者當場拍板；〔提案〕＝我方提出未明確追認；〔懸〕＝明確留待之後。

## 〇、觸發：size 不重要，逼出對靜態的重新定位

使用者踩剎車〔決定〕：**檔案大小不重要**。size 本質是「資源占用的 proxy」；而資源占用已有軸在管（cpu/gpu… 等**執行時**的），**靜態占用**現在沒有消費者需要 → 擱置。判準回到 roadmap 的**方法論剎車**：目標問題（更好、更能利用 LLM 的軟體工程範式＝程式碼輔助助手）＝停止鍵。一個屬性要不要進 metadata，看「那個 LLM 消費者用不用得上」，不是「檔案系統有沒有」。

這一刀順帶**消掉了前一則在建的「靜態屬性三層化」表**（size 在 fs/遠端/碼各長什麼樣）——結論是**別排**：靜態只活在檔案這一 tier，其他 tier 的靜態各自內部消化，不進我們的描述子。

## 一、修正後的骨架〔決定〕

| tier | 存取性質 | 靜態怎麼辦 |
|---|---|---|
| **檔案 tier（中間層）** | 「一切皆檔案」在此成立 | **靜態的唯一家**：perm/owner/envelope 留著 |
| **檔案內部（sub-file 數據塊）** | 動態：一個 server 供特定數據塊存取（資料庫老路） | 由該 server/DB 內部處理 |
| **程式內部（running program 狀態/功能）** | 動態：全暴露成 CLI 呼叫（或 DB 形式、或外部 .json） | 裡面想幹嘛幹嘛，靈活；但暴露面全動態 |
| **程式碼內部（源碼）** | 不考慮，就文字檔 | — |
| **分散式** | 動態：所有檔案存取、函數呼叫全走動態 | 靜態由遠端自己的 RESTful API ＋ DB 代替 |

**一句原則**：
> **靜態只是「檔案 tier」的性質；比它細（檔內、程式內）或比它寬（遠端）全走動態，各自用 DB/REST/json 自備靜態。檔案，是唯一一層由我們的描述子替它記靜態的粒度。**

**與既有軸的接點**〔提案〕：「檔案內部走資料庫」＝ `persistent`／「重量級函式變 server」軸的現身；`lifecycle`（one_shot vs persistent）已能宣告「這是 `cat` 得到的靜態檔，還是 server 前臺的 dynamic queryable」。envelope 靜態描述整體、interior 由 `schema` 描述其被動態查詢的結構 → 對上前一則統一骨架的 static/dynamic 分段。

## 二、口號降級〔決定〕：一切皆檔案 → 一切皆動態存取點

> **不再是「一切皆檔案」，而是「一切皆動態存取點（call/query）；檔案 tier 額外多帶靜態屬性」。**

靜態從「宇宙基底」降成「檔案層的加值」；**動態才是那個 universal**。這是真正的概念位移，取代前一則 §三/§四 的三層 static/dynamic 對應（那兩節就此作廢重寫）。

理由的漂亮處：這樣**遠端與 sub-file 被統一處理**（都動態），靜態變成「本地檔案塊」這個特例能廉價提供的 bonus，而不是硬要在每一層都湊出來的東西。上一則辛苦排的 HEAD/manifest/schema-bound 三層靜態故事，在這版**不需要**——那是把 file tier 的乾淨靜態，硬投影到不該有它的地方。

## 三、關鍵切分：perm/owner ≠ size，命運不同〔決定＋提案〕

使用者同一段裡「size 不重要」卻說 perm/owner「還是很好、要保留、要多思考」。兩者**理由不同**，必須分開：

- **size ＝資源 proxy** → 擱置（消費者現在用不上）。
- **perm/owner ＝治理／存取控制** → **保留，而且是核心**。它不是隨便一個 stat 欄位，是**「拒絕為預設 ＋ 憑證准入」那條治理主線**在檔案層的現身：誰、憑什麼權限、能不能呼叫這個。與上一則「歸化契約不能騙」的安全半邊同源。

> **resource-static 擱置，governance-static 核心。** perm/owner「要多思考」正因為它是**安全／准入**，是這套 LLM 範式的骨幹，不是資源記帳。這條線一劃，「靜態要保留哪些」就不再模糊。

〔懸〕perm/owner 的具體設計（照 Linux rwx/owner/group 制度為模板到什麼程度、如何對上憑證准入）留待之後。

## 四、扣回北極星：消費者收成兩個動詞

這版的真正好處（不只是整齊）：把 LLM 消費者的世界收成**兩個動詞**——
1. **讀靜態 envelope（`--metadata`）來規劃**；
2. **發動態呼叫（CLI/API/query）來執行**。

就是 hub → skill 清單 → 呼叫 那個迴圈。**smart model 讀 envelope 生資產、dumb model 動態呼叫填碼，兩個動詞打天下。** 所以這修正**直接讓目標問題（程式碼輔助助手）的消費者更好用**，不是建築天文學。

## 五、對前一則 note 的效力

- **作廢重寫**：前一則 §三（三層量尺）、§四（統一描述子的三層靜態）中「每層都要湊靜態」的部分 → 改為本則「靜態＝檔案 tier 專屬、universal＝動態」。
- **仍成立**：統一描述子的 static/dynamic **分段**本身、input_schema 缺口、guarantee:none 拆分、caller 折疊、compose() 原語——這些不受影響（它們是關於動態/複合的，本次修正只動靜態的定位）。
- **新增待辦**：perm/owner ＝治理軸的落地設計〔懸〕。
