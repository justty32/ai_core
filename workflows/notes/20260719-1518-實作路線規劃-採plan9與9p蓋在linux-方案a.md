## 二、新增事實：Lua/Fennel 亦可 handle「fd＋四通道＋庫」這塊〔決定〕

使用者補：前兩篇把 Lisp 講成理想載體（執行態天然是 list 環境、路徑天生對得上）；
**其實 Lua 也可以 handle 這塊**——做那個「fd＋argc/argv/shared memory/pipeline」的暴露與交互庫。

〔提案・待追認〕這對實作路線意義重大，因為它把抽象地基接上**既有試驗田**：

- 〔錨定〕`sub_projs/handy/` 目前就是 **Fennel/LuaJIT**（`llme` dispatcher＋`zhtw` 薄包裝）——見 memory「handy 子專案」。
- **Fennel ＝編譯到 Lua 的 Lisp**：它同時滿足「Lisp 語意（list 環境、s-expr）」與「Lua 執行載體」兩面。
  → 所以「理想載體是 Lisp」與「Lua 也能 handle」**不衝突**：Fennel 正好是兩者的交集，可能是路線一的**最省力起點**。

〔疑〕但要分清：使用者說的「Lua 能 handle」是指
（a）純 Lua 也能寫出那個暴露庫（不必是 Lisp），還是
（b）就用 Fennel（Lisp-on-Lua）當第一實作？兩者對「規範層語言中立」的姿態不同。

## 三、關鍵定調：採 Plan 9 思想＋9P 協議，蓋在 Linux 上（方案 A）〔決定〕

Claude 指出：**「階段一：歸一於路徑」基本上就是 Plan 9 已做完的事**——貝爾實驗室做 Unix 那批人做的後繼 OS，把「一切皆檔案」貫徹到 Unix 沒膽貫徹之處。前幾篇推導出的東西幾乎逐項命中它：

|---|---|
| 歸一於路徑、通用協議＝檔案（世界篇 §十一） | **9P 協議**：單一極簡檔案協議（walk/open/read/write/clunk…），任何資源都講它 |

### 3.1 定案：方案 A——採思想＋9P，實作在 Linux〔決定〕

使用者拍板 **方案 A**：**採 Plan 9 的模型與 9P 協議，蓋在 Linux 上（FUSE / `v9fs` 9P mount），handy 試驗田原封保留**；
**不去 boot Plan 9/9front 當底座**——那會丟掉整個 LuaJIT/Fennel＋真 LLM 後端的 Linux 生態。
**「以後有需要再換到 Plan 9（p9）」**——真 OS 這條留作長遠 OS-as-game 若想要乾淨基材時的逃生門，非現在。

- 〔錨定〕9P 在 Linux 活得好好的：核心 `v9fs`、QEMU virtio-9p、WSL2 就是用 9P 互看檔案——量產技術非考古。
- 〔錨定〕plan9port（lib9p＋Plan 9 userland）＋各語言 9P 庫（含 Lua）可用。

### 3.2 順手清掉的舊懸案〔決定〕

- **四通道**（§4.3）→ 塌縮成 Plan 9 慣例：**data 檔讀寫 ＋ ctl 檔寫控制命令**。
- **「元素須是 list 且具特定成員承接 argc/argv」規範** → 換成 Plan 9 慣例：**一個目錄含 `ctl`/`data`/`status` 等約定檔**（更成熟、幾十年驗證）。
- **Godot 節點樹類比**（兩層底座篇 疑點 3）→ **Plan 9 namespace 比 Godot 更貼**：它本就統一「磁碟＋執行態」為一棵可 walk 的樹，正是 Godot 類比想要、但 Godot 只做到記憶體那半的東西。
- **語言中立** → 由「講 9P」保證，接上回到初心宗旨。

## 附錄 A、Linux「一切皆檔案」交互設施盤點〔錨定・技術地基〕

> 動手選「怎麼跟一個路徑元素交互」前，先把 Linux 實際提供的整套 file/fd 設施攤開——才知道是在已知調色盤裡挑，而非憑空發明。這份盤點推出了 §三「四通道該塌縮成讀寫＋帶外控制」與方案 A 採 Plan 9。

### A. 通用動詞（幾乎對任何 fd 都成立）——Unix 的核心答案

| syscall | 作用 |
|---|---|
| `read` / `write` | **唯一的通用介面**。「一切皆檔案」的精神＝同一對動詞作用於所有東西 |
| `mmap` | 把 fd 映進位址空間 ＝ **shared memory** 那條 |
| `ioctl` | **逃生艙**：物件專屬的帶外控制命令。read/write 表達不了的都塞這 |
| `fcntl` | fd 控制：非阻塞旗標、**檔案鎖**、`O_ASYNC`→就緒時發 `SIGIO` |
| `poll`/`select`/`epoll` | 等就緒（可讀/可寫），多路複用基石 |
| `lseek` / `close` / `dup` | 定位 / 生命週期 / 複製 |

### B. 特殊 fd 型別（核心把「本來不是檔案的」也做成 fd）

- **pipe / FIFO（有名管線）** ＝ **pipeline** 通道；單向位元組流
- **socket**（`AF_UNIX` 本機 / `AF_INET` 網路）——**最通用的雙向 IPC**；能靠 `SCM_RIGHTS` **把活 fd 傳給別的 process**、還能傳身份憑證。「取元素＝拿 fd」要跨 process 時的關鍵
- **`/proc`、`/sys`、`/dev`**——偽檔案系統，把**進程與核心狀態掛成路徑**。**「執行態可路徑定址」的現成先例**
- **`*fd` 家族**（近代核心趨勢：萬物 fd 化好讓 `poll` 統一對待）：
  - `signalfd`——把**信號變成可讀 fd**
  - `timerfd`——計時器到期變 fd 事件（**直接對應 tick**）
  - `inotify`/`fanotify`——**監看路徑變動**（直接對應「借樹」偵測狀態改變）
  - `eventfd`——輕量計數器信號
  - `pidfd`——**把 process 當 fd 持有**（現代版 process-as-file）
  - `memfd`——RAM 裡的匿名檔，可經 fd 傳遞共享

### C. 帶外機制（process-directed，非 fd-directed）

- **signal**：有，但朝進程發、很粗（一個號碼＋選配 siginfo），非朝 fd。要 file 化用 `signalfd`，或 `O_ASYNC`+`SIGIO` 讓 fd 就緒時發信號
- **SysV IPC**（訊息佇列/號誌/共享記憶體）：老式，用 **key 而非路徑**定址——**跟「一切皆路徑」不合拍**，可略
- **shared memory 三條路**：`mmap` 一般檔（共享）／POSIX `shm_open`（落成 `/dev/shm/<名>`，**仍是路徑**）／`memfd`

### D. 三個對本專案的啟示

1. **`/proc` 是祖師爺**：「把執行中 process 的環境掛成路徑」是 `/proc/<pid>/…` 的既有範式，階段一先吃透它。
2. **read/write 是唯一通用動詞、ioctl 是帶外逃生艙**：本專案「四通道（argc/argv、shared memory、pipeline）」放進此坐標＝*argc/argv＝exec 啟動時*、*shared memory＝mmap*、*pipeline＝pipe/socket 流*。而 Unix 更省的答案是「**開路徑拿 fd → read/write → 控制走 ioctl**」——四通道會**塌縮**。**Plan 9 更進一步用 `ctl` 檔取代 ioctl**（見 §三）。
3. **`*fd` 家族＝直覺的背書**：核心近十年就在做「萬物 fd 化好讓 poll 統一」，與「歸一於路徑/fd」同構；且現成好用——**timerfd 給 tick、inotify 給借樹偵測、pidfd 給 process-as-file**。
