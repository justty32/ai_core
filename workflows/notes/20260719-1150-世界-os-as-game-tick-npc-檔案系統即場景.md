> 記於 2026-07-19。承接 [0718 §1.2 Linux-as-Lisp](20260718-2324-so與腳本共存-呼叫閘道-value跨邊界.md)

## 〇、一句話

把整台作業系統當成一個**回合制世界**：**檔案系統即場景**、**資料夾即空間分層與定義**、**process/腳本即 NPC**、**使用者即玩家（且同時是場中的角色）**、**tick 即世界時鐘**。這是一個**很緩慢的循環**——與 CPU 週期、與 in-process 記憶體管理**無關**；狀態靠 git 落盤，故一切以**文字 ＋ Linux 原生特性**為根據。

## 一、兩層底座

- **下層＝Linux-as-Lisp（既有抽象層）**：檔案＝函數、`exec`＝apply、pipe＝組合、fs＝environment（見 0718 §1.2）。
- **上層＝OS-as-game（本篇新語意）**：在同一副基材上換一組隱喻——作業系統即遊戲、agent 即 NPC、使用者即玩家。兩層講的是**同一堆 Linux 事實**，只是視角不同。
- **可接入・Lisp**：既然 Linux＝Lisp，**Lisp 本身也能無縫接入 os as game**（見 §十）。

## 三、三軸縮放：時間／空間／規模

〔決定〕場景有**時間／空間／規模**三類縮放。可稍微限縮在特定視角看：

### 3.1 空間＝檔案系統
當前的檔案分佈＝當下時間點的**局面**。用**資料夾分層**，以及在資料夾裡放**定義**。

### 3.2 時間＝tick 推進
要從當前局面推進到下個時間點，就**執行 tick**——它是**推動所有 NPC 行動的起始點**。（詳見 §四。）

### 3.3 規模＝複雜度分層（尺度階）
〔決定〕規模＝workflow 的複雜性分層，一條由簡到繁的階梯：

```
檔案  →  拆檔成資料夾  →  資料夾正規化  →  子專案
```

〔錨定〕資料夾分層在本系統**一物三用**：同時是**空間**分層、**時間**分層（見 §4.3 tick 套 tick）、**複雜度**分層。

### 3.4 略圖規則（LOD）〔未定〕
三軸縮放時「細節怎麼省略」有一套規定——**使用者明說還沒想好，先別寫死**。（對映遊戲的 level-of-detail：時間快轉時 NPC 行動如何摺疊、空間縮小時子資料夾如何抽象成一個代理值……全部留白。）

## 四、tick：世界時鐘

### 4.1 tick 做什麼
〔決定〕tick 是**推動所有 NPC 行動的起始點**；它會**等到所有 NPC 都執行完之後才結束**。

### 4.2 誰終止一輪 tick
〔決定〕**玩家也是角色**，所以 tick **也要等玩家結束**；而**玩家的結束＝玩家說「這輪到此」**。→ 一輪 tick 的實質終止訊號，就是玩家（作為場中角色）下的收尾令。

〔決定・使用者參與〕tick 執行期間，玩家可與各類**正在運行中的 NPC 交互**；此刻玩家本身就**參與進這一輪 tick、成為其中的 character**。

### 4.3 tick 套 tick（＝空間套空間）
〔決定〕tick 可以巢狀，正如空間可以套空間（資料夾下有子資料夾）。機制：

- **一個 tick 只負責運行「指定空間」（指定資料夾底下）的 NPC**。
- 故資料夾分層**同時是對空間與時間做分層**。
- **上層資料夾的 tick 有時會驅動下層資料夾的多次 tick**（外層一步、內層跑好幾回合）。

### 4.4 tick 收尾
〔決定〕收尾時 tick 會：

1. **給所有正在運行的 NPC 發送結束指令**，讓他們**暫存進度**（checkpoint）。
2. **（選擇性）git commit** 所有檔案更改——**不一定每 tick 都 commit**；commit 只是方便回檔，**由 tick 自己決定**要不要存這一檔。
3. **順手把 server 型程式收掉**。對 server：可用**「假存活」**——在 tick 看來該 server 已**訪問不到**即可，**不必真動到 server 本身，而是 tick 這邊自己做封鎖**。〔實作細節・到時就知道〕

## 五、NPC：住民

### 5.1 NPC 是什麼
〔決定〕NPC 即 **process／腳本**。它能**讀取自己的位置**（自己在哪個資料夾底下），並**根據所在資料夾的相對位置去操作其他檔案**——如同一個人有其定位。這組「對檔案的操作」＝它的**手**（ABI/API）。

〔決定・組成〕NPC 底下可以掛 sub-process；NPC 也可以是**一段事件**，也就是**一個人的行動**、或**一群人的行動**。〔粒度・未定〕「NPC 到底是 process、是事件、還是可遞迴含子 NPC」——使用者還有更多想法，**先別寫死**。

### 5.2 主體性軸
〔決定〕NPC **不一定要有主體性**。**「有主體性」＝他有自己的記憶、思想、個性等等，也就是「人」**；沒有主體性的 NPC 就是純事件／決定性腳本／一段被觸發的行動。（注意：這根軸**不是**「有沒有掛 LLM」，而是「**有沒有自我（記憶＋思想＋個性）**」。）

### 5.3 空間隔離＝軟邊界
〔決定〕tick 空間限定（只跑指定資料夾下的 NPC）與「NPC 可依相對位置伸手到別處檔案」之間**不設硬隔離**——**這是遊戲規劃者自己去承擔的風險**。

## 六、界面：shell 指令 ＋ 檔案 CRUD

〔決定〕使用者需要一個好界面與這套系統交互。交互方式**只要兩種就夠**：

1. **下 shell 指令**——在 Linux-as-Lisp 下，這不是「撰寫程式」，而是**修改程式的行為**。
2. **創／改／刪檔案**。

「shell 指令 ＋ 檔案 CRUD，這就夠了。」

### 6.1 〔決定・升級〕Lisp REPL 當兩世界的統一界面
把上面兩種動作各升一級，界面就橫跨兩個世界（§八）統一起來：

- **shell → Lisp REPL**（界面本身換成 REPL）。
- **修刪改檔案 → 擴展成修刪改 process 內的 list**（Lisp 執行期即時修改 / live runtime modification）。

| | 慢世界 | 快世界 |
|---|---|---|
| 被編輯的資料 | **檔案** | **process 內的 list** |
| 動作 | 修/刪/改（REPL eval） | 修/刪/改（Lisp 執行期修改） |

〔本質〕Lisp 的**同像性**讓「檔案（data）」與「process 內 list（data）」成為**同一種可操作物**——玩家（作為 character）用同一組 REPL 動作，在慢世界改檔、在快世界改活的 list，是**同一件事的兩個尺度**。「那一切就都能通了」：這把 §六界面、§八兩世界、§九通聯、§十 Lisp 接軌全接成一條。

## 七、為何一切走文字 ＋ Linux 原生特性

〔決定・根據〕

- 這是一個**很緩慢的循環**：與作業系統的 **CPU 週期無關**、與 in-process 內各類**記憶體管理無關**。
- **因為要用 git 保存**，所以都推薦用**文字 ＋ Linux 原生特性**作為 NPC 的根據。
- 也就是說：**NPC 都是腳本**，並充分利用 **symlink 等 Linux 檔案特性**作為資料組織形式。

## 八、兩種世界類型：慢·明朗 ↔ 快·隱含

〔決定〕**tick 驅動的對象決定世界類型**。os as game 平常是「tick 驅動一個資料夾（一個場景）」；但若 **tick 要驅動一個檔案**呢？

〔決定・tick 帶背景認知〕tick 的驅動**不只是普通呼叫函數**，還帶著**一套 agent/NPC 機制的背景認知**——所以「驅動一個檔案」不是裸 exec，而是**把它當 NPC 喚醒**。

〔決定・分形內縮／對外對內〕**tick 驅動檔案 ＝ 執行該檔案、把它變成一個 process**：

- 這個 process 裡面**又有 process（或 thread）**。
- **對外**：整個 process 是一個 **NPC**。
- **對內**：process 內部**也各有 NPC**，但**隱性、不對外公開**。

〔本質〕於是有**兩種世界類型**，快世界＝慢世界的**分形內縮**（同一套 tick/NPC 機制，驅動對象從資料夾降到檔案，世界就切檔）：

| | **慢世界** | **快世界** |
|---|---|---|
| 驅動對象 | 資料夾（場景） | 檔案（→ process） |
| 質地 | 明朗、簡單、純靠 OS | 隱含、靈活 |
| 速度 | 慢（不管 CPU/記憶體） | 快（in-process） |
| NPC 可見性 | 對外公開 | 對內隱性、不公開 |
| 話語 | 檔案／腳本／JSON | process／thread／struct |

〔錨定〕**in-process/.so/`Value`（0718 so-value 筆記）正是活在快世界裡的那套**——這就是先前〔衝突 2〕的解：兩者不對衝，是**同一機制的兩層尺度**。

## 九、兩世界通聯協議

〔決定〕既有兩種世界，就要做**一套標準／協議**，使兩個世界（慢·明朗 ↔ 快·隱含）能**通聯**。

〔提案・接回既有物〕[0718 so-value](20260718-2324-so與腳本共存-呼叫閘道-value跨邊界.md) 的**呼叫閘道 ＋ 一個 `Value` 兩種物理形態**很可能就是這條協議的雛形：**跨行程 JSON＝慢世界的話語、in-process struct＝快世界的話語，同一 `Value` 語意兩面孔**。〔待追認〕

## 十、Lisp 接入：兩套庫

〔決定〕既然 **Linux＝Lisp**，Lisp 本身也能無縫接入 os as game。為此做**兩套庫**：

1. **`--meta` 那套** → 讓 **Linux 真正等同於 Lisp**（補上 Linux-as-Lisp 缺的「血肉」；對映 [0718 §1.2](20260718-2324-so與腳本共存-呼叫閘道-value跨邊界.md) 點名的三塊血肉：統一值模型／便宜呼叫／closure）。
2. **一個 Lisp lib** → 讓 **Lisp 與 Linux 無縫接軌**、輕鬆套用 OS 的哲學。

〔分工・互補〕os as game 是**非常緩慢**的系統（不管 CPU 週期、不管記憶體管理）；**Lisp 接入後正好補上這塊**——即 §八的**快世界**那一端。

### 10.1 〔決定〕做什麼：Lisp lib ＋ OS 層級標準（兩者互補、缺一不可）
盤點兩條路的利弊，說明為何不能只挑一條：

- **A・只做一套 Lisp lib**：利＝能弭平快慢世界鴻溝、**用統一方式操作**；弊＝**所有程式與檔案都要用 Lisp 寫**，且 **Linux 的一些機制被架在 Lisp VM 之上、不夠穩妥**。
- **B・做標準規範，讓快慢世界互相通訊、趨於一同**：利＝**既能用到 OS 的穩妥、又能用多語言的優勢**；弊＝**操作界面必須仰賴 shell**、要改檔案內容之類（不如 REPL 統一優雅）。

〔決定〕**兩個都要**：除了 Lisp lib，還需要 **OS 層級的標準**。Lisp lib 給統一優雅的界面（A 的利、對映 §6.1 REPL），OS 標準給跨語言跨世界的穩妥通訊（B 的利、把 §九的通聯協議升級成**真正的 OS 層級標準**）。兩者互相補掉對方的弊——**os as game 才能在這套平臺上運作順暢。**

## 十一、多根軸的弭平，與通用協議＝檔案

### 11.1 又一根軸：動態／靜態（記憶體／硬碟）
〔決定〕除了快/慢世界（§八），還有一根**動態／靜態**軸：

- **靜態 ＝ 硬碟 ＝ 檔案系統**
- **動態 ＝ 記憶體 ＝ process**

〔提案〕這塊也可套 Linux 哲學（一切皆檔案）→ **讓 process 乃至 thread 也成為檔案**（近似 `/proc`）。

### 11.2 分層可以很靈活
〔決定〕目前的 **2×2 分法（快/慢 × 動/靜）只是比較大的鴻溝、必然要分層的天然分隔點**；實際**分層可以很靈活、可手動自建**：

- 例：tick 這塊，拿「大資料夾＋小資料夾」自訂——跟大資料夾**平級**的算一層、底下再分一層。**這本身就是複雜度的分級。**
- **記憶體內的分層 ＝ 由程式自己弄**（相對於檔案系統資料夾的結構性分層，記憶體那層由 process 動態決定、更靈活）。

### 11.3 層間協議＝複雜度分級的體現
〔決定〕分層之後，**兩層之間要做相容 ＋ 協議**——這就是複雜度分級的體現。例：**某層可以把複雜度包裝變簡單，用協議搞定。**

### 11.4 〔決定〕通用協議＝檔案（Unix 檔案哲學）
總之：要有一個**貫穿所有分層、統一「時間 × 空間 × 複雜度」的通用協議**。**這個協議 ＝ 檔案**，根基是 **Unix 檔案哲學**（一切皆檔案）。

一旦認定唯一協議是檔案，前面那堆分層／兩套庫／通聯全塌回同一抽象——**「事情就這樣變簡單了」**：複雜度不是靠更多機制壓下去，是靠**選對唯一協議**自然消解。

### 11.5 檔案的本質＝路徑＋副檔名；三軸收斂到路徑
〔決定〕**檔案的本質 ＝ 把「我在哪（路徑）」＋「我是啥（副檔名）」擺在一起**：路徑＝**空間座標（where）**、副檔名＝**型別/身分（what）**。〔錨定〕這正是 [0708 §四](20260708-1040-一切皆檔案-list-統一描述子-複合路徑.md) 統一描述子的 `identity`——身分是檔案存在形式**自帶**的，非外掛 metadata。

於是 §三的三軸縮放，全部化約成「**路徑**」這一個維度：

| 軸 | 用路徑怎麼表示 |
|---|---|
| **空間** | 路徑直接就是空間座標，在檔案系統中收斂 |
| **時間** | 時間也是空間的一個維度 → 用路徑表示「指定時間的空間狀態」＝**版本管理**（簡單打包封存／git） |
| **複雜度** | 資料夾的**分層深度** |

〔延後〕更複雜的相對論式時間是**另一套東西**，使用者心裡有想法、之後再說。

## 十二、與既有線的接點、命名與開放項

〔錨定・既有〕

- **「repo 才是持久層」**（memory 鐵則）＝本篇 git 存檔語意的同一件事：fs 本身 durable，不靠 image dump。
- **Linux-as-Lisp**（0718 §1.2）＝本篇下層底座。

〔待辦・命名對齊〕本篇把 NPC 暫稱「**住民**」，但 `handy` 子專案已用「**住戶**」稱它的 callable。兩者是不是同一個東西？要對齊，別各叫各的。

〔已收斂・與 in-process/.so 對接〕本篇「NPC 都是腳本、純文字、與 in-process 記憶體無關」與 [0718 so-value 筆記](20260718-2324-so與腳本共存-呼叫閘道-value跨邊界.md) 的 in-process `Value`/`.so` 快路徑，**本輪已由 §八「兩種世界」框架接上**：in-process/.so 不是對衝，是**快世界**那一層（tick 驅動檔案→process→對內隱性 NPC）。**通聯協議見 §九**（`Value` 兩形態）。剩下的〔開放〕只是**協議細節與追認**，不再是路線衝突。

〔旁註・prior art，待錨定〕「檔案系統當沙盒、有記憶的 agent 當 NPC、回合推進」近似 Stanford *Generative Agents / Smallville*；「玩家寫腳本驅動 NPC 在世界裡行動」近似 *Screeps*。日後落實時該掃這兩者、界定白地在哪。
