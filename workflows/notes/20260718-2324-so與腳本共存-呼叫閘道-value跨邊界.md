## 〇、一句話

一個**函式**在系統裡的存在形式，統合在「**呼叫綁定**」這一層、不在「檔案」這一層。OS 早給了兩套通用 ABI——**process ABI**（argv/exit/stdio，粗粒度、有隔離、慢）與 **linking ABI**（C ABI，typed、in-process、快、脆、無隔離）。腳本與 `.so` 是**對等的兩種材料**，各有各的邊；黏起它們的，是**一份 meta ＋ 一個 argv 同形的呼叫契約**，落地就是給既有 `lib/call.py` 閘道再加一個 `.so`（dlopen）kind。而「值怎麼跨邊界」的答案是：**同一個 typed `Value`、兩種物理形態**——in-process 傳 C struct（零序列化），跨行程才序列化成 JSON。

## 一、三個 facet 的收斂（願景層）

### 1.1 檔案雙形態 → 統合點在「綁定」不在「檔案」
〔錨定事實〕Linux 一個 ELF `ET_DYN` 檔可同時是 `.so` 與可執行檔（給 `DT_INTERP` ＋一個處理 argc/argv 的 entry；glibc `libc.so.6` 即此）。但這是 party trick——只有編譯型語言能產出，省的邊際。

〔決定・提問逼出〕真正有價值的統合不是「同一檔案既是 .so 又是 exe」，而是「**同一函式定義、能被 process 叫／被 C ABI 連／被腳本 import**」。這**不需要同一個檔案**，一份 `--metadata` ＋ codegen 就能長出多綁定。

〔劃清〕LLM 中和了什麼、沒中和什麼：
- **中和：編寫摩擦**——寫 C/Rust、驅動編譯器、修 linker flag。「編譯型語言太麻煩」這個*歷史理由*失效。
- **沒中和（架構本身，與誰寫無關）**：① 隔離（in-process 一崩全崩）；② 零相依鐵律（`.so` 要目標機有編譯器）；③ ABI/版本協調（`.so` 被 link 後改介面＝破壞性）。

### 1.2 Linux-as-Lisp：骨架免費、血肉才是工
〔提案〕把整個 OS 當 Lisp 用時，**檔案就是函數**——但 Linux 只免費給了 Lisp 的**骨架**，沒給**血肉**，血肉那塊正好是 ai_core 要做的事。

| Lisp | Linux 對應 |
|---|---|
| environment（名→值） | filesystem（path→file） |
| 符號解析順序 | `$PATH` |
| `let`/作用域 | 目錄/子目錄 |
| `apply` | `exec` |
| 組合 | pipe `\|` |
| **code=data（同像）** | 同一檔案：`cat` 它＝quote／`./` 它＝eval |
| 持久 image（要 dump） | fs 本身 durable（＝「repo 才是持久層」） |

**沒給的血肉（＝真正的工）**：① **統一值模型**——Lisp code/data 共用一種結構（list），Linux 檔案是異質 blob，`--metadata`＋wire format 就是在人工製造這個缺掉的統一性；② **便宜的呼叫**——`(f (g x))` 是函式呼叫，pipe 是 fork/exec，慢幾個數量級；③ **closure**——檔案沒法 close over 環境，**但解就是鑄刀**：把 arg 焗進去產生新檔案＝檔案化的 closure＝部分求值，Lisp closure 在記憶體、你的在磁碟。

〔提案・效能焦慮真解＝兩層〕**組合層當 Lisp（檔案/exec/pipe），實作層用各語言生態**。不會把 `add1` 做成檔案去 exec——`file=function` 只在粗粒度成立（呼應 0708 §一「粒度是硬邊界」）。`.so`/C-ABI 快路徑就是給少數*熱的組合邊*把 fork/exec 成本補回來。

〔錨定・誠實定位〕「一切皆檔／OS 即函數式組合」不是本專案發明的基材，站在巨人肩上——**Plan 9**（一切皆檔、fs 當通用介面）、**Unix pipe 哲學**、**Nix/Guix**（檔案＝純函數、內容定址，且 Guix 真用 Scheme 描述）、**nushell**（typed structured data 走 pipe＝「值模型」的現成參考答案）。基材成熟＝好事；**白地在上層**：LLM-as-peer ＋ `--metadata` 自我描述 ＋ 結晶飛輪 ＋ 憑證治理，四個疊在老基材上才是新組合（與 roadmap 判斷一致）。

### 1.3 script ⇄ .so：對等共存的兩種材料
〔決定〕不是「腳本可塑、.so 只是結晶端」的單向梯度，而是**對等**，按邊的取捨自由挑；LLM 兩種都便宜地代寫，選擇免費且可反覆改。有些東西生來就該是 `.so`（效能原語）、有些永遠腳本（膠水），「熱路徑證明後 script→重編 .so」只是其中一條常見軌跡。

| | 腳本（text） | `.so`（binary） |
|---|---|---|
| 生成/修改 | LLM 原生、即改即跑、可 diff/grep/include | LLM 寫源碼再編（作者摩擦已被中和） |
| 速度 | 慢（fork/exec＋序列化） | 快（in-process、typed、零序列化） |
| 隔離 | 有（process 邊界） | 無（一崩全崩） |
| 相依 | 只要直譯器 | 目標機要編譯器 |
| 演進 | 鬆（改一行） | 緊（ABI 破壞性、要 lockstep） |

### 1.4 〔原則・後補〕LLM 砍平「人因軸」→ 語言選擇退化成純能力題
〔提案〕過去挑語言混著「好不好寫、熟不熟、避開難的」＝**人因**。LLM 把這根軸砍平到 ≈0（語言多複雜都能 handle）。砍平後**露出底下真正的軸＝能力/本質**——它 LLM 碰不到：
- **生態**：庫/runtime 能力。要 torch 就得活在 Python process 裡；LLM 生得出膠水，**生不出只存在於某語言世界的庫**。
- **語言本質**：GC vs 手動、動態 vs 靜態、GIL/green-thread/homoiconicity/borrow-checker——**不是「學得會的複雜」，是不同計算基材**；LLM 寫再完美的 C 也不會讓 C 長出 GC。

〔推論〕於是「選語言」退化成**純能力配對**：每個函式按「這活需要哪個基材/生態」挑材料，**切換成本 ≈0**——因為 **LLM 付編寫成本、`.so`/引擎 adapter 付互通成本**（§二、§四）。這兩個成本被吃掉，語言選擇才成為純能力題。

〔錨定・給「語言中立」補上硬理由〕這正是「回到初心／規範層語言中立」一直缺的地基：**契約層語言中立（meta ＋ `Value` ＋ C ABI），實作層多語言、按能力選、自由切**。之所以值得付 adapter 稅容納 lua/py，就因 essence＋生態無可替代，而 LLM 讓「為每件事挑對的刀」變免費。

〔旋鈕・對映本輪一段岔想〕「全用 C/C++ 當腳本寫」很誘人（`tcc -run` 讓 C 當腳本、`cling` 是 C++ REPL，現成），因為同質 C 把整張 §3.1 的 2×2 塌回唯一乾淨那格（`.so↔.so`、零 marshal，adapter/marshal 全免）；C++ 又比 C 高一截（`std::string`/`variant`/json 抬高膠水地板）。但那等於**賭「所有活都落在 C++ 能力半徑內」**，torch/瀏覽器當場證偽。故：
- 不把「全 C」當方案，把 **C-as-script 當 `.so` 塊的*快速前端***——唯一「腳本手感」與「.so 原生零 marshal」能兼得的語言（C 編譯夠快、產物本就是 C ABI）。
- 實作走 **folder-as-callable `_exec` 現場編譯＋按 source hash 快取**（改 `.c`→自動重編 `.so`→dlopen；`tcc -run` shebang 或 compile-cache 皆可——§4.2 已收回 drop-shebang）；此快取＝內容定址的凍結計算，踩上 0708 §一「data↔code 互轉 / Nix derivation」那條線。

〔旋鈕一句話〕**同質 C/C++＝最乾淨/最少生態；多語言＝最多生態/付 adapter 稅。答案不是挑一格，是讓切換免費。**

## 二、核心物：呼叫閘道 lib（接 `call.py`，加第四 kind）

→ **`.so` 就是 `from_spec` 裡的第四個 `kind`（dlopen）**。閘道形狀早定；本輪工作＝加這個 kind ＋ 定「值怎麼跨邊界」（見 §三）。

## 三、〔決定〕值跨邊界：一個 Value、兩種物理形態

使用者拍板 **wire format 用 typed**。提問「C ABI 的 struct 不是通用的嗎」逼出正確模型——**C ABI struct 確實通用，但只在「同架構、同進程」這個硬邊界內**：

- **通用處（in-process 熱路徑）**：同 ABI 目標上 struct layout 是編譯器全體同意的固定事實，LuaJIT ffi / ctypes / libffi 靠的就是這個。**同進程根本不需要 JSON——直接傳 struct，各語言 FFI 按已知 layout 讀。** in-process，struct 本身就是那個 typed value，零序列化。
- **不通用處（四個 gotcha，第一個致命）**：① **指標只在同一位址空間有效**——含指標的 struct 跨進程即垃圾，必須攤平＝序列化（這是 JSON/CBOR 存在的根本理由）；② layout 綁架構/位元序，跨機/落盤不可攜；③ 字串/list/record 這些變長物要靠指標，回到①；④ struct 無自述，兩端得共用 `.h`、加欄位即 silent corruption。

〔決定〕定一個 canonical **`Value` tagged-union struct**（＝ turn-1「lib 當共同標準」的精確化）：
```c
typedef enum { V_NULL,V_INT,V_FLOAT,V_BOOL,V_STR,V_BIN,V_LIST,V_RECORD } VType;
typedef struct Value {
  VType type;
  union { int64_t i; double f; bool b;
          struct { char* p; size_t n; } str;   // 指標 → 只 in-process
          struct { struct Value* items; size_t n; } list;
          struct { char** keys; struct Value* vals; size_t n; } rec; } as;
} Value;
```
**同一個 typed 值、兩種物理形態**，閘道按綁定自動選、呼叫方無感：
- **in-process / `.so`**：直接傳 `Value*`（帶指標、豐富、零序列化）。struct 就是 wire。
- **exec / 落盤 / 過網**：把 `Value` 走一遍攤平成自述 bytes。**JSON 預設**（守規範、當 exec 地板）；序列化入口帶長度、不用 NUL 結尾 `char*`：`call_bytes(const char* fn_name, const uint8_t* in, size_t in_len, uint8_t** out, size_t* out_len)`。

〔決定・這把「typed 逆規範」的擋路決定溶掉了〕`workflows/spec/data_format.md` 管的是**跨行程/跨工具的 wire**（既有硬立場＝JSON）。**in-process 傳 struct 根本沒跨那條邊界 → 不違反規範。** 於是：
- typed `Value` struct ＝ in-process 表示（**規範中立**）
- JSON(+schema) ＝ 跨行程邊界（守 `data_format.md`；且 `lib_spec/s1-io-entries.md` 既有 `type:{base:text|binary}`＋`schema:(JSON Schema)` 已是「接縫 typing」的載體）
- 〔延後〕CBOR 從「擋路決定」降級成「日後要不要給跨行程路徑再加一個緊湊二進位編碼」的優化選項，**現在不必決、也不逆規範**。

〔必記殘餘代價〕struct 路徑＝**最緊耦合**（共用 `.h`、版本 lockstep）——對回信任層：只有驗證過的熱邊值得付這耦合換速度，其餘走 JSON 鬆耦合。

### 3.1 〔修正・提問逼出〕第二根軸：邊界 × 知不知道簽名
§三上文把物理形態綁在「in/out 進程邊界」——**對，但只講了一半**。「JSON 解析來解析去很煩」其實不是 out-process 的稅，是**「不知道對方簽名」的稅**（反證：in-process 的泛型入口 `do(json_t*)` 也照樣用 JSON）。真正是兩根**正交**軸：

- **軸一・進程邊界（in/out）**：決定指標活不活、要不要序列化、有沒有隔離。in＝struct/指標活著、可零拷貝；out＝**一律**序列化（指標過不了界）。
- **軸二・簽名已知否（known/unknown）**：決定要不要走「萬能入口＋自述 payload」。known＝typed 直呼；unknown＝泛型入口＋`Value`＋現場發現。

| | **known**（build 時有 `.h`） | **unknown**（現場才知） |
|---|---|---|
| **in-process** | typed 直呼具名函數，~零 marshal | 萬能入口 `call_value(Value*)`，要 marshal |
| **out-process** | （少見）照樣序列化 | `exec` ＋ argv/JSON |

「typed 零 marshal」是 **known 的禮物、不是 in-process 的**；「JSON 煩」是 **unknown 的稅、不是 out 的**。

〔提案・萬能入口符號〕規定每個合規 `.so` 都導出萬能入口，**不必事先知道它的具體函數**就能泛型呼叫：`dlopen`→找萬能符號→按 `fn_name`＋`Value` 派發。**這一招同時吃下「in-process 泛型」與「未知的 `.so`」兩個 unknown 格。**

〔訂正・萬能入口是兩個符號、不是一個〕原稿把 in-process 的 `Value*` 入口與序列化 bytes 入口都叫 `call`，但 C 的 `.so` **無法用同一符號名導出兩種簽名**——這正是 §3.1 兩根軸（in/out × known/unknown）在符號層的體現，命名必須跟著分家。定為**兩個萬能符號**：
- `int call_value(const char* fn_name, Value* in, Value* out)` — **in-process** 直收帶指標 `Value*`（軸一 in、軸二 unknown）；
- `int call_bytes(const char* fn_name, const uint8_t* in, size_t in_len, uint8_t** out, size_t* out_len)` — **序列化** bytes 入口（§三，跨行程／落盤／過網用）。

閘道側可只硬性要求其一、另一個由閘道用 §三的 `serialize/deserialize` 就地包成：最小合規＝導出 `call_value`，`call_bytes` 讓閘道在 `Value` 攤平前後補上；反向亦可。**兩個符號、一份 `Value` 語意，物理形態分家。**

〔修正・§四 case 分類〕故一個「build 時不知其 C ABI」的 `.so`**不必然掉去 exec**：可現場 `--emit-header` 問簽名、或直接用萬能符號。exec 的真理由是**隔離/信任**（不想污染位址空間／怕 ABI 對不上），**不是「無知」**——把這兩個 exec 理由拆開。

〔小修・型別〕in-process 萬能入口收 **`Value*` 不收 `json_t*`**：`json_t` 是某 JSON 庫的樹，收它＝把 JSON 庫請進記憶體、又把 wire 序列化和記憶體表示混一談。**`Value` 在記憶體、JSON 只在線上。**

〔背書・`--emit-header`〕CLI 標準加一個「輸出本檔 `.h`（怎麼用的說明）」選項＝`--metadata`（`cli_spec` 已要求）的 **`.h` 投影**：一份 meta、兩個面孔——`.h` 給 C linker（typed 那半）、JSON schema 給其他人（0708 §四 identity/static/dynamic）。**但別過度期待**：編譯好的 C 程式無法「現場讀 `.h` 就獲得 typed 呼叫」（C 靜態編譯），`.h` 的用途是 build-time/codegen/LLM 生綁定/libffi/驗證；現場動態 in-process 靠的是萬能 `Value` 入口、不是讀 `.h`。

### 3.2 〔提案・未追認〕第三物理形態：shm 無指標 binary（把 C ABI 偷塞進 out-process）
out-process 不必只有 argv/JSON。**同機同架構**可用 **shared memory ＋ 無指標/offset-based 固定 layout**（＝FlatBuffers/Cap'n Proto 的 zero-encoding），收方 `mmap`＋加自己的 base 直讀、**零 parse**。前例：X11 MIT-SHM（socket 協議、同機偷用 shm）、Android Binder。
- **關鍵條件**：payload 必須**無指標**（存 offset 非絕對指標，因對方 mmap base 不同）。一旦無指標，**同一坨 binary in-process（cast 直讀）與跨行程同機（mmap＋base）皆零 parse**。
- 於是物理形態是**三種**（在 struct↔JSON 之間插一格），合成一條**四格連續譜**，按「信任 × locality」在握手時用 capability flag 協商，不支援就退回 pipe/JSON：

| 形態 | 適用 | 信任 |
|---|---|---|
| 帶指標 `Value` struct | 同 proc、最快最豐富 | 同 proc |
| **無指標 binary arena + shm** | 跨 proc 同機同架構、零 parse | 可信快路徑 |
| pipe JSON/CBOR | 跨機/跨架構/自述、parse 稅（地板） | 任意（kernel 隔離） |
| 序列化（分散式層） | 跨機 | — |

- **成本**：無指標 layout＝手刻迷你 FlatBuffers（零相依下養得起、是碼），只對**熱/高頻/同機**邊值得；只同機同架構（跨機＝0708 §三分散式層照序列化）；shm＝共享可變面 → **信任快路徑（信任門第三次出現）**。
- **大圖**：規範**橫跨四格**、不是只在 out-process；速度是握手協商出的編碼選擇——**out 與 in 不是兩國，是同一把尺按 locality 滑動的刻度**。此提案未經使用者追認。

## 四、〔決定〕`.so → 腳本` 兩條都要；門檻＝可信 ∧ 支援

〔決定〕`.so` 要回叫腳本時：**支援的語言 → in-process 嵌直譯器；不支援 → exec 出去**（同一 wire、呼叫方無感，in-process vs exec 只是同一入口背後的實作細節）。

### 4.1 〔修正・提問逼出〕「嵌直譯器」的精確樣子：引擎 `.so` 才是那個 `.so`，腳本函數不是 peer
使用者一度想把「script 四捨五入成 `.so`」，推到「叫腳本函數＝讀它 `.h`、link、直呼」——**卡住了，因為腳本函數根本沒有 `.h`**。拆開，一句「script = .so」壓了三個不同的東西：

| 口語的「script」 | 實體 | 是 `.so` 嗎 |
|---|---|---|
| **引擎**（libpython.so / libluajit.so） | 直譯器本體，有 C ABI、有 `.h` | ✅ **真的是 `.so`** |
| **腳本文本**（foo.py） | *資料*，餵給引擎的 argument | ❌ 無 `.h`、無 C ABI |
| **腳本裡的某函數**（一個 `def`） | callable，活在引擎 runtime 裡的*值* | ❌ 只能透過引擎 C API（`PyObject_CallObject`）叫，無自己的 `.h` |

〔提案・乾淨模型〕**整個系統只有 `.so`；腳本不是 peer `.so`，是躲在「引擎 `.so`」後面的資料**：
- **`.so ↔ .so`**：對等、讀 `.h`、link、in-process 直呼，**零 marshaling**（同一 C struct）。
- **`.so → 腳本函數`**：**不直接叫**。叫**引擎 `.so`**（讀*它*的 `.h`），叫它「跑這段文本／用這些參數叫這個名字的函數」。**引擎 `.so` ＝萬能 adapter**：一份 `.h`，host 任意多腳本函數。→ 每語言只貢獻**一個** `.so`（它的引擎）。

〔本質・使用者感到「不對」的來源〕`.so↔.so` 是零 marshal peer；但 **`.so↔腳本函數` 永遠要付 `Value ↔ 引擎物件`（PyObject/lua value）的轉換，即使 in-process 也要**——腳本函數的值活在引擎物件世界、不是 C struct。「四捨五入成 `.so`」只在*引擎那層*成立，它 host 的函數**不是**零 marshal peer。此不對稱是本質、非 bug。

〔提案・「那個庫」的長相〕＝一個 **per-process 引擎 loader**：給定語言 X → 確保其引擎 `.so` 已 `dlopen` 進本 process（＝使用者說的「看自身 process 內有沒有該語言執行環境」）→ 回 handle；`.so` 交 `Value` 給它、它跑文本/叫函數、marshal 回 `Value`。此即本節「in-process 嵌直譯器」的實作。exec 那條不消失：引擎 adapter＝in-process 實現，spawn 子進程＝out-of-process 地板（不可信/無引擎時）。

〔決定・(a) 拍板，收回稍早的「放棄 shebang」〕本節稍早曾傾向「放棄 shebang、腳本一律經引擎進」；本輪 (a)/(b) 使用者選 **(a)**：**腳本的 CLI 面就用 OS shebang**（`./foo.lua` 直接跑），`.so` 面另走 `--gen-h`，**兩個面孔獨立**。故「沒有裸 `./foo.py`」那句**作廢**——裸跑回來了。§4.1 的核心（`.so`/link 面經引擎 adapter、腳本函數非 peer）不變，只是「一切經引擎」這個推論被限縮到 `.so` 面；CLI 面走 shebang。

### 4.2 〔決定・(a)〕每個檔案兩個面孔：CLI 走 shebang、`.so` 走 `--gen-h`
〔決定〕**每個檔案都可同時提供 CLI 與 `.so` 兩面孔**（含腳本語言），on-demand：
- **CLI 面**＝OS shebang（見上）。編譯型語言（Go/Rust）本就能出 `.h`，不必說。
- **`.so` 面（link 腳本）**：`--gen-h` 產出 **`.h`（介面宣告）＋`.c`（marshaling shim，把 `.h` 宣告的 typed C 呼叫轉成引擎吃得下的形式）**；依副檔名看引擎支不支援，不支援就不 link；支援＝啟動時查引擎環境是否載入、沒有則載入，把腳本拉進來 in-process 跑。`.h` 的「可用函數」由腳本自宣告（＝它的 typed manifest / input_schema，0708 §4）。

〔定位・補完 §3.1 的 2×2〕這是「腳本的 **known/typed** in-process 路徑」，補上 §4.1 引擎泛型 adapter（**unknown/動態**）缺的那格：

| 叫腳本函數 | 格 | 機制 | 給誰 |
|---|---|---|---|
| 引擎泛型 adapter（§4.1） | unknown/動態 | 一個 `call_value(fn_name,Value*)`、runtime 字串派發、不編譯 | 動態呼叫方 |
| `--gen-h`+`.c` shim（本節） | known/typed | 每檔生 `.c`→編成 `.so`→typed 具名符號、靜態綁定 | 編譯型 caller 要 typed 靜態綁定 |

**互補非取代。** 三個必記修正：
1. **marshaling 沒消失、搬進生成的 `.c`**：拿到 typed *介面*＋靜態綁定，**不是 `.so↔.so` 零成本**；每呼叫照付 `Value↔引擎物件`。＝「跟 `.so` 一樣*方便*、不一樣*快*」。
2. **「link 腳本」藏一步 codegen＋編譯**（`.c`→`shim.so`、連 libengine），按 source-hash 快取（套 §1.4）。故 **§4.1 泛型 adapter 當預設、生成 shim 當 opt-in**，別讓每個 link 背編譯。
3. **腳本須自宣告 typed manifest**（`--gen-h` 才有型別可吐）——動態語言不原生做，是強加的歸化工（LLM 可代生）。

〔紅利・反漂移〕CLI 面與 `.h`/`.c` 面**都從同一份「腳本＋manifest」生成**，天生一致、不漂——回答早先「兩份實現 drift」憂慮：生成即抗漂移。
〔護欄〕「都提供兩面孔」是 **on-demand/opt-in at boundary**（0708 §5：檔案化 opt-in、預設私有）——`.so` 面在「有人要 link」時才生，五行膠水腳本不被上繁文縟節稅。

〔提案・補門檻〕真正的門不只「支援」，還要「可信」，因為 in-process **同時失去崩潰隔離*和*資源上限**：
> **可信 ∧ 語言支援 → in-process；其餘一律 exec（安全地板）。**
in-process 快路徑因此接上憑證治理飛輪：驗證過的函式才「升級」拿到同進程綁定。v0「支援語言」大概就 {LuaJIT}（handy 現成、可嵌、極小）；Python 靠 libpython（重、GIL）晚點再說。

〔錨定・repo 印證此警告且更狠〕
- `docs/kb/knowledge_base/note_06...` **B3**：標準庫連 in-process 記憶體/時間上限都設不了，`--call-timeout` 只對 subprocess 有效。→ `.so` in-process 踩這個沒解的洞。
- 既有安全立場（note_06、`20260706-2257` §殘餘風險「FFI 一穿全毀」）：**FFI/重量級/高不確定 → 踢出去當 OS process**。與「讓 .so 進 in-process」直接張力，**靠上面的信任門＋exec 地板化解**（不是推翻既有立場，是給它一個 opt-in 的例外閘）。

## 六、待辦與最小起手

**可直接動工**：
1. 給 `call.py` 加 **`kind: "so"`（dlopen）** target：`from_spec` 認 `{"kind":"so","path":...,"entry":...}`，`.call` 走 dlopen ＋**萬能入口符號** `call_value(fn_name, Value* in, Value* out)`（in-process）／`call_bytes(fn_name, ...)`（序列化）（§3.1，兩符號一份 `Value` 語意，吃下所有 unknown 格）。先只做 **exec 地板等價**（`.so` 也能被 argv 叫），in-process struct 快路徑當第二步。
2. 定 **`Value` struct ＋ 每語言 FFI mirror**（先 C ＋ LuaJIT），與 in-process↔JSON 的 `serialize/deserialize`（JSON 預設）。
3. `.so` 的 `--metadata`／導出 meta 符號，對齊 0708 §四骨架（含 `input_schema` 缺口 #1）；CLI 標準加 **`--emit-header`**（meta 的 `.h` 投影，供 build-time/codegen/libffi/驗證，非現場 typed-call 用）。

**〔延後/懸〕**：
- CBOR 緊湊二進位編碼（跨行程優化，非必要）。
- `.so → 腳本` in-process 嵌直譯器（先只做 exec fallback；in-process 是「可信∧支援」的 opt-in 優化）。
- in-process 資源上限（note_06 B3 沒解的洞）——在信任門後，先接受「in-process＝放棄資源上限」。
- Python 當 in-process 支援語言（libpython/GIL）——v0 先只 {LuaJIT}。

**一句排程原則**：**exec 是地板**（永遠能用、語言中立、有隔離、守 JSON 規範），**in-process struct 是對「可信∧支援∧熱」函式的 opt-in 優化**。三輪討論在這裡與 §三、§四合流。
