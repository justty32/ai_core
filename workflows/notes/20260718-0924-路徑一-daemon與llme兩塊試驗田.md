## 〇、一句話

路徑一被使用者**重新定錨**了：北極星＝**把整個作業系統變成一個 AI agent**（一堆小程式的工具集，非單體）。近期落地＝兩塊**共存、互不隸屬的試驗田**——`llme`（把 cllm 多 endpoint 呼叫包漂亮）與 `daemon`（常駐 server，client 注入命令、開/操縱 agent session）。共享同一個**方法論**：拿現成程式、腳本包裝、按慣例組合。

## 一、本場的〔決定〕

### 1.1 方法論定調（使用者核心價值）
〔決定〕路徑一的建構方式＝**拿現有程式、用腳本包裝、按慣例組合、資料夾＝callable**。使用者對「用現有程式＋腳本靈活包裝」明說「**正是我想要的**」。→ 這是 0708「一切皆檔案／資料夾＝callable」落到地上的第一個真實身體，不是新發明。

### 1.2 北極星與消費者
〔決定〕終極目標＝**整個 OS 變成 AI agent**（工具集，非一個程式）。
〔錨定〕第一個消費者已存在＝使用者自己「常駐程式幫我做事」的工作流——這**回答了 0718 §四.1 停止鍵那題**：路徑一不再是無消費者的 yak-shave（見 §四）。

### 1.3 一塊試驗田，陸續往裡丟小程式
〔決定〕**daemon 與 llme 都是試驗田材料，可以共存**——不是一棵樹強行分兩層。
〔修正・落 handy 子專案後使用者澄清〕**它們不是「兩塊」獨立的田，而是同一塊田（＝handy 子專案這個資料夾）裡的頭兩個住戶**；handy 本身就是那塊試驗田，之後往裡面放一大堆小程式／小腳本，daemon 與 llme 只是起手兩個。（本則標題與下文「兩塊試驗田」措辭以此修正為準——實為一塊田、多住戶。）

## 二、試驗田 A：`llme`（多 endpoint dispatcher）

〔決定・第一個具體物〕在 cllm（＝很好的 llm_entry，惟只服務單一 endpoint）之上，寫個小程式把多個 cllm 呼叫方式彙集成 subcommand：

```
llme opus --stream 你好
```

組織形狀〔決定〕：
- 一個**資料夾**：內含 `_exec` ＋各 endpoint 要用的 config。
- 資料夾外一個轉發入口 **`llme.sh`** 小腳本，直接把呼叫轉給 `_exec`。
- ＝ folder-as-callable 的第一個真身（0708 §二）。

〔延後・隨意〕subcommand→config 的慣例（Claude 提案「`opus`→同名 `opus.json`，`_exec` 讀 `$1.json`→設環境→轉呼 cllm」；使用者說實作時再定、隨意）。

## 三、試驗田 B：`daemon`（常駐 agent 編排器）

〔決定〕常駐型 server；透過其他程式（client）**注入指令**，daemon 幫忙處理。最簡處理＝啟動另一個 AI agent session，或向某 agent 注入指令。

**注入指令分兩級**〔決定〕：
- **開新的**（近期只做這個）：`claude -p "..."` headless，官方支援、容易。
- **操縱活著的**：用折衷——**讓 agent 自己定期 poll 一個檔案讀命令**（拉，不是外部推）。
  - → 乾淨繞過「CC 沒有官方對外注入 API」那面牆（Q5 生死線）。
  - → 完全在主軸上：**命令通道本身就是一個檔案**（一切皆檔案）。daemon 只管**寫檔**，活 agent 自己**讀檔**。

**動機是開放式的**〔決定〕：起因是「覺得 Claude Code 的 cron 定時器有時不太靈光」，但使用者澄清 **cron 只是舉例，反正總是會有事情要做**。→ daemon 不綁單一殺手級功能，是**通用**的常駐試驗田。

**唯一有下游後果的懸念**〔延後〕poll 的「**已讀**」問題（agent 怎麼知道哪些命令是新的、不重跑）。Claude 給兩形狀，使用者未選：
- (a) **append-only log＋游標**（NDJSON，記讀到第幾行）＝順手複用 0718 §6.3「三條路共用的第一磚」。
- (b) **收件匣目錄**（每命令一檔，處理完 `mv` 到 `done/`）＝更像資料夾哲學，但要處理併發寫。

## 四、擋掉的兩個使用者自標陷阱（記錄者職責）

1. **「從零重造」陷阱**（roadmap §1／0713 §9.5「port，別憑記憶重建」）。本場對映到現有 repo 實體，確認**三塊都有現成、不要重寫**：
   | 使用者要的 | 現成實體 | 位置 |
   |---|---|---|
   | daemon 骨架（常駐＋socket＋佇列＋單例計量） | `llm_entry.cpp` 的 `--serve <sock>` | `sub_projs/ver_1/try_implement/core_handy/examples/llm_entry.cpp` |
   | 偵測 key 失效→開網頁重登→patch config→重試 | `llm-login`（OAuth＋PKCE、`xdg-open`、`llm-login token` 給腳本 `$(...)`、`status`） | `sub_projs/cllm/tools/llm-login/` |
   | 命令 log／memoize／可靠度原料 | NDJSON trace 構想 | 0718 §6.3 |
   - cllm 本體在 `sub_projs/cllm/`。

2. **「§四.1 無消費者 yak-shave」陷阱**——「反正總有事要做」正是「可以永遠做下去卻沒有消費者」的形狀。**解藥＝使用者把它定性為「試驗田」而非「基礎建設」**：試驗田低風險、隨時可棄；只要守住「這是玩、不是一次蓋對」，§四.1 的危險就被中和。〔提案・使用者語氣默認〕daemon 是**可隨時推倒的試驗田**，不是非做好不可的地基。

## 五、〔延後〕清單（本場明說之後再說／隨意）

- **守衛（keycheck）**：轉發前預檢 key vs 事後 401 攔截（llm-login 現設計是事後、cllm 零改動）——只是舉例，之後再說。
- **`llme` config 慣例**：實作時再定、隨意。
- **poll「已讀」形狀**：(a) append-log＋游標 vs (b) 收件匣目錄——未選。

## 六、下一步的最小起手（〔提案〕，未追認）

若要動手，風險最低的第一鏟：**試驗田 B 的「開新的」那半**——一支常駐小程式，client 寫命令進一個檔／socket，daemon 讀到就 `claude -p` 起一個 headless run。它同時吃到 append-log（§三懸念 a）當命令通道，是三條路共用磚，且不需承諾任何慣例。llme 那塊可並行、互不擋。
