## 〇、一句話

世界篇把「Linux-as-Lisp」與「OS-as-game」當**並列**兩層底座；本篇修訂為**有序兩階段**——
**先「歸一於路徑」**（把一切、包括執行與呼叫，統合到檔案系統、皆可用路徑定址），
**後「成局」**（在這副已統一的基材上，用檔案系統做成遊戲，類 Godot 節點樹）。
空間與複雜度由**路徑**直接統合；時間則靠 **tick 這個狀態轉移函數**外掛，代價是暫時忽略 CPU 週期。

## 一、重定調：兩層底座是「有序兩階段」，非並列〔決定〕

世界篇 §一把兩層當同一堆 Linux 事實的兩種視角、並列。**本篇修訂**為有因果順序：

### 1.1 階段一＝Linux-as-Lisp 的**真正本質＝歸一於路徑**〔決定〕

Linux-as-Lisp 的本質，不只是「檔案＝函數、exec＝apply」的比喻，而是一個**統合意圖**：

> **把一切統合到檔案系統下，一切都可以用路徑找到。**

「一切」明確包含三類，逐階遞進：

1. **靜態物**：記憶體或硬體內的東西（檔案、資料）——傳統檔案系統已做到。
2. **可被執行的 / 正在執行的東西**：可執行檔、跑著的 process/thread——要能用路徑定址（/proc 式）。
3. **呼叫與使用方式**：一個東西「怎麼被叫用」本身，也要收斂成路徑可尋。

這三類收齊，才叫「歸一於路徑」。這比世界篇 §十一「通用協議＝檔案」**更進一步**——
不只*協議*是檔案，是連**行為／執行／呼叫**都收斂到路徑。

### 1.2 階段二＝OS-as-game 建在其上〔決定〕

**統一之後，才是** os-as-game：用這副「萬物皆路徑」的檔案系統做成遊戲。
使用者給的類比：**類似 Godot 的節點系統**（場景樹、每個節點有 NodePath）。

### 1.3 對世界篇 §一的覆蓋

本節**覆蓋**世界篇 §一「兩層講的是同一堆 Linux 事實、只是視角不同」的並列敘述：
兩層仍是同一堆事實，但**有先後**——階段一是**前提**，階段二是**其上的建築**。

## 二、三軸再定位：空間＋複雜度由路徑統合，時間靠 tick 外掛

使用者點明：前面講的統合，**大多是在講空間與複雜度**（它們天然能被路徑吃下）。
時間是唯一啃不動、需要另一套機制的軸。

### 2.1 當前檔案分佈＝此刻的空間狀態〔決定〕

因物理世界限制，**我們現在看到的檔案分佈，就是「這個時間點」的空間狀態**（＝世界篇的「局面」）。

### 2.2 tick＝世界狀態轉移函數〔決定〕

想看下個時間點的空間狀態 → **跑 tick**。tick 的職責被顯式定義成一個**轉換**：

```
tick(當前檔案分佈, 使用者影響) → 下個時間狀態的檔案分佈
```

- 它把當前局面，**轉換**成下一個時間狀態的局面；
- 且**納進使用者的影響**（使用者不是旁觀者，是轉換的輸入之一）。

這給了世界篇 §十一「時間本身也是空間的一個維度」一個**操作定義**：
時間推進 ＝ 對空間狀態跑一次轉換函數。

### 2.3 tick 可指定時間跨度（時間跳躍）〔決定，單位未定〕

若把 tick 視為「移動到指定時間之後的檔案分佈」，那 tick 就要能**指定跳到多久之後**（帶 Δt）。
→ tick 不只是「推一輪」，而是「快轉到 t+Δt 的局面」。
〔未定〕Δt 的單位（見 §五 疑點 4）。

## 三、承認的抽象代價：CPU 週期被忽略〔決定〕

tick 這套「很緩慢的循環」，**暫時忽略 CPU 週期**——
「這是沒辦法的事情，一定得做些抽象」。使用者明確接受這個代價。

〔疑〕**邊界**：世界篇的**快世界**（tick 驅動檔案→process，即 0718 的 in-process/.so 那套）**恰恰在乎效能**。
所以「忽略 CPU 週期」應只適用**慢世界**；快世界那層不忽略。這條邊界要在兩世界通聯協議裡標清楚。

## 四、已知裂縫：Lisp 數字 index vs 檔案系統字串 index〔錨定・實作細節延後〕

Linux＝Lisp 的對映有一道裂縫：

- **Lisp 的 list**：每個元素的**天然 index 是數字**（`nth`／`car`/`cdr` 的位置語意）。
- **檔案系統**：所有元素的 **index 是字串**（路徑名／檔名）。

使用者定位：**這是實作細節，想解決就能解決，之後再說。**

〔疑〕其實 Lisp 半邊本就有解——`alist`/`plist`/`hash-table` 都是**字串（symbol）鍵**；
而「**目錄＝名字→inode 的 alist**」是極自然的對映。所以裂縫只在「純 `list`」那半邊，
比表面小。也許不是「調和數字與字串」，而是「**選對 Lisp 資料結構**（keyed 而非 positional）去對映目錄」。

## 五、疑點／待釐清（Claude 提、待使用者回應）

1. 〔疑・傾向修訂〕**「有序兩階段」定案為修訂世界篇 §一並列敘述**——本篇 §1.3 已按此寫。請追認，或改回「並列＋補一句因果註」。
2. 〔疑・傾向掛鉤〕**「歸一於路徑」是兩塊舊筆記的上位總綱**：
   - 「呼叫與使用方式收斂到路徑」＝ 0718〈呼叫閘道〉（in_process/subprocess/http/.so → 一個 call）的哲學上位；
   - 「正在執行的東西可路徑定址」＝ 世界篇 §十一開放項 process-as-file（/proc 式）。
   建議把這篇當**收束句**，顯式把那兩塊掛到「歸一於路徑」下。
3. 〔疑〕**Godot 類比的張力**：NodePath（`/root/Player/Sprite`）確實是「用路徑定址正在執行的東西」，很精準；
   但 Godot 統一的是**記憶體內執行態的樹**，本篇要統一的是**磁碟＋執行態**為同一棵路徑樹。
   這道 gap ＝ 世界篇動態/靜態軸（記憶體/硬碟）未弭平處，是階段一最硬的工。
4. 〔疑・單位未定〕**tick 的 Δt 與「等所有 NPC 跑完」有張力**：世界篇的 tick 時長是**湧現的**（等所有人＋玩家喊停）；
   本篇要「指定跳多久之後」。Δt 是**回合數**（跑 N 個 tick）？還是 tick 內部有個**模擬時鐘**可快進？兩種模型不同。
5. 〔疑〕**「使用者影響」是輸入參數還是執行期介入**？`tick(state, user_influence)` 讀起來像啟動前給定；
   世界篇說玩家「執行期間持續介入、喊『這輪到此』才收尾」。傾向後者（持續介入、收尾封存），寫法待對齊。
