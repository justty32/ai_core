## §5 可中斷性

<!-- 要描述的問題：是否可以被暫停、暫停後可否恢復、恢復後副作用為何 -->
<!-- 已知形式：不可中斷、可暫停（SIGSTOP）、可恢復（checkpoint/restore） -->

### Phase 1 分析

可中斷性應拆為兩個層面：

**§5a OS 排程層（單次執行內）**：執行途中被作業系統或使用者暫停/恢復/延遲

| 情形 | 說明 |
|---|---|
| 不可中斷 | 執行必須連續完成，中斷會導致狀態損毀（e.g., atomic write） |
| 可暫停（SIGSTOP） | OS 可掛起程序，資源仍佔用，恢復後繼續執行 |
| 可恢復（checkpoint） | 程式自行序列化執行狀態，殺掉後可從斷點重啟 |

**§5b 跨執行狀態層（跨呼叫）**：中斷一次執行後，對 §3 的跨呼叫狀態有何影響

| 情形 | 說明 |
|---|---|
| 無殘留 | Stateless，中斷不影響後續執行 |
| 狀態損毀 | 外部狀態寫到一半，下次執行讀到壞資料 |
| 狀態一致（需 §6 執行保證配合） | 透過 transactional 或 idempotent 保證中斷後狀態仍安全 |

> §5a 與 §5b 要一起看，§5b 的安全性很大程度取決於 §6 執行保證的設計。資源釋放行為見 §4。

## §6 執行保證

<!-- 要描述的問題：對系統狀態的承諾，獨立於執行次數或中途失敗 -->
<!-- 已知形式：idempotent、transactional（ACID）、dry-run -->

### Phase 1 分析

屬於 metadata 描述範疇，是程式對呼叫方的承諾聲明。

| 保證類型 | 說明 |
|---|---|
| 無保證 | 預設值；呼叫方自行承擔重複執行或中斷的後果 |
| Idempotent | 重複執行與執行一次結果相同；中斷後安全重試 |
| Transactional（ACID） | 要麼完全成功，要麼完全不發生；中斷後回滾，不留部分狀態 |
| Dry-run | 執行但不實際修改外部狀態；用於預覽副作用 |

**中斷後如何處置**由此軸決定，與 §5 可中斷性對接：
- Transactional → 中斷後回滾，§5b 狀態損毀風險由保證機制吸收
- Idempotent → 中斷後可安全重跑，§5b 無殘留問題
- 無保證 → §5b 的損毀風險由呼叫方自行處理

## §7 組合模式

<!-- 要描述的問題：執行單元如何與其他單元構成更大的結構 -->
<!-- 已知形式：fan-out/fan-in、proxy/wrapper、hook/callback、agentic loop -->

### Phase 1 分析

**此軸不在本規範的主要描述範圍內。**

組合模式描述的是呼叫鏈（call chain）的結構，屬於系統架構層面，而非單一執行單元的屬性。單一程式本身不需要知道自己被如何組合；這是呼叫方（hub、router、agentic loop 等）的關切點。

> 若有需要，可另立文件討論「呼叫鏈設計」，但不應塞進單一執行單元的 metadata。

## §8 環境模式

<!-- 要描述的問題：執行環境是預先存在還是動態建立與銷毀 -->
<!-- 已知形式：固定環境、ephemeral/JIT -->

### Phase 1 分析

**此軸同樣不在本規範的主要描述範圍內。**

程式本身通常不管理自己的執行環境。預設假設：**穩定的 Linux 環境**，其餘狀態由使用者自行承擔。

唯一例外：可以設計一個**包裝程式（wrapper）**，用於修改其子程式的執行環境（切換工作目錄、設置環境變數、resource limit 等）。這個包裝程式自身是一個執行單元，其行為用其他軸描述；環境管理只是它的輸出副作用。

> Ephemeral/JIT 環境（如 Docker container、venv）是部署/執行層的選擇，不影響程式介面設計。
