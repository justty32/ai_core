## 7. 這個目標會逼出哪些待決規範

v0 切片會自然撞到 `sub_projs/ver_1/try_implement/DECISIONS.md` 裡的幾個懸案，到時它們就從「憑空猜」變成「擋路、得解」：

| 懸案 | 為何被這個目標逼出來 | 出處 |
|---|---|---|
| **語意 / 用途描述欄位**（Gap C / B1） | 生資產一定撞到——八軸只描述執行特性，沒有「這環節做什麼、吃什麼參數」。 | DECISIONS B1 |
| **in-process 沙箱**（Gap E / B3） | 要跑程式碼 / 測試來驗證笨模型的產出，需要資源與安全隔離邊界。 | DECISIONS B3 |
| **memoize 快取**（候選軸 C） | 便宜≠免費；笨模型也要錢 / 算力。重複輸入該狂快取。 | DECISIONS C |
| **`nondeterministic` 軸（含證書）** | §3.4 的治理原則需要它落地——不只「這裡隨機」，而是「測試組 / 模型 / 穩定度 %」的證書。 | DECISIONS C |
| **組合軸推導**（A4） | 資產被組成調用鏈時，複合 metadata 要能自動算。優先序待 v0 驗證後再定。 | DECISIONS A4 |

> 結論：**v0 切片本身就是規範的篩子**——跑通它的過程，會告訴我們上面哪幾條是真的擋路、哪幾條可以再放著。

**② 這次檢討想得出什麼（roadmap 的緣起）**

> 我是這樣想的，我想要定下一個初步的roadmap，在我們把八軸,複合,多函數合作等概念都定下標準後，我們等同是有了工具，那麼我們該先拿哪個問題作爲目標呢？

**③ 緊迫性 + 第一目標＝程式碼助手**

> 並且我突然想到一點，那就是，其實我會想要儘快把這些東西做出一點成果，至少在當下，比較聰明的大模型還算便宜的時候，先讓飛輪轉幾圈，打個骨架，這樣可以省下今後不少功夫。然後第一個目標問題，我想先做一個程式碼輔助助手，好比說我有個一個程式框架，以前用其他很聰明的大模型+ai agent，如claude code，請他去讀框架原始碼，然後去生skill，之後再去輔助編寫相關程式碼，那當然很好，但很花錢。目前的想法是這樣，還是先讓聰明的大模型去生skill，但是在編寫相關程式碼部分，我們可以做細化，比如把一些固定snippet寫入prompt，配合行數助手精準標記要寫入的目標...你覺得呢？等等，其實結構化抽取也不錯

**④ 搜尋報告例子 + skill 超重（是函數/調用鏈）+ LLM 退縮到兩介面**

> 然後我還有個想法，好比說我想要讓模型去搜尋某個topic，然後做報告，然後我對那個topic稍有了解，知道它應該要先讀取某些文件，然後再去某些網站用APi取得資料，所以我會給一個prompt，模型就會依據這套prompt去採用一套。然後這一套還可以繼續改進，比如說模型可能不知道API有哪些，那麼模型本身會跑另一個已經固定的流程，去分析網站的形式，若是A形式，那可以透過某些方式取得API列表之類，除此之外，還有使用者習慣這個功能，在做這一整套流程時，模型會知道我是繁體中文使用者，所以他產出的文件都會是繁體中文。然後你說的ai_core的skill，那確實會超級重，那不會只是一個文字檔，而是一個函數，並且會受到所使用的模型型號所影響，有所變化，乃至於不限於一個函數，而是一整個調用鍊條。然後我預期中的使用方式會是這樣，好比說我給了一個promp，這個prompt會先經過一個固定函數，判斷說是否符合某些特定format比如說"改XX文件的XX行：..."，然後去進行相關處理，而只有在...部分這塊沒有被預先定義的格式，才會拿模型去進行分析，並且有可能會先讓笨模型思考，然後若笨模型思考沒通過邏輯測試，那會讓聰明模型重新思考一次。在我的構想中，在成熟的系統中，LLM只會成爲人類意圖與程式可理解的意圖的轉換接口，還有一些尚未被定義的糢糊地帶的暫時處理者。所以，嚴格來說，其實這不太能說算是skill。
