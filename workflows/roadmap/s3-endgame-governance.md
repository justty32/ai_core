## 3. 終點形狀與治理原則（本專案的核心）

### 3.1 終點：LLM 退縮到兩介面

在成熟系統裡，系統大部分是確定性程式碼，**LLM 只剩兩個介面**：

1. **意圖翻譯層**：人類的模糊意圖 → 程式可理解的結構化意圖。
2. **模糊前沿的暫時處理者**：還沒被固化成程式碼的模糊地帶，暫由 LLM 扛著。

這兩個洞是**永遠不會完全閉合**的（總有新意圖、總有新模糊案例）；其餘的洞都該隨時間被填成程式碼。

### 3.2 單位不是「skill」，是「帶 LLM 留白的程序」

我們一度叫它「skill」，但它太重了，這個詞不準。它其實是：

> **一段大部分為確定性程式碼的「程序 / 套」，中間留著少數 LLM 才填的洞。**

最精準的類比——**LLM 是 cache-miss handler，或一個巨大 `switch` 的 `default:` 分支**：

- 請求先過一個**確定性前濾網**（format matcher，如「改 XX 檔 XX 行：…」）。命中 → 走純程式碼。
- 只有**落穿前濾網的、沒被預先定義的那塊**（範例裡的 `…`），才掉進 LLM。
- 它也不限於一個函數，而可以是**一整條調用鏈**；且**會隨所用模型型號而變化**（見下）。

### 3.3 洞的大小 = f(模型能力)

關鍵性質：同一段程序，**笨模型 → 塞更多 matcher、洞更小**（不敢信它）；**聰明模型 → matcher 更少、洞更大**（可放手）。同一套程序、不同的「填充比例」。

- 「先笨模型想、沒通過邏輯測試再叫聰明模型」＝ **per-request 動態縮放這個洞**。
- 橫切的 **使用者習慣層**（如「我是繁中使用者 → 產出都用繁中」）疊在整條鏈上，是 `llm_call` context binding 的 user-profile 級升級。

### 3.4 治理原則：拒絕為預設 + 憑證准入

**舉證責任在「讓 LLM 進來」那邊。** LLM 預設零領地；要佔據某個環節，得通過**兩部分閘門**：

1. **必要性**：千錘百鍊證明**確定性程式碼辦不到**、非 LLM 不可。
2. **穩定性**：在測試組下**做得夠穩**，過可靠度門檻。

兩個類比：
- **最小權限（least privilege）**：LLM 預設無權，每個環節單獨申請、單獨核准。
- **新藥審批**：一個 LLM 環節 = 一款藥，臨床試驗證明**療效**（非它不可）＋**安全性**（夠穩），才**核准用於該適應症**。

**後果——每個 LLM 留白帶一張證書**（這把飄很久的 `nondeterministic` 軸做實了）：

> 不是含糊的「這裡是隨機的」，而是「**此環節經測試組 B、用模型 Z、認證穩定度 X%**」。LLM 不是被信任，是被**認證**；可被**撤照**（哪天有人想出確定性解法就踢出去）。整個系統因此**可稽核**：永遠知道哪幾個點是「沒辦法才放 LLM 進來」、各自多穩、用什麼擋著。

### 3.5 兩個 regime，飛輪是它們之間的遷移

「拒絕為預設」的嚴格和「趁便宜先轉幾圈」的寬鬆，表面衝突，其實是**兩個 regime**：

| regime | 時機 | 姿態 |
|---|---|---|
| **開機期** | 現在，聰明模型便宜 | 寬鬆——讓 LLM 多做、先把骨架搭起來、先轉幾圈 |
| **成熟期** | 未來，只剩笨模型 | 嚴格——拒絕為預設、憑證准入、footprint 極小 |

> **飛輪 = 從寬鬆遷往嚴格的力**：持續地 (a) 把 LLM 環節換成確定性程式碼（**撤照**）、(b) 給活下來的 LLM 環節發**證書**（**認證**）。開機期容許鬆，正因為飛輪保證它會收緊。

### 3.6 最硬的未決問題：固化引擎

飛輪的「免費能量」來自**固化（crystallization）**——把「LLM 老在處理的同一類模糊案例」凍成確定性 matcher / 分支。**但固化不是免費的：誰做？**

- **手動**：人觀察、人寫新分支 → 這專案是「一個好用的工具」。
- **自動**：聰明模型定期掃 log、自動提案新的確定性分支 → 這專案是「一個會自我改進的系統」。後者難非常多。

此問題決定 v0 切片要不要把「固化」也做進去（見 §8）。**目前未決。**
