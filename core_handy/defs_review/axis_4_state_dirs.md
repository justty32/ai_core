# 軸 4 `state_dirs` — defs 不足審查

> 審查對象：`notes/axis_4_state_dirs.md`（2026-06-24 重定位：軸 4 = 純設施，**不在描述性九軸**；描述面外包給軸 3）。
> 現況：設施 API 待定（自由函式 `ac::state::path/load/save` vs 託管物件 `ac::StateStore`）。

軸 4 已被判為 impl（設施），所以「defs 不足」嚴格說不適用它本身。
但**重定位本身造成了一個描述面的損失**——這才是本軸對 defs 層的意義。由強到弱：

---

## 1.〔描述能力消失〕權威 `state_dirs` 會進 `--metadata`，濃縮後整個從描述面蒸發

權威的 `state_dirs` 是會序列化進 `--metadata` 的（`{config,cache,state,data}` 的子集 list），
它讓 hub 知道**副作用具體落在哪些目錄、嚴重性如何**：
- `data` 目錄被動 → 中斷風險高（軸 6）、需備份；
- `cache` 被動 → 無所謂，可隨時刪；
- 波及範圍 → hub 評估該不該重試/平行/隔離。

把軸 4 整個移出描述面後，hub 對這些**只剩軸 3 的一個 `bool stateful`**——
答不出「動了哪個目錄、多嚴重」。**這是權威有、濃縮後消失的描述能力。**
（對照軸 1：傳輸身分也被推給設施 → 同一個「描述面被掏空、只剩 hub 不夠用的抽象」的模式。）

建議：即使設施在 impl，仍保留一個**輕量描述**讓 hub 看見波及範圍。
候選——在軸 3 或軸 5 的 `extra` 標 `state_dirs:"data,state"`，或讓 `StateStore` 託管時**自動把用到的目錄回填進 metadata**。

## 2.〔託管不回填〕「託管即隱含 stateful」目前不連動 → metadata 易與實際不符

筆記載明：用 StateStore 託管與軸 3 `stateful` **目前不連動**（純選用）。
後果：作者用了 `StateStore` 寫 data 目錄，卻忘了在 `register` 標 `stateful=true` →
**metadata 謊報無狀態**，hub 據此放心平行/重試 → 踩雷。
這正是軸 3/軸 4「一個說有、一個提供託管」的對稱關係該閉合處：
建議**託管行為自動回填 `stateful=true`（及用到的目錄）**，讓描述與實作無法偏離。

---

## 設施 API 待定（D-API）— 屬正常未完成，非「不足」

自由函式 vs 託管物件（傾向物件式 `StateStore{program_name}`）尚未拍。
這是 impl 階段事，按筆記節奏（描述面收尾後才統一重做 impl）屬正常進度，不計入 defs 不足。
唯一要在**現在**就釘的是上面第 1、2 點——它們是「描述面 ↔ 設施面」邊界的決定，會反過來影響 metadata 形狀。

## 根源懸案

軸 3 + 軸 4 應作為**一對**重新對齊：
**「有外部狀態」這個事實，描述面要暴露到多細？** 只一個 bool（現狀）／加波及目錄／加讀寫別。
這題與軸 3 點 1（讀 vs 寫）、軸 1 點 3（傳輸身分）是同一張網。
